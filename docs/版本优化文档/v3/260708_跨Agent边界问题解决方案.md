## 二、第一优先级：致命问题解法

### 修复 #1：伏笔 ID 体系分裂

**解法：确定性模式下废除 plot_planner 的 §4 自建伏笔系统，改为纯引用模式**

具体修改点：

**a) plot_planner §4 增加模式分支**

在 §4 开头增加：

```
### 模式判定

若当前为「确定性细化模式」→ 执行「伏笔引用模式」（本节约束）
若当前为「标准模式」→ 执行原「伏笔管理系统」（§4.1~§4.3 原规则不变）

## 伏笔引用模式（确定性细化模式下强制启用）

本模式下，你不再自建伏笔 ID（F001/F002 等），改为以下行为：
1. 从注入包 §4（本卷需埋设的伏笔清单）读取所有伏笔 ID 及其内容
2. 从注入包 §3（本卷确定性事件链）读取每章的"新埋设伏笔ID"和"已有伏笔回收ID"
3. 章纲的伏笔管理段落改为"伏笔引用登记表"：

┌──────────┬──────────┬──────────┬────────────┬──────────┐
│ 伏笔ID   │ 级别     │ 内容摘要  │ 本章操作    │ 实现方式  │
│ (destiny)│ (A/B/C)  │          │ (埋设/回收) │           │
├──────────┼──────────┼──────────┼────────────┼──────────┤
│ B-07     │ B        │ xxx      │ 埋设       │ 描述如何埋│
│ A-02     │ A        │ xxx      │ 回收       │ 描述回收  │
└──────────┴──────────┴──────────┴────────────┴──────────┘

4. 本模式下禁止生成 F-xxx 格式的伏笔 ID
5. 若本卷事件链中的某个伏笔 ID 在注入包 §4 的清单中找不到 → 标注 "⚠️ 伏笔ID {X} 未在注入包中定义，本章暂不埋设"
6. 若本章确实需要埋设注入包未声明的本地临时伏笔（纯功能型，单一卷内回收），使用 LOCAL-NNN 前缀，且必须在注入模式下完成当前卷全部章节后统一登记到伏笔引用表的独立分区
```

**b) destiny_designer 的伏笔 ID 命名规范**

在 §步骤 5 的伏笔网络表头部增加：

```
伏笔 ID 命名规范：
- A 级伏笔（主题级）：A-{序号}，如 A-01、A-02
- B 级伏笔（跨卷级）：B-{序号}，如 B-01、B-02
- C 级伏笔（卷级）：C-{卷号}-{序号}，如 C-03-01
```

**c) 注入包 §4 增加伏笔内容摘要字段**

```
## §4 本卷需埋设的伏笔清单

| 伏笔ID | 级别 | 内容摘要（≥15字） | 埋设章号 | 如何埋设（场景/道具/对话/旁白） |
|--------|------|-------------------|---------|------------------------------|
| B-07   | B    | xxx               | 第45章   | 拍卖道具的异常反应暗示...     |
```

这样 plot_planner 不需要回溯 03-伏笔网络就能引用完整的伏笔信息。

**d) 标准模式兼容**

标准模式下保留 §4 原有的 F-xxx 伏笔管理，但增加一条：
- 生成 F-xxx 时同时标注对应的 destiny 伏笔级别估算（如 F001(B级)/F002(C级)），以便后续与 destiny 增量同步时有映射依据。

### 修复 #2：注入包 §5 爽点预算来源缺失

**解法：由 plot_planner 根据自身已有规则计算爽点预算，不依赖注入包 §5**

理由：爽点预算分配本身是叙事形式层（HOW）的决策，不是故事事件层（WHAT）的决策。它属于 plot_planner 的职责域。destiny 不应越界生成也不具备计算所需的信息。

**具体修改点：**

**a) destiny_designer 的注入包 §5 改为引用声明**

```
#### §5 本卷爽点约束提示

不从 00~04 提取预算分配表，改为引用声明：
- 本卷事件链中使用的爽点类型集合：{从 02 事件表统计}
- 盐值 pleasure_rotation 的策略提示：{直接引用 salt 中相关字段原文}
- 以下爽点约束由 plot_planner 自行计算，本节不提供：
  · 各类型的主推/次要/密度分配
  · 与前卷的轮换衔接
```

**b) plot_planner 在确定性模式下从注入包事件链自行计算预算**

```
在确定性细化模式下，爽点预算按以下规则计算（替代注入包 §5）：

1. 扫描注入包 §3 事件链中本卷所有章的 "爽点类型" 列，统计各类型使用次数
2. 检查是否符合 salt 中 pleasure_rotation 的频率约束（如有）
3. 检查是否符合总纲领 §6 的章均密度要求
4. 若统计结果与频率约束冲突 → 优先以注入包事件链为准，在章纲末尾标注：
   "⚠️ 注入包爽点分布与盐值约束偏差：{具体差异}，建议人工 review"
5. 预算分配写入卷纲「本卷爽点预算」段落

预算是根据注入包事件链反算出来再逐章细化分配的，而不是从注入包读出来的。
```

**c) 效果**：
- destiny 只需要保证事件链中的爽点类型本身合理（它毕竟是设计事件的人），不需要做额外的预算计算
- plot_planner 根据注入包事件链反算预算——这是它本该做的叙事规划工作
- 冲突不会静默覆盖，而是显式标注供 review

### 修复 #3：确定性模式下 §3/§4 不应并行运行

**解法：§3 和 §4 都需在确定性模式下切换为替代规则**

**a) §3（角色成长阶段）的修改**

```
### 模式判定

若当前为「确定性细化模式」→ 执行「角色状态校验模式」
若当前为「标准模式」→ 执行原「角色成长阶段追踪」（§3.1~§3.2 原规则不变）

## 角色状态校验模式（确定性细化模式下强制启用）

1. 本章不再标注"隐忍蛰伏/暗中布局/..."等 6 阶段标签
2. 改为从注入包 §2 读取每个本章出场角色的本卷初/末状态
3. 对每个出场角色执行一致性校验：
   - 本章事件是否与该角色的本卷初状态自洽？
   - 本章事件是否导致该角色的状态变化，且变化方向朝本卷末状态收敛？
4. 若校验不通过 → 标注 "⚠️ 角色状态冲突：{角色名} 本章{事件描述} 与注入包定义的{状态描述}不一致"
5. 章纲的「成长阶段」字段改为：「本卷角色状态（来源：注入包§2）」
```

**b) §4（伏笔管理）的修改**——见问题 #1 的解法

**c) 效果**：
- 确定性模式下不再用通用 6 阶段模型套具体角色
- 角色约束的权威来源是注入包，而非 plot_planner 的自治模型
- 冲突发生时是"标注 + 放行 + 供 review"，而不是"两套规则打架无裁决"

## 三、第二优先级：严重问题解法

### 修复 #4：注入包与 02 源数据同步校验

**解法：在 chief_editor §1.5 加载注入包时增加同步校验**

**具体修改点：**

在 chief_editor §1.5 的步骤 2（"若存在"分支）中，读取注入包后、使用前，增加校验：

```
2.1 校验注入包事件链与 02-剧情命运谱 的一致性：
    读取 versions/{version}/上帝之眼/02-剧情命运谱/卷0X-剧情.md
    提取该卷事件表的"章号"和"核心事件"两列
    对比注入包 §3 事件表的"章号"和"核心事件"列：
    - 章数不一致 → ❌ 停止，日志记录"注入包事件链行数({a}) ≠ 02源数据行数({b})"
    - 章数一致但核心事件内容不一致 → ❌ 停止，日志记录具体差异的章号

2.2 校验注入包角色与 02/01 的一致性：
    注入包 §2 中列出的角色名必须在 01-人物命运谱/ 中有对应文件
    注入包 §2 中列出的角色名必须在 02 本卷事件表的"出场角色"列中至少出现一次

2.3 校验注入包伏笔与 03 的一致性：
    注入包 §4 中列出的伏笔 ID 必须在 03-全书伏笔网络.md 中有对应条目
    03 行数 = 注入包 §4 行数（至少注入包的 ID 是 03 的子集）
```

**实现注意**：这些校验不要求严格的字符串匹配（模型输出有微小措辞差异是正常的），采用"章数相等 + 相似度 > 阈值"的策略。校验失败时不是静默继续，而是标记"⚠️ 注入包数据可能过期，建议重建"。

## 四、第三优先级：管道问题解法

### 修复 #8：destiny 的单点阻塞

**解法：将 destiny_designer 的调用策略从"全量一次性"改为"batch 分卷生成"**

**具体方案：**

不在一次调用中生成全书所有卷的 00~05，改为：

```
策略 A（两阶段 destiny）：
  Phase 1 新增步骤 1.5：生成 00（全书命运总谱）+ 全部 01（人物命运谱）+ 03（伏笔网络）+ 04（世界观展开谱）
  Phase 2 每卷开始前：用 increment 模式生成该卷的 02（剧情命运谱）+ 05（注入包）

策略 B（卷级 lazy 生成）：
  保留 destiny 的一次性全量设计，但只输出 00+01+03+04
  02（逐章事件链）分摊到每卷开始时生成
  05（注入包）在 02 生成后即时生成

策略 C（兜底机制）：
  不做分卷，保持全量一次性，但增加以下兜底：
  - 若 destiny 超时/失败 → 回退到 plot_planner 标准模式
  - 章纲文件开头标注 "⚠️ 上帝视角生成失败，使用标准模式（无全局一致性担保）"
  - quality_reviewer 对这些章的评分自动下调 5%（惩罚不确定性的风险）
```

**推荐策略 A**，因为：
- 00+01+03+04 是全局性的，确实需要一次全量设计
- 02（逐章事件）是具体的，可以在卷级 batch 中生成，减小单次 token 压力
- 每卷的 02+05 生成失败只影响当前卷，不影响已完成的卷

**修正 chief_editor 流程**：

```
§1.5 加载全局命运：
  → 检查 00/01/03/04 是否存在，不存在则调用 destiny（全量模式，只生成 00~04）
  → 检查本卷 02/05 是否存在，不存在则调用 destiny（增量模式，只生成卷 X 的 02+05）
```

**destiny_designer 需要改造成支持两种调用模式**，或者新增参数控制生成范围。

### 修复 #9：无增量重建机制

**解法：destiny_designer 增加卷级 rebuild 参数**

**具体修改点：**

在 destiny_designer 的永久置顶增加：

```
- 输入5（可选·增量重建）：rebuild_volume — 整数，指定只重建某个卷的 02+05 数据
  若传入 rebuild_volume = {N}：
    只执行步骤 4（生成卷 N 的逐章事件链）和步骤 7（生成卷 N 的注入包）
    并重新执行步骤 8（全局自检，检查卷 N 修改后是否与其他卷的角色状态/伏笔网络一致）
    不修改 00/01/03/04
  若未传入：全量生成所有卷
```

**chief_editor 调用侧**：

```
# 在 §二（卷纲规划）开始时
若该卷的 02/05 已存在但 _自检报告.md 中标注了历史不一致：
  → 调 @destiny_designer rebuild_volume={X}
  → 更新自检报告
  → 重新加载该卷注入包
```

**额外保障**：增量重建后必须重跑步骤 8 的完整自检，因为修改一个卷的事件可能影响跨卷伏笔回收点、角色状态过渡等。

### 修复 #10：automation_manager 对 destiny 无感知

**解法 A（最小改动量）**：不把 destiny 加入 Phase 1，但 automation_manager 在 Phase 1 完成后输出中增加提示：

```
在 Phase 1 完成汇总表后追加：

╔════════════════════════════════════
║  ⚠️  重要：Phase 1 仅生成框架产物
║     命运设计书（上帝之眼）将在 Phase 2 启动时自动生成
║     需要确保以下输入文件就绪：
║       - base_whitepaper.md      ✅
║       - project_salt.json        ✅
║       - 仿写衍生总纲领.md        ✅
║     缺少任一文件将导致 Phase 2 无法启动
╚════════════════════════════════════
```

**解法 B（彻底修）**：在 Phase 1 末尾增加一个 `phase1.5` 步骤，由 automation_manager 调用 destiny_designer 生成 00+01+03+04（全局部分）。02+05（逐章事件+注入包）保留到 Phase 2 卷级生成。

```
Phase 1.5：
  对 active_books 中的每本书：
    检查 00+01+03+04 是否存在
    若不存在 → 调用 @destiny_designer mode=global（只生成 00~04）
    若失败 → 标记状态 "destiny_failed"，不阻塞其他书
    若成功 → 继续
  → 全部完成 → 输出 Phase 1.5 汇总
```

**推荐解法 B，但分两步走**：先实施解法 A（零改动成本），确认管道稳定后迁移到解法 B（配合修复 #8 的分卷策略）。
