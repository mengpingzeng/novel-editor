# Step 自动化处理流程不按顺序处理问题修复

> 日期: 2026-07-07
> 修复范围: `scripts/run_pipeline.sh`, `.opencode/agents/automation_manager.md`

---

## 一、问题描述

执行 `bash ./scripts/run_pipeline.sh step` 时，每本小说没有按照"Phase1 → Phase2 → Phase3"的顺序逐本完成，实际表现为：

- 每本小说执行到 Phase1 后，Phase2 执行到一半就跳到下一本
- Phase3 从来没有被触发

---

## 二、文件系统证据

```
书籍         state.json phase  .phase1_done  v3 Phase1 产物  .phase2_done  v3 正文
──────────────────────────────────────────────────────────────────────────────────
凡人修仙传    pending           v2 (旧)       无 v3 目录       v2 (旧,5章)   v2 有内容
斗破苍穹      pending           v3            ✅(总纲+盐值)    ✅(5章标记)    ❌ 空
甄嬛传        pending           v3            ✅(总纲+盐值)    无             ❌ 空
白夜行        pending           v2 (旧)       仅 salt.json     无             ❌ 空
赘婿          pending           v2 (旧)       无 v3 目录       无             ❌ 空
```

核心矛盾：**`iteration-state.json` 中所有 book 的 phase 都是 `pending`，但磁盘上斗破苍穹和甄嬛传的 v3 已经完成了 Phase1。斗破苍穹甚至 `.phase2_done` 标记为 `chapters_completed: 5`，但 `02-正文/` 目录完全为空。**

---

## 三、根因分析

### 根因 1：agent exit code 不可靠，Phase2 假成功

chief_editor agent 的初始化 SOP (`chief_editor.md` Step 1.5) 强制要求：

> 先调用 `@destiny_designer` 生成注入包，失败就**暂停整个流水线**。没有注入包 = 不允许继续写作。

当 `@destiny_designer` 找不到或执行失败时，chief_editor 文本输出"暂停"后停止运行，但 **`opencode run` 的进程 exit code 仍然是 0**（LLM agent 进程没有崩溃）。

**shell 脚本只看 exit code**，认为一切正常：
```bash
run_chapter "Phase 2.1" ...    # agent 内部失败但 exit 0
# 脚本认为 "成功", 继续章循环
# 最后写入 .phase2_done {"chapters_completed": 5}
# → 实际 02-正文/ 为空!
```

**证据**:
- 斗破苍穹 `.phase2_done` 声称 5 章完成
- `workspace/books/斗破苍穹/versions/v3/02-正文/` **完全为空**
- `workspace/books/斗破苍穹/versions/v3/01-大纲/01-卷纲/` **完全为空**
- `自动化处理日志.md` **不存在**（chief_editor 根本没进入工作状态）

### 根因 2：state 备份恢复模式丢失所有进度

`run_phase1_for_book` (旧代码 L286-335) 使用了"备份→修改→运行→恢复→合并"模式：

```bash
cp "$STATE_FILE" "$state_backup"          # 1. 备份
python3 -c "修改 state"                    # 2. 临时修改
trap "cp 恢复备份" EXIT                    # 3. 设 trap 兜底
run_phase "Phase 1" "/iterate step"        # 4. 运行 agent
new_phase=$(读取当前书 phase)               # 5. 提取 phase
cp "$state_backup" "$STATE_FILE"           # 6. 恢复备份
update_book_phase "$name" "$new_phase"     # 7. 只合并单书 phase
```

问题链：

1. Phase1 agent 运行时，正确处理了斗破苍穹、甄嬛传的 Phase1 并写入 `phase1_done` 到 state
2. Shell 脚本恢复备份 → **斗破苍穹和甄嬛传的 phase1_done 全部丢失**
3. 只合并了当前书（凡人修仙传）的 phase，但它恰好在 agent 的 excluded 列表中 → phase 仍是 `pending`
4. 验证 `phase_ge("pending", "phase1_done")` → **FAIL** → 书标记为失败

**结果**: state 永远被回滚，"已完成"的进度不断丢失，循环往复。

### 根因 3：agent 目录嵌套导致 @destiny_designer 不可用

`automation_manager.md` Step h 的复制命令：

```bash
cp -r project-agents-template/.opencode/agents/ workspace/books/{book}/.opencode/agents/
```

当 v3 在 v2 基础上重跑时，目标目录 `.opencode/agents/` **已存在**，`cp -r` 会把源目录**塞进**目标目录：

```
期望:
  workspace/books/斗破苍穹/.opencode/agents/destiny_designer.md  ← 正确位置

实际:
  workspace/books/斗破苍穹/.opencode/agents/agents/destiny_designer.md  ← 嵌套多一层
```

chief_editor 从 `.opencode/agents/` 启动，`@destiny_designer` 引用解析不到正确路径，导致 Step 1.5 必然失败。

---

### 根因 4：automation_manager agent 指令引用了错误的处理列表

agent 指令 Step 0 写的是：

> 0. **版本号决策**（在处理第一本书之前，**对每本书**执行一次）

"对每本书"导致 agent 遍历 `books` dict（5 本书）而不是 `active_books`（1 本书）。且 Step 1 只说"按列表顺序"没有指定是哪个列表。这是 agent 在日志中自行重写 `active_books` 为 4 本书的根源。

---

## 四、修复方案

### 修复 A：`run_phase1_for_book` — 移除备份恢复模式

**文件**: `scripts/run_pipeline.sh` (L271-312)

删除整个 `state_backup` / `cp 备份` / `trap EXIT` 逻辑。agent 修改 state 后**直接持久化**，不再恢复旧备份。

```
旧: 备份→修改→运行→恢复到备份→只合并单书 phase
新: 修改→运行→直接读取 state→验证 phase≥phase1_done
```

**效果**: agent 批量处理的书（如果你不限制 active_books）其 phase 也会正确保留，断点续跑能正确跳过已完成书。

### 修复 B：Phase 2.0 — 增加卷纲文件校验

**文件**: `scripts/run_pipeline.sh` (L385-393)

chief_editor 运行完成后，检查 `01-大纲/01-卷纲/卷纲-第1卷.md` 是否真实存在：

```bash
local vol_outline="$book_dir/versions/$ver/01-大纲/01-卷纲/卷纲-第1卷.md"
if [ ! -f "$vol_outline" ]; then
    fail "Phase 2.0 agent 返回成功但卷纲文件不存在"
    return 1
fi
```

### 修复 C：Phase 2.x 章节循环 — 每章完成后校验终稿

**文件**: `scripts/run_pipeline.sh` (L407-411)

每章 `opencode run` 返回后，检查 `02-正文/第N章-终稿.md` 是否存在：

```bash
local ch_final="$book_dir/versions/$ver/02-正文/第${ch}章-终稿.md"
if [ ! -f "$ch_final" ]; then
    fail "第${ch}章 agent 返回成功但终稿不存在"
    return 1
fi
```

**效果**: 假成功的章节（exit code 0 但无产出）会立即被识别为失败，不会写出假的 `.phase2_done`。

### 修复 D：dryrun 同样增加输出校验

**文件**: `scripts/run_pipeline.sh` (L510-532)

与 step 模式同理，dryrun 的 Phase2.0 和每个章节完成后都增加了终稿文件存在性验证。

### 修复 E：automation_manager — cp 先删后复制

**文件**: `.opencode/agents/automation_manager.md` (L95-97)

```bash
# 旧: cp -r ... → 已存在目录会嵌套
# 新: rm -rf + cp -r → 确保每次都清空后重新复制
rm -rf workspace/books/{book}/.opencode/agents/ && \
  cp -r project-agents-template/.opencode/agents/ \
  workspace/books/{book}/.opencode/agents/
```

### 修复 F：automation_manager — 明确约束只处理 `active_books`

**文件**: `.opencode/agents/automation_manager.md` (L63-73, L175-181)

**Step 0** 从"对每本书"改为"**仅对 `active_books` 中的书**执行"：

```
旧: 0. **版本号决策**（在处理第一本书之前，对每本书执行一次）
新: 0. **版本号决策**（仅对 `active_books` 中的书执行）
```

**Step 1** 从"按列表顺序"改为"**按 `active_books` 列表顺序**"：

```
旧: 1. 按列表顺序，逐本执行以下步骤
新: 1. 按 `active_books` 列表顺序，逐本执行以下步骤
```

**核心原则**新增第 2 条：

```
新增: 2. **仅处理 `active_books`**：只处理 state 文件中 `active_books` 列表里的书，
      不擅自修改 `active_books` 或处理 `books` 中的其他书
```

**文件头部**增加强制性声明：

```
新增: **重要：你只能处理 `active_books` 中列出的书。不要修改 `active_books` 列表本身。
      不要主动将 `books` 中的其他书加入处理范围。**
```

## 五、关键行为变化对比

| 场景 | 修复前 | 修复后 |
|------|--------|--------|
| chief_editor 内部失败但 exit 0 | 写入 `.phase2_done`，state 更新 phase2_done | **检测到终稿不存在 → 报失败 → 不写标记** |
| Phase1 agent 成功写入 state | **被 backup restore 抹掉**（下个 Phase1 执行时） | **持久保留**，后续书自动跳过已完成阶段 |
| v3 重跑 agent 目录嵌套 | `agents/agents/` 嵌套，`@destiny_designer` 找不到 | 先删后复制，目录结构正确 |
| Phase3 可达性 | Phase2 假成功 → state 被重置 → 永远到不了 Phase3 | Phase2 真成功才写标记 → Phase3 正确触发 |
| agent 自行重写 active_books | 指令歧义，agent 遍历 `books` 全量处理 | **明确约束只处理 `active_books`，禁止修改列表** |

## 六、修复后的完整执行链路

```
step_inner 读取 active_books = [A, B, C, D, E]

  ┌─ 书 A ─────────────────────────────────────┐
  │ Phase1: active_books 临时设为 [A]           │
  │   → agent 被约束只处理 A，写 phase1_done     │
  │ Phase2.0: chief_editor 初始化 → 校验卷纲文件 │
  │ Phase2.1~N: 逐章写 → 每章校验终稿文件        │
  │   → 无终稿 = 失败，不写 .phase2_done         │
  │ Phase3: reviewer_orchestrator 审稿           │
  │ ✅ A 完成 (phase=phase3_done)                │
  └─────────────────────────────────────────────┘

  ┌─ 书 B ─────────────────────────────────────┐
  │ Phase1: phase=phase1_done → 跳过            │
  │ Phase2 → Phase3                             │
  │ ✅ B 完成                                   │
  └─────────────────────────────────────────────┘
  ...
```

**最坏情况兜底**：如果 agent 仍然违反约束（LLM 非确定性），Phase1 验证 `phase_ge(pending, phase1_done)` 会直接失败并报清晰错误：
```
❌ Phase 1 完成但 phase 未更新: pending (agent 可能处理了其他书而未处理 {name}，请检查 agent 输出日志)
```

## 八、补充修复（2026-07-07）：scope 文件隔离 — 彻底阻断 agent 跨书处理

> 修复 A~F 部署后，流水线仍然出现 agent 一轮处理多本书的问题。本节为补充修复。

### 问题复现

修复 A~F 后再次执行 `bash ./scripts/run_pipeline.sh step`，日志显示：

```
step — 5 本书 × 5 章
第 1/5 本: 凡人修仙传 (phase=pending, version=v3)

→ agent 先 Write iteration-state.json（版本号决策）
→ 然后读取 workspace/repo/斗破苍穹, 甄嬛传, 白夜行, 赘婿
→ 为这 4 本书创建 v3 目录
→ 依次处理 斗破苍穹 Phase 1（compliance → style → facade → salt → outline → done）
→ 再处理 甄嬛传 Phase 1（style → facade → salt → outline）...
```

agent 完全忽略了 `active_books = [凡人修仙传]` 的约束，转而处理了 `books` 字典中的全部书。

### 根因

修复 F 只在 prompt 层面加了"仅处理 `active_books`"的文字约束，对 LLM agent 没有强制力。`run_phase1_for_book` 中：

```bash
# 只修改了 active_books，books 字典原封不动包含全部 5 本书
s['active_books'] = [{'name': '$name', ...}]   # 凡人修仙传
s['books']  # ← 仍含 斗破苍穹, 甄嬛传, 白夜行, 赘婿, 凡人修仙传
```

agent 读取 state 文件后看到完整的 `books` 字典，热心地把所有人的 Phase 1 都处理了。

**本质问题**：LLM prompt 约束 ≠ 程序约束。agent 可以随时绕过文字指令。

### 修复 G：scope 文件物理隔离

**核心思路**：`iteration-state.json` 是持久化的断点续写文件，不应被 agent 直接修改。改为：

1. **Shell 创建 scope 文件**：`workspace/.pipeline_scope.json`，结构和主 state 完全一致，但 `active_books` 和 `books` **只包含当前一本书**
2. **Agent 读写 scope 文件**：agent 优先检测并使用 scope 文件，看不到其他书的数据
3. **完成后 Shell 同步**：从 scope 读取当前书的 phase/version，写回主 state

**改动文件**：

#### `scripts/run_pipeline.sh` — `run_phase1_for_book` 重写

```bash
# 旧逻辑（V3）:
#   修改 active_books → 运行 agent → 直接读主 state 的 phase
#   问题: books 字典仍含全部书，agent 遍历全部处理

# 新逻辑（V4）:
#   从主 state 提取当前书 → 创建 scope 文件(仅含当前书) → 运行 agent
#   → 从 scope 读 phase → 同步回主 state（只更新当前书）

SCOPE_FILE="$ROOT_DIR/workspace/.pipeline_scope.json"

# 1. 创建 scope：active_books + books 都只有当前书
python3 -c "
scope = {
    'active_books': [{'name': '$name', ...}],  # 只有当前书
    'books': {'$name': {...}}                   # 只有当前书
}
with open('$SCOPE_FILE', 'w') as f: json.dump(scope, f)
"

# 2. 运行 agent（agent 读 scope 文件，只看到一本书）
run_phase "Phase 1 - $name" "/iterate step"

# 3. 从 scope 读取当前书 phase，同步回主 state
python3 -c "
scope = json.load(open('$SCOPE_FILE'))
s = json.load(open('$STATE_FILE'))
s['books']['$name']['phase'] = scope['books']['$name']['phase']
json.dump(s, open('$STATE_FILE', 'w'))
"
```

#### `.opencode/agents/automation_manager.md` 三处改动

**改动 1：状态文件选择规则**
```
正式模式（step）：
- 优先读取 workspace/.pipeline_scope.json（由流水线脚本自动生成，仅含单书）
- 若 scope 文件不存在（手动执行 /iterate step），回退到 workspace/iteration-state.json
```

**改动 2：模式一中新增状态文件选择**
```
**状态文件选择**：优先使用 workspace/.pipeline_scope.json（若存在），
否则回退到 workspace/iteration-state.json。之后的读写都使用你选择的那个文件。
```

**改动 3：Step i（写入 phase）改用 scope 优先**
```bash
STATE='workspace/.pipeline_scope.json'
if [ ! -f "$STATE" ]; then STATE='workspace/iteration-state.json'; fi
python3 -c "
with open('$STATE') as f: ...
"
```

**改动 4：核心原则新增第 7 条**
```
7. 状态文件写入规则：若 scope 文件存在则写入 scope，不存在则写 iteration-state.json。
   不要同时写两个文件。
```

### 效果对比

| 场景 | 修复 G 前 | 修复 G 后 |
|------|----------|-----------|
| agent 能否看到其他书 | ✅ 能（`books` 字典含全部书） | ❌ 不能（scope 文件只有当前书） |
| agent 能否处理其他书 | ✅ 能（prompt 约束无强制力） | ❌ 不能（物理上看不到数据） |
| 手动执行 `/iterate step` | 正常 | 正常（回退到 iteration-state.json） |
| `iteration-state.json` 完整性 | 可能被 agent 篡改 | 始终由 shell 脚本维护，agent 不直接写 |

### 设计原则

- **`iteration-state.json`**：持久化的断点续写主文件，只有 shell 脚本有权写入
- **`workspace/.pipeline_scope.json`**：每次 Phase 1 调用时由 shell 生成、agent 读写的临时作用域文件
- **单向同步**：scope → 主 state（shell 读取 scope 后写回主 state），绝不反向
