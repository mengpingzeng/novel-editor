## 二、管线架构概述

### 2.1 三阶段流水线

```
Phase 1 (Framework)  → automation_manager (根级 .opencode/agents/)
Phase 2 (Content)    → chief_editor       (per-book .opencode/agents/)
Phase 3 (Review)     → reviewer_orchestrator (workspace/reviewer/.opencode/agents/)
```

### 2.2 关键 Agent 及职责

| Agent | 层级 | 模型 | 职责 |
|-------|------|------|------|
| `master_outline_generator` | Phase 1 | GLM 5.2 | 生成仿写衍生总纲 |
| `salt_architect` | Phase 1 | — | 生成 project_salt.json |
| `destiny_designer` | Phase 2 | — | 生成上帝之眼（全书命运注入包） |
| `plot_planner` | Phase 2 | V4 Flash | 生成卷纲 + 章纲 |
| `content_writer` | Phase 2 | V4 Flash (temp 0.6) | 撰写正文初稿/重写 |
| `quality_reviewer` | Phase 2 | GLM 5.2 | 章内容 100 分制评分 + 修改建议 |
| `chief_editor` | Phase 2 | V4 Flash | Phase 2 总协调，含 score-loop |
| `reviewer_orchestrator` | Phase 3 | deepseek-v4-pro | Phase 3 总协调 |
| `input_monitor` | Phase 3 | V4 Flash | 输入膨胀监控 |

### 2.3 关键流程

**Quality Loop（chief_editor 中的评分重写循环）**：
```
content_writer (fresh) → quality_reviewer → 评分
  ├── ≥60: 通过，promote 到终稿
  ├── <60: 触发 rewrite（最多 2 次）
  └── 取最高分版本
```

**当前纪要生成流程**：
```
每章纪要为 quality_reviewer 的完整输出（~25KB/章）
Ch5 写作时拼接 Ch1~Ch4 全部纪要（累积 66.7KB）
```

### 2.4 关键架构约束

1. **Phase 1↔2 时序割裂**：总纲在 Phase 1 生成，上帝之眼（含终卷详细设计）在 Phase 2 才生成
2. **伏笔跨卷全量追踪**：伏笔系统依赖全量跨章状态（FS-C1 埋在 Ch1，回收在 Ch29）
3. **模型能力梯度**：执行层用 V4 Flash（temp 0.6），评审审稿层用 GLM 5.2

## 三、全部 12 项优化方案的可行性分析

改动发生在四个层面：
- **Agent Prompt 层**：修改 agent 指令 `.md` 文件（5 项）
- **Chief Editor 流程层**：修改 chief_editor 的 bash 验证步骤（3 项）
- **管线架构层**：修改跨 Phase 的输入传递机制（2 项）
- **上游模板层**：修改 salt / 注入包 / 总纲的输出规范（2 项）

### 3.1 Agent Prompt 层

#### 方案 ①：卷纲补齐 forbidden_patterns 章节

| 维度 | 内容 |
|------|------|
| **改什么** | `project-agents-template/.opencode/agents/plot_planner.md` §1.2 卷纲模板 |
| **当前状态** | 模板第 217 行有 `【本卷受限模式】` 占位符，但实际生成的卷纲中 forbidden_patterns 出现 0 次 |
| **根因** | 模板只给两行空白占位，agent 在生成大量结构后容易跳过短段落。salt 有 4 条 forbidden_patterns，设计中已规避，但未显式"举证" |
| **方案** | 将占位符改为必须填写的检查表格式。每行强制填写"禁止项 → 本卷规避方案"，不可留空。增加指令："如无可规避方案则标注'待卷纲审核确认'" |
| **可行性** | ⭐⭐⭐⭐⭐ 极高。纯 prompt engineering，零代码改动 |
| **预期效果** | 卷纲 D4 从 8/15→13-15/15，卷纲总分 87→92-94 |
| **风险** | 无 |

#### 方案 ②：总纲禁止清单补充正反例

| 维度 | 内容 |
|------|------|
| **改什么** | `.opencode/agents/master_outline_generator.md` |
| **当前状态** | 总纲 §10 有 6 条禁止项，每条仅有规则描述。"少内心独白""适配移动端"未量化。v2 报告 P8 已指出但 v3 未修复 |
| **根因** | generator prompt 没有要求为每条禁止项输出正反例对照，也未量化模糊表述 |
| **方案** | 在 generator 的输出规范中，§10 每条必须输出三行：`[规则]` + `[✅ 正例]` + `[❌ 反例]`。将"少内心独白"→"内心独白占比 ≤15% (约 300字/章)"；"适配移动端"→"段落长度 80-150 字 (3-5 句)，单行不超过 25 字" |
| **可行性** | ⭐⭐⭐⭐⭐ 极高。纯 prompt engineering |
| **预期效果** | 总纲 D1 明确性 21→23-24/25，总纲总分 86→90-92 |
| **风险** | agent 生成的正反例需要人工抽检质量（首次上线建议抽查） |

#### 方案 ③：场景原创度强化

| 维度 | 内容 |
|------|------|
| **改什么** | `plot_planner.md`（章纲模式）+ `project_salt.json`（新增字段） |
| **当前状态** | L7 维度 9 扣 1 分：开篇"公开判定废物"场景骨架与原著相似。章纲 §10 差异化措施覆盖角色/爽点/节奏但未覆盖场景骨架 |
| **核心限制** | plot_planner 在章纲层面没有读取原著白皮书——它的输入是注入包事件链 + 总纲约束，无法自行判断"这个场景是否跟原著太像"。这是 prompt-only 方案效果最差的一项 |
| **方案** | 分两层：①在 salt `anti_similarity` 中新增 `scene_skeleton_avoid` 字段，由 salt_architect 从白皮书 §五 提取 5-10 个原著高辨识度场景骨架作为黑名单；②plot_planner 章纲 §10 差异化措施中逐条对照黑名单自查 |
| **可行性** | ⭐⭐⭐ 中。关键是 salt_architect 能否可靠提取原著场景骨架（需要增加对应的 prompt 约束） |
| **预期效果** | L7 维度 9 从 11→12/12，签约审稿 96→97 |
| **风险** | 如果 salt_architect 提取的场景骨架不准确或不全面，plot_planner 对照检查形同虚设 |

#### 方案 ④：拟声词/句式变异轮换

| 维度 | 内容 |
|------|------|
| **改什么** | `content_writer.md` |
| **当前状态** | "轰"在 Ch1/Ch2/Ch4 重复 3 次，"差得远"在 Ch3/Ch4 连续两章 |
| **方案** | 在 content_writer 的"AI 味消除"清单中增加：相邻 5 章不得重复使用同一拟声词。提供变异词库（轰→嗡/咚/砰，震颤→战栗/酥麻，差得远→还有差距/不是一个量级）。建议由 chief_editor 在每章写作前动态生成"前 5 章已用拟声词黑名单"并注入 content_writer prompt |
| **可行性** | ⭐⭐⭐⭐ 高。少量 prompt 改动 + 静态词库 |
| **预期效果** | L7 维度 4 从 11→12/12 |
| **风险** | 变异词库需要人工维护（不同赛道用词不同） |

#### 方案 ⑤：对话占比弹性规则

| 维度 | 内容 |
|------|------|
| **改什么** | `master_outline_generator.md` + `plot_planner.md` + `content_writer.md`（跨 3 个 agent） |
| **当前状态** | 总纲 §7.1 硬性要求全部章对话占比 ≥40%。Ch2（签到探索+魂毒发作）天然叙事占比高，与约束存在结构性矛盾 |
| **方案** | 总纲增加："战斗/修炼/独处场景章 ≥35%，全卷均值 ≥40%"。plot_planner 在章纲 §一 元数据中增加"章节类型"标注。content_writer 根据章类型执行不同对话占比约束 |
| **可行性** | ⭐⭐⭐ 中。跨 3 个 agent 的联动改动，任何一个漏了规则就不生效。需要一次性全部修改并确保 prompt 中互相引用 |
| **预期效果** | 解决 Ch2/Ch3 类结构性矛盾，章内容质量一致性提升 |
| **风险** | 联动失败风险——3 个 agent 改漏一个则规则不生效 |

### 3.2 Chief Editor 流程层

#### 方案 ⑥：禁令文本硬检查

| 维度 | 内容 |
|------|------|
| **改什么** | `chief_editor.md` + 新增 Python 检查脚本 |
| **当前状态** | Ch3 L177 出现"血誓殿"，违反注入包 §6.1。章纲 §10.3 已标注约束但 content_writer 未执行。quality_reviewer 发现了但只是打分 |
| **为何不在 agent 层改** | content_writer 用 V4 Flash (temp 0.6)，它连章纲标题都能写错，指望它自觉遵守禁词表不可靠。必须用硬检查（非 prompt） |
| **方案** | 在 chief_editor 中，content_writer 生成初稿后、进入 quality_reviewer 之前，插入 bash 脚本：从注入包 §6 解析禁词列表，正则扫描正文全文，命中则阻断并自动触发 rewrite，rewrite prompt 中注入"上一稿因禁词 XXX 被驳回" |
| **可行性** | ⭐⭐⭐⭐ 高。~30 行 Python，核心逻辑简单 |
| **预期效果** | 杜绝禁令违反，流程审计断层 #1 归零 |
| **风险** | 禁词表需要从注入包中可靠解析（当前注入包 §6 是 Markdown 表格格式，不同书格式可能有差异） |

#### 方案 ⑦：标题一致性校验

| 维度 | 内容 |
|------|------|
| **改什么** | `chief_editor.md` 的 bash 验证步骤 |
| **当前状态** | Ch2 章纲标题"第一道裂缝"→正文标题"古丹之声"，无任何对比环节 |
| **讨论后修正** | 告警策略从"不匹配就阻断"改为"不匹配时将章纲标题作为 hard constraint 注入 quality_reviewer 的检查维度"——让 reviewer 判断这个命名改动是否合理，而非粗暴阻断 writer 可能的创造性优化 |
| **方案** | bash 脚本提取章纲元数据 `章节标题` + 正文 `# 第X章 XXX` 的标题部分，比对。不一致时将差异信息注入 quality_reviewer prompt，增加检查项"正文标题是否比章纲标题更优？" |
| **可行性** | ⭐⭐⭐⭐⭐ 极高。约 15 行 bash |
| **预期效果** | 流程审计断层 #2 归零 |
| **风险** | 几乎无风险。告警不阻断流程，仅增加 reviewer 检查项 |

#### 方案 ⑧：质量门禁 SLA 改进

| 维度 | 内容 |
|------|------|
| **改什么** | `chief_editor.md` 的 score-loop 逻辑 |
| **当前状态** | hard gate 是 60 分（及格线），<60 触发重写。Ch2(79)/Ch3(76) ≥60 直接通过 |
| **方案** | 分层策略：≥80→通过；70-79→通过但将扣分项注入下一章 writer 额外约束；<70→强制重写；<50→阻断通知 chief_editor |
| **可行性** | ⭐⭐⭐ 中。改数字简单，但重写要生效需同时修改纪要到 writer 的反馈链路 |
| **预期效果** | Ch2/Ch3 类低分章节得到实际修复 |
| **关键依赖** | 强依赖方案 ⑥（禁令检查）：如果禁令没被拦截，重写只是复制同样的错误。强依赖方案 ⑩（纪要闭环）：重写时 writer 需要知道具体要改什么 |

### 3.3 管线架构层

#### 方案 ⑨：滑动窗口 + 伏笔状态滚动摘要

| 维度 | 内容 |
|------|------|
| **改什么** | `chief_editor.md` + `content_writer.md` + `quality_reviewer.md` |
| **当前状态** | 每章拼接全部历史纪要，形成"纪要越大→输入越大→新纪要更大"的恶性循环。Ch5 正文输入 145KB（首章 3.41 倍），纪要输入 138KB（4.08 倍） |
| **核心矛盾** | 纯滑动窗口（只读 2-3 章纪要）会切断跨章伏笔追踪——伏笔 FS-C1 埋于 Ch1 预测 Ch29 回收，Ch20 的 writer 只能看到 Ch17-19 纪要就不知道这笔伏笔 |
| **方案** | 拆分为两条信息通道：**细则通道（短窗口）**：正文撰写只读取当前章纲 + 最近 2 章纪要 + 最近 1 章终稿。**骨架通道（滚动摘要）**：维护独立 `伏笔状态滚动摘要.md`，每章追加一行（伏笔ID/状态/预期回收章），正文撰写前必须读取。**纪要压缩**：纪要 ≤15KB/章，通过项不逐条列出 |
| **可行性** | ⭐⭐⭐ 中偏高。需要精心设计伏笔摘要格式和 chief_editor 的上下文拼接逻辑 |
| **预期效果** | Ch5 正文输入从 145KB→~60KB，消除 341% 膨胀。输入监控 35→70-80 |
| **风险** | 伏笔摘要格式设计不当会导致信息丢失；纪要压缩后 quality_reviewer 的修改建议可能不够详细 |

#### 方案 ⑩：纪要修改建议闭环化

| 维度 | 内容 |
|------|------|
| **改什么** | `chief_editor.md` 的 score-loop 逻辑 |
| **当前状态** | quality_reviewer 的修改建议仅用于打分。终稿中所有建议（改"血誓殿"措辞、精简字数、修正标题）均未执行 |
| **方案** | 评分 < 90 时，从纪要中提取"扣分项描述"，格式化为 content_writer rewrite 的输入约束。格式例如：`[本轮重写必须修复] 1. 删除L177"血誓殿"替换为"行省分殿" 2. 标题改回章纲指定的"第一道裂缝" 3. 精简字数至2200以内` |
| **可行性** | ⭐⭐⭐ 中。需要纪要输出包含机器可解析的修正建议（而非散落在自然语言描述中） |
| **预期效果** | 质量反馈闭环建立，"发现→修复→验证"三步骤完整 |
| **关键依赖** | 强依赖方案 ⑧（门禁 SLA）：需要 SLA 来驱动重写触发。强依赖方案 ⑧ 的分层策略：70-79 分时注入下一章约束，<70 分时触发本章重写 |

### 3.4 上游模板层

#### 方案 ⑪：总纲新增结局设计章节

| 维度 | 内容 |
|------|------|
| **改什么** | `master_outline_generator.md` + `chief_editor.md`（跨 Phase 回填） |
| **当前状态** | 总纲无 §十一·结局设计。C1 扣 2 分 |
| **根本矛盾** | 总纲在 Phase 1 生成，上帝之眼的全书命运总谱在 Phase 2 才生成。总纲生成时没有终卷详细设计 |
| **方案** | 分两步：①Phase 1 时 generator 输出概要级结局设计（高潮类型 + 情感落点方向），基于 salt 的 `volume_position_variants.final_volume`；②Phase 2 destiny_designer 完成后，chief_editor 从 `00-全书命运总谱.md` 提取终卷事件链回填总纲 §十一 |
| **可行性** | ⭐⭐☆ 中偏低。跨 Phase 写操作增加架构复杂度 |
| **预期效果** | C1 从 5/7→6-7/7，流程审计 +1-2 分 |
| **风险** | 跨 Phase 回填可能破坏 Phase 1 产物的完整性 |
| **建议** | 推迟到 v5+，当前终卷离首卷 5 章有 1200+ 章距离，改动风险远高于收益 |

#### 方案 ⑫：salt 增加场景骨架黑名单

| 维度 | 内容 |
|------|------|

## 四、耦合关系图

```
     禁令检查(⑥) ──┬── 门禁SLA(⑧) ── 纪要闭环(⑩)
                     │
                     └── 标题校验(⑦)
```

⑥ 是 ⑧ 的前置——只有禁令先被拦截，SLA 的重写才有意义。

```
     滑动窗口(⑨) ──── 纪要闭环(⑩)
```

⑨ 改变输入结构，⑩ 改变输出使用方式——纪要闭环依赖纪要文件格式，如果滑动窗口改了纪要格式，闭环的"扣分项提取"逻辑需要同步。

```
     场景原创度(③) ──── salt黑名单(⑫)
```

③ 负责检查，⑫ 提供检查用的数据。

```
     对话弹性(⑤) ──── 需要章纲类型标注 ← plot_planner 改动
     跨 3 个 agent (master_outline_generator / plot_planner / content_writer)
```
