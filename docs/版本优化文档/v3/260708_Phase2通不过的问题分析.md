## 三、问题 1：为什么书A Phase2 失败后会跑到书B 的 Phase1？

### 答案：while 循环的设计如此

`step_inner()` 函数（`scripts/run_pipeline.sh:638-660`）：

```bash
while IFS='|' read -r name platform track; do
    result=$(run_book_pipeline "$name" "$platform" "$track" "$chapters")
    local rc=$?
    if [ $rc -eq 0 ] && [ -n "$result" ]; then
        completed_books="$completed_books $name"     # 成功：入成功列表
    else
        failed_books="$failed_books $name"           # 失败：入失败列表，继续下一本
    fi
done <<< "$books_data"
```

关键点：

- `while` 循环遍历全部 `active_books`，每本书调用 `run_book_pipeline`
- `run_book_pipeline` 串行执行 Phase1→2→3（函数内是顺序的）
- 但 `run_book_pipeline` **返回非零时不会 break**——`while` 循环继续读下一行，处理下一本书
- 最终汇总：`completed_books` 和 `failed_books` 分别列出

**这不是状态管理的 bug**。JSON 状态正确地记录了斗破苍穹 `phase1_done`，后续调用 `get_book_phase` 时也能正确返回。问题在于：书A 的 Phase2 **失败**后，循环没有停下来，而是继续书B 的 Phase1。这对用户而言就是"书A 没做完就去做书B"。

### 这种设计的意图

循环的语义是 "尽最大努力处理所有书，最后报告结果"，不是 "一本不通就全停"。如果用户期望的是 "斗破苍穹 Phase2 失败就停止、修复后再继续" 的交互模式，当前设计不满足。

## 四、问题 2：`destiny_designer` 是什么，为什么 Phase2 通不过

### 4.1 它在流程中的位置

V3 的 `chief_editor` agent（`chief_editor.md:59-106`）在 Phase 2.0 "初始化项目并生成第1卷卷纲" 时，强制执行 **Section 1.5：加载本卷注入包（强制·v3 上帝视角命运设计）**：

```
Phase 2.0 启动
  → 检查 versions/v3/上帝之眼/05-卷级注入/卷01-注入包.md 是否存在
  → 若不存在 → 检查 Phase1 的 3 个输入文件是否齐全
  → 齐全 → 调用 @destiny_designer
  → 若 @destiny_designer 失败或自检不通过 → 暂停流水线 → Phase 2.0 失败
```

铁则原文（`chief_editor.md:61`）：
> ⚠️ 铁则：没有注入包 = 不允许继续写作。不确定性生成在 v3 中被视为不符合质量标准。

**v2 的 Phase 2.0 没有这一步**——v2 直接读取总纲领生成卷纲即可。v3 新增了一个必须通过的关卡。

### 4.2 `destiny_designer` 的任务规格

它是一个 subagent（`destiny_designer.md` 共 386 行），要求在 **单次 agent session 内** 为全书生成完整的"上帝之眼"目录，共 8 步：

| 步骤 | 输出文件 | 规模 |
|------|---------|------|
| 步骤 1 | 提取总卷数 | 从总纲领/盐值推导，找不到 → 直接终止 |
| 步骤 2 | `00-全书命运总谱.md` | 三幕结构 + 全部角色命运坐标系 + 信息不对称设计 + 结局设计 |
| 步骤 3 | `01-人物命运谱/` | **每个命名角色**一个独立文件（性格底层逻辑 + 逐卷人生表 + 关系交叉 + 独立故事线） |
| 步骤 4 | `02-剧情命运谱/` | **每卷一个文件**，卷内逐章确定性事件链，每行核心事件 ≥20 字，禁止占位符 |
| 步骤 5 | `03-伏笔命运谱/全书伏笔网络.md` | 三级伏笔网络（A/B/C），每个伏笔的埋设→暗示→回收轨迹 |
| 步骤 6 | `04-世界观展开谱/世界观展开节奏.md` | 逐卷揭示矩阵 + 不可提前揭示清单 |
| 步骤 7 | `05-卷级注入/` | 每卷一个注入包，7 个固定章节，不含未来卷信息 |
| 步骤 8 | `_自检报告.md` | 5 大类检查，**全部 PASS 才允许完成**，最多 3 轮修正 |

最终输出目录结构（`destiny_designer.md:332-353`）：

```
versions/v3/上帝之眼/
├── 00-全书命运总谱.md
├── 01-人物命运谱/        （N 个命名角色文件）
├── 02-剧情命运谱/        （8 卷文件，每卷 ~80 行事件链）
├── 03-伏笔命运谱/全书伏笔网络.md
├── 04-世界观展开谱/世界观展开节奏.md
├── 05-卷级注入/          （8 卷注入包文件）
└── _自检报告.md
```

### 4.3 为什么它几乎不可能完成

**原因 1：单次 session 输出量远超模型限制**

- 8 卷 × 80 章/卷 = 640 章。每章事件 ≥20 字 + 伏笔/角色字段 + 爽点类型，仅步骤 4 就要输出数万字
- 步骤 3 需为 20+ 个命名角色各生成独立文件（每个数百字）
- 步骤 7 需为 8 卷各生成独立注入包
- 总计 13+ 个文件，预估输出总量可达 10 万字符以上
- 模型单次输出 token 有上限，中途截断则全部失败

**原因 2：自检（步骤 8）要求跨文件全局一致性**

自检项（`destiny_designer.md:281-319`）：

```markdown
## B. 逐章细化度检查
卷 XX：事件链行数 = __，预期章数 = __ → [PASS / FAIL]

## D. 跨文件一致性
- 00 声明总卷数 = 02 文件数 = 05 文件数
- 00 中声明的角色在 01 中均有对应文件
- 每个伏笔 ID 在至少 1 个 02 卷中有"埋设" + 至少 1 个 02 卷中有"回收"

## E. 叙事一致性
- 角色死亡卷之后，02/05 中不再出现该角色
- 境界递进跨卷线性，无回退
- 关键物品跨卷追踪无断裂
```

在单次 session 内生成大量关联内容，越往后模型越容易遗忘早期约束（如在步骤 4 中遗忘步骤 3 设定的角色状态，或伏笔 ID 跨卷不一致）。最多 3 轮修正，3 轮后仍未通过 → 标记失败。

**原因 3：步骤 1 即可直接阻断**

如果总纲领或盐值中无法提取到明确的"全书 N 卷"数字，`destiny_designer` 立即返回错误（`destiny_designer.md:84-91`），不生成任何文件。`chief_editor` 收到后暂停流水线——Phase 2.0 直接失败。

**原因 4：`run_phase` 的 1 小时超时**

`run_pipeline.sh:65-67`：
```bash
if [ "$TIMEOUT_PER_PHASE" -gt 0 ]; then
    timeout "$TIMEOUT_PER_PHASE" opencode run ...
```

默认 `TIMEOUT_PER_PHASE=3600`（1 小时）。`destiny_designer` 生成如此大量的内容，很可能超过 1 小时，导致超时退出。

### 4.4 与其他问题的交叉

前置文档 `260707_step自动化处理流程不按顺序处理问题修复.md` 中已修复了修复 B/C：
- Phase 2.0 结束后校验卷纲文件是否存在
- 每章结束后校验终稿是否存在

但这些校验只能检测产出缺失，**不能绕过 `destiny_designer` 本身**。即使卷纲校验 `[ ! -f "$vol_outline" ]` 能正确报错，它也只证明了 Phase 2.0 失败了，而不是帮它成功。

## 五、总结

| 层面 | 结论 |
|------|------|
| 状态管理（`iteration-state.json` + scope 文件） | ✅ 正确。Phase 区分、版本号决策、断点续跑跳过均可正常工作 |
| `run_book_pipeline` 内 Phase 串行逻辑 | ✅ 正确。Phase1→2→3 是函数内顺序执行的 |
| 书A 失败后跳到书B | ⚠️ 设计如此。`while` 循环不因单书失败 break，而是遍历全部汇报结果 |
| Phase2 实际可通过性 | ❌ `destiny_designer` 在单次 session 内的输出量和自检要求远超模型实际能力，是当前 Phase2 的必然阻断器 |
| 修复 A~G 的覆盖盲区 | 修复 A~G 解决了 state 回滚、agent 跨书处理、假成功标记等问题，但未涉及 `destiny_designer` 本身的可行性 |

### 核心矛盾

V3 在 Phase1 和 Phase2 之间插入了一个"必须生成全书完整命运设计书"的关卡（`destiny_designer`），而该关卡在当前技术条件（单 agent session 的 token 限制、上下文遗忘、自检严格度）下基本无法通过。Phase1 输出再完整，Phase2 也通不过——这就是**没有任何一本书能跑完所有 phase** 的根因。
