# V3 断点续跑优化 — 基于 iteration-state.json 的状态驱动

> 日期：2026-07-07 | 触发原因：标记文件方案无法满足"手动修改版本号重新运行"的需求

---

## 一、问题背景

### 旧方案（标记文件）

断点续跑完全依赖文件系统标记：
- `workspace/books/{书名}/.phase1_done` → Phase 1 跳过
- `workspace/books/{书名}/.phase2_done` → Phase 2 跳过
- `workspace/books/{书名}/versions/v{N}/.phase3_done` → Phase 3 跳过

**致命缺陷**：如果要修改版本号从头跑（比如 `v2 → v3`），必须手动删除所有书的标记文件，或者逐本删除。操作繁琐且容易遗漏。

### 新方案（state JSON 驱动）

所有断点续跑信息集中在 `workspace/iteration-state.json` 一个文件中，标记文件保留作为兜底兼容。

```json
"books": {
    "凡人修仙传": {
        "version": "v3",        // 目标版本号：agent 读取此值创建对应版本目录
        "phase": "pending",     // 阶段：pending | phase1_done | phase2_done | phase3_done | done
        "score": null,          // 签约审稿分数（Phase 3 后自动填充）
        "passed": false         // 是否通过审稿（Phase 3 后自动填充）
    }
}
```

## 二、核心设计

### 2.1 阶段流转（phase 状态机）

```
pending ──→ phase1_done ──→ phase2_done ──→ phase3_done ──→ done
   │              │              │              │
   │ Phase 1 完成  │ Phase 2 完成  │ Phase 3 完成  │ (可手动标记为 done 不再重跑)
```

### 2.2 谁写什么时机

| 阶段 | 写入者 | 写入时机 | 写入字段 |
|------|--------|---------|---------|
| phase1_done | `automation_manager` agent | Phase 1 框架生成 → 写入 `.phase1_done` 标记时同步 | `phase`, `version` |
| phase2_done | `run_pipeline.sh` bash 脚本 | 全部章节循环完成后 | `phase` |
| phase3_done | `reviewer_orchestrator` agent | Phase 3 审稿 → 写入 `.phase3_done` 标记时同步 | `phase`, `score`, `passed` |

### 2.3 读写策略

- **读优先**：`get_book_phase` 优先读 `iteration-state.json`，如果 `phase` 字段存在（包括 `"pending"`），以 JSON 为准
- **写双轨**：标记文件和 JSON 同时写入（防止 JSON 损坏时丢失状态）
- **兜底**：如果 `phase` 字段在 JSON 中不存在（旧数据），fallback 到标记文件

### 2.4 版本号决策

`automation_manager` 读取 `books.{name}.version`：

1. **已设置且非空**（如用户设为 `"v3"`）→ 直接使用，不自动递增
2. **未设置或为空** → 扫描 `versions/` 目录取最大号 +1，写入 JSON

这样用户可以：
- 显式写 `"version": "v3"` → agent 用 v3 创建文件夹
- 留空 → agent 自动递增分配

## 四、使用手册

### 4.1 场景一：正常首次运行

```json
// workspace/iteration-state.json
"凡人修仙传": {"version": "v3", "phase": "pending", "score": null, "passed": false}
```

直接运行 `bash scripts/run_pipeline.sh step`，所有阶段从头执行。

### 4.2 场景二：断点续跑（Phase 1 已完成，继续 Phase 2/3）

脚本自动读取 `phase` 字段，跳过已完成阶段。

```json
// 自动回写后：
"凡人修仙传": {"version": "v3", "phase": "phase1_done", "score": null, "passed": false}
// → Phase 1 SKIP, Phase 2/3 RUN
```

### 4.3 场景三：代码优化后，指定新版本重头跑

**只需改 JSON 一行**：

```json
// 改之前
"凡人修仙传": {"version": "v2", "phase": "phase3_done", "score": 78, "passed": true}

// 改之后
"凡人修仙传": {"version": "v3", "phase": "pending", "score": null, "passed": false}
```

运行 `bash scripts/run_pipeline.sh step` 即从 v3 从头开始。

### 4.4 场景四：仅重跑审稿

```json
"凡人修仙传": {"version": "v3", "phase": "phase2_done", "score": null, "passed": false}
```

Phase 1/2 跳过，Phase 3 重新审稿。

### 4.5 场景五：标记某书为"已完成"

```json
"凡人修仙传": {"version": "v3", "phase": "done", "score": 85, "passed": true}
```

全阶段跳过。

## 五、兼容性

- 标记文件仍然保留写入（双轨），作为 JSON 损坏时的兜底
- `phase` 字段不存在（旧格式 JSON）→ fallback 到标记文件，旧流程不受影响
- `run_phase1_for_book` 的备份/恢复机制保留（避免 agent 写到临时 state 时丢失多书数据）

---

## 六、风险与边界

| 风险 | 等级 | 缓解措施 |
|------|------|---------|
| JSON 写入中途崩溃导致格式损坏 | 低 | 双写标记文件兜底，写入前不删备份 |
| agent 未按预期更新 state JSON | 中 | 脚本在 Phase 1 和 Phase 3 完成后验证 phase 是否更新；Phase 2 由脚本自身直接写入 |
| phase 语义歧义 | 低 | `phase_ge` 函数用严格枚举匹配，不依赖字符串比较 |

## 七、验证结果

```
bash -n scripts/run_pipeline.sh → Syntax OK

get_book_version("凡人修仙传") → "v3" (from JSON)
get_book_phase("凡人修仙传") → "pending" (from JSON, respects pending)

phase_ge("pending", "phase1_done") → false → RUN Phase 1
phase_ge("phase1_done", "phase1_done") → true → SKIP Phase 1
phase_ge("phase2_done", "phase3_done") → false → RUN Phase 3
phase_ge("phase3_done", "phase2_done") → true → SKIP Phase 2
```
