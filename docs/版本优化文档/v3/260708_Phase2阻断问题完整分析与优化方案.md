## 二、状态管理与循环设计的分析

### 2.1 状态管理：正确

`iteration-state.json` + scope 文件机制工作正常。`get_book_phase` 正确读取 `phase` 字段，Phase 跳过判断（`phase_ge`）逻辑无 bug。`run_phase1_for_book` 的 scope 创建→agent 执行→scope 同步→主 state 更新的链路在斗破苍穹上完整执行成功（状态从 `pending` → `phase1_done`）。

### 2.2 书A失败后跳到书B：设计如此，非 bug

`step_inner()` 的 `while` 循环（`run_pipeline.sh:638-660`）：

```bash
while IFS='|' read -r name platform track; do
    result=$(run_book_pipeline "$name" "$platform" "$track" "$chapters")
    local rc=$?
    if [ $rc -eq 0 ] && [ -n "$result" ]; then
        completed_books="$completed_books $name"
    else
        failed_books="$failed_books $name"     # 失败→记入失败列表，while继续
    fi
done <<< "$books_data"
```

循环内没有任何 `break` 或 `return`。意图是"遍历所有书、每本书尝试完整跑、最后汇总报告"，不是"一本不通就全停"。`run_book_pipeline` 内部是 Phase1→2→3 串行的，但一本书的失败不阻断后续书的处理。

### 2.3 Phase2 的阻断点：`destiny_designer`

V3 的 `chief_editor` 在 Phase 2.0 时强制执行 Section 1.5（`chief_editor.md:59-106`）：

```
Phase 2.0 启动
  → 检查 versions/v3/上帝之眼/05-卷级注入/卷01-注入包.md 是否存在
  → 若不存在 → 检查 Phase1 的 3 个输入文件是否齐全
```

铁则原文：**"没有注入包 = 不允许继续写作"**。

`run_pipeline.sh:397-404` 中 Phase 2.0 校验卷纲文件存在性，但这个校验只能检测产出缺失，不能绕过 `destiny_designer` 本身。

## 三、dryrun 能通过但 step 不通过的根因

### 3.1 关键证据：两个版本的 `destiny_designer.md`

| 文件 | 时间 | Step 1 行为 | 步骤数 |
|------|------|-----------|--------|
| dryrun v4 `.opencode/agents/destiny_designer.md` | **Jul 7 17:40** | 直接生成"00-全书命运总谱"（**自主设计**卷数） | 7 |
| root + template `destiny_designer.md` | **Jul 7 19:00** | 先提取总卷数，"找不到→禁止继续"（**被动查找**） | 8 |

斗破苍穹的 `destiny_designer.md`（Jul 8 10:06 复制）与 root 完全一致——拿到了新版。

### 3.2 时间线

```
Jul 7 17:40   dryrun v4 Phase1 复制 agents ← 旧版 destiny_designer (7步)
Jul 7 17:41   dryrun v4 Phase2 → chief_editor → @destiny_designer (旧版) → ✅ 1卷25章
Jul 7 19:00   project-agents-template 更新 → destiny_designer 升级为 8步严格版
Jul 8 10:06   step 模式 Phase1 复制 agents ← 新版 destiny_designer (8步)
Jul 8 10:06+  step 模式 Phase2 → chief_editor → @destiny_designer (新版) → ❌
```

### 3.3 新版 Step 1 为什么阻断了

新版 `destiny_designer` Step 1 要求从两个来源提取总卷数：

```
1. 从总纲领查找 "全书.*卷粗略划分" → 找不到（master_outline_generator 输出中没有此节）
2. 从 salt.json 推断 → 只有 first_volume + final_volume 变体 → "至少2卷"，推不出具体数字
3. → ❌ 无法确定总卷数 → 禁止继续
```

`master_outline_generator` 的输出格式（10 节，`master_outline_generator.md:92-271`）没有"全书 N 卷粗略划分"节。最接近的第 7 节"节奏模型"和第 9 节"剧情模板"都不声明总卷数。而白皮书 §一（全书结构地图）**确实有**原作的卷/部划分数据，但未被传递到总纲领中。

### 3.4 真正的架构矛盾

这不是简单的"数据缺失"问题。`destiny_designer` 是唯一同时持有以下全部信息的 agent：

| 信息来源 | 提供的数据 | 与卷数决策的关系 |
|---------|-----------|---------------|
| 白皮书 §一 | 原作卷/部划分、总章数、转折点 | 参考基准 |
| 白皮书 §五 | 原作大/小高潮间隔 | 参考基准 |
| 盐值 rhythm_diff | 节奏调整幅度（大高潮间隔 -40%、日常章 -52% 等） | 缩放因子 |
| 盐值 volume_rhythm_profile | phases、pacing_signature、default_volume_length | 卷内结构参数 |
| 总纲领 §二 | 平台目标字数预算 | 总量约束 |

**没有上游 agent 同时拥有这些全部信息来做卷数决策。** `master_outline_generator` 看到的是白皮书 + 盐值 + 平台规则，但缺少最终总字数的交叉计算。`salt_architect` 定义节奏参数但不知道实际会用多少卷来实现。

旧版 `destiny_designer` 的 Step 1 是"**设计**卷数"——它基于输入自行推断，这是符合上帝视角定位的行为。新版 Step 1 把设计降级为"**查找**卷数"，要求别人写好给自己，这是方向性的退化。

## 四、优化方案

### 4.1 目标

恢复 `destiny_designer` 的上帝视角设计权。Step 1 从"被动查找"转为"主动计算"，同时保留数据必须有来源的约束。

### 4.2 改动范围

**只改一个文件**：`.opencode/agents/destiny_designer.md` 的 Step 1。

**不需要改的文件**：

| 文件 | 原因 |
|------|------|
| `master_outline_generator.md` | 它不应拥有卷数决策权——它缺少盐值节奏参数和总字数预算 |
| `project_salt.json` / `salt_architect` | 盐值提供节奏参数（每卷章数、高潮间隔），不提供总卷数 |
| `iteration-state.json` | 卷数是命运设计书的输出，不应前置到状态文件 |
| `run_pipeline.sh` | 流程控制不受影响 |
| `chief_editor.md` | Section 1.5 铁则正确——没有注入包确实不该继续；阻断点不在它 |
| `project-agents-template/` | 需要的是进化，不是回退 |

### 4.3 Step 1 改动思路

**改动前（新版）**：
```
Step 1: 提取全书卷数（强制·不可跳过）

1. 从总纲领查找 "全书N卷粗略划分" → 提取N
2. 从 salt.json volume_position_variants 推断
3. ❌ 无法确定 → 禁止继续
```

**改动后**：
```
Step 1: 设计全书卷数

输入（已有，不需上游额外提供）：
  a) 白皮书 §一：原作卷/部划分、总章数
  b) 盐值 volume_rhythm_profile.default_volume_length（每卷预估章数）
  c) 总纲领 §二：平台目标字数标准（optimal_min / optimal_max）

计算逻辑：
  每卷章数 = default_volume_length（盐值，如 25）
  每章字数 = 平台字数标准中位数（总纲领，如 2000）
  总卷数 ≈ 平台目标总字数 / (每卷章数 × 每章字数)，向上取整
  交叉校验：与原作卷数对比，应用 rhythm_diff 调整比例

参数缺失时的兜底：
  - 缺目标总字数 → 从白皮书 §一"原作总字数"按 rhythm_diff 比例缩放
  - 缺每卷章数 → 从白皮书 §五"大高潮间隔"推算
  - 所有推算手段耗尽仍无法得出 → 明确列出缺失参数，返回错误

输出：
  记录 TOTAL_VOLUMES = N
  写入 00-全书命运总谱.md 头部
  后续步骤强制引用
```

### 4.4 关键设计原则

1. **数据来源可追溯**：每卷章数来自盐值，字数标准来自总纲领，不是凭空虚造。上帝视角 ≠ 拍脑袋。

2. **计算而非查找**：本质是拿着白皮书的结构参数、盐值的节奏参数、总纲领的字数预算做交叉计算。这是设计行为。

3. **有兜底有报错**：参数缺失时先尝试从现有数据推算，推算耗尽才报错。不像现在查一次没有就停。

4. **不改变上游**：`master_outline_generator`、`salt_architect` 都不需要改。它们各自提供自己领域的参数，由上帝视角做跨域决策。

### 4.5 风险评估

| 风险 | 概率 | 缓解 |
|------|------|------|
| LLM 推理的卷数与实际需求偏差 | 中 | 计算结果写入 00-命运总谱，可人工审计 |
| 白皮书 §一 数据缺失 | 低 | §一是 master_outline_generator 强制读取章节，Phase1 已验证存在 |
| 盐值 default_volume_length 为空 | 中 | 兜底：从白皮书 §五 大高潮间隔推算 |
| 总纲领无显式目标总字数 | 中 | 兜底：从赛道常规字数范围估算（如玄幻升级流 200-300万字） |

### 4.6 不建议的方向

| 方案 | 为什么不建议 |
|------|------------|
| 回退到旧版 (7步) | 放弃新版的价值（Step 1 校验、8步完整流程），不是进化 |
| 总纲领新增卷划分节 | 把决策权推给信息不完整的上游 agent |
| salt.json 加 `total_volumes` | 新增字段需要修改 `salt_architect`，且数据从哪来仍不明确 |
| 让用户手动填卷数 | 每本书每次运行都要人工介入，违背自动化初衷 |
