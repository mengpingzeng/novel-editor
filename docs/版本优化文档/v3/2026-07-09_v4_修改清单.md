**用户反馈**：差异化应该主要由盐值 + 总纲在上游保证，destiny 应聚焦于按约束展开剧情，不应承担繁重的差异度自检。

**优化方向**：上游增强 + 下游精简。

### 二次修改详细

#### 2.1 master_outline_generator §六 —— 卷级升级为 phase 级 + 预检

**改动前**：
- `volume_differentiation` 数组（36 行，每卷一行）
- 没有事件类型重叠检查——destiny_designer 必须在生成后自己做 H1

**改动后**：
- `phase_differentiation` 数组（~120 行，每 phase 一行）。每行基于盐值 `volume_rhythm_profile.phases` 的章节分组。
- **新增 §6.3「逐 Phase 事件类型重叠预检」**：
  - 提取原著对应 phase 区间的事件类型序列
  - 从总纲推导仿写预期的事件类型序列
  - 逐位比对（同构=1 分/近似=0.5 分/偏离=0 分/新增=-1 分，新增抵扣 ≤20%）
  - 四级判定：SAFE（<20%）/ CAUTION（20-35%）/ DANGER（35-50%）/ PRE-FAIL（>50%）
  - PRE-FAIL 不阻塞输出，以降级值 50% 继续，在 writer_note 中记录为"需人工复核的 phase"
- 每个 phase 行的新字段：
  - `original_event_type_sequence`：原著事件类型序列
  - `expected_derivative_event_type_sequence`：仿写预期序列
  - `overlap_rate`：预检的重叠率（0~1）
  - `overlap_assessment`：SAFE/CAUTION/DANGER/PRE-FAIL
  - `ban_forbidden_event_mapping`：禁止的事件类型映射（CAUTION ≥1 条，DANGER ≥2 条）
  - `required_new_event_types`：强制新增的事件类型（CAUTION ≥1 个，DANGER ≥2 个）
  - `danger`：bool 标记——下游 H 项对此 phase 执行严格检查
- 效果：差异化约束从"方向性"（每卷一行建议）升级为"可执行"（每 phase 的禁止清单 + 强制新类型 + 预检分数）

#### 2.2 destiny_designer 步骤 8 H 项 —— 精简

**改动前**（v4 第一轮）：

| 子项 | 内容 | 问题 |
|:---|:---|---:|
| H1 | 逐章事件类型序列比对（三维度：类型/角色/结果） | 与上游预检重复——同一份比对逻辑在 L2 和 L4 各做一次 |
| H2 | 差异化衰减趋势（前后半对比） | — |
| H3 | core_diff 落地检查（required_new_event_types 是否出现） | — |
| H4 | 高辨识度节点命中检查 | — |

**改动后**（v4 第二轮）：

| 子项 | 内容 | 改动说明 |
|:---|:---|---:|
| H1 | **约束表落地检查**（替代旧 H1+H3）——验证 destinny 生成的事件链是否遵守约束表 | 不再做逐章比对。直接从约束表读取预检的 `ban_forbidden_event_mapping`、`required_new_event_types`、`danger`、`overlap_assessment`——然后检查事件链是否违反禁止清单 / 缺失强制事件类型。危险标记的 phase 严格检查。 |
| H2 | **差异化衰减趋势**（保留）——检查各 phase 间的 ban 违反/缺失新事件是否存在衰减 | 不再读取原著数据。只检查 destiny 自己的生成质量——"前紧后松"模式。 |
| H3 | **高辨识度节点命中检查**（保留）——白皮书 §X 的极高/高节点是否在事件链中出现 | 保留为兜底安全网。这是最硬的防线——如果居然直接映射了原著"极高"辨识节点，说明整条约束链失效了。 |

**核心变化**：
- 旧 H1（逐章事件类型比对）完全移除 —— 这项逻辑已前移到 master_outline_generator 的预检（§6.3）
- 旧 H3（core_diff 落地检查）被合并到新 H1 的子检查 b/c 中
- 旧 H4 改名为 H3
- destiny 不再做原著比对——它只检查"自己是否遵守了上游约束表的禁令和强制要求"

**职责转移对照**：

| 检查内容 | 旧位置 | 新位置 | 原因 |
|:---|:---|:---|:---|
| 事件类型序列比对（逐章比对原著） | destiny H1 | master_outline §6.3 预检 | 上游做比对更合适——它在生成约束表时就知道全图；destiny 在逐章生成时做比对的上下文不足 |
| 禁止事件映射检查 | destiny H1 | destiny 新 H1.a | destiny 仍需要检查是否遵守了上游的 ban 清单——这是约束落地检查 |
| 强制新事件类型检查 | destiny H3 | destiny 新 H1.c | 同上 |
| 差异化衰减趋势 | destiny H2 | destiny H2（保留） | destiny 的生成质量自检——保留在生成端 |
| 高辨识节点命中 | destiny H4 | destiny H3（保留） | 兜底安全网——保留 |
