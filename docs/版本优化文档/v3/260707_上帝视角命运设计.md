## 一、背景：当前架构的问题

### 1.1 问题定位

当前 V2 架构中，小说写作流程的核心依赖是 `仿写衍生总纲领.md`——一份由 `master_outline_generator` 一次性生成的 10 节静态文档。

| 问题 | 具体表现 | 后果 |
|------|---------|------|
| **粒度太粗** | 人物只列了"底层逻辑+表层变量"，没有按卷展开变化弧线。例：柳青衣在总纲领中只有"待命名·第一核心盟友"一行描述 | content_writer 在写各卷时对同一角色没有一致的演变认知 |
| **无时序维度** | 所有信息平铺，无法回答"第 X 卷时角色 A 处于什么状态、为什么要做这件事" | plot_planner 每卷从零推理，容易前后矛盾 |
| **角色是工具人** | 配角只标注"关系函数"和"结构性作用"（如"推动主线/提供信息/制造冲突"），没有独立动机和自身视角的成长史 | 每个角色只会服务于主角，不像一个有血有肉的人 |
| **伏笔无全局网络** | 伏笔由 plot_planner 在写章纲时临时管理（F001..Fnnn），没有全书级的伏笔设计 | 跨卷伏笔靠运气才能对上，经常出现"埋了但忘了回收" |
| **剧情是反应式的** | 每章章纲由 plot_planner 根据总纲领"临时规划"，而非从一个确定性总谱中提取 | 无法确保卷八的开篇能和卷一的结尾形成有意识的呼应 |

### 1.2 根因

当前架构是一个**自下而上的生成流**：

```
白皮书 → 总纲领 → plot_planner 临场发挥 → 写一章是一章
```

总纲领只是"全局模板"，不是"确定性设计"。它告诉 plot_planner "你应该遵守什么规则"，但不告诉他"这卷具体发生了什么"。这导致每章的创作是反应式的——plot_planner 根据上下文推断"下一步该发生什么"，而不是从一份已经确定的剧本中引出。

## 二、核心理念：上帝之眼命运设计书

### 2.1 哲学

```
上帝在开写前就已经知道一切，但角色（和写作 agent）只知道本卷能知道的事。
每个人的人生轨迹是确定的，但只有在对应的卷，那条轨迹才会被"打开"。
```

### 2.2 顶层设计

在 Phase 1（框架生成）完成后、Phase 2（正文写作）开始前，新增一个 **Phase 1.5**：一次性地为整本小说生成一份完整的 **上帝之眼命运设计书**。

这份设计书包含：

| 模块 | 内容 | 视角 |
|------|------|------|
| **00-全书命运总谱** | 三幕结构、8卷总览、角色命运坐标系、信息不对称设计、结局锚点 | 上帝视角 |
| **01-人物命运谱** | 每个角色的性格底层逻辑（形成原因）、各卷状态变化、独立动机、独立人生叙事 | 每个角色都是自己故事的主角 |
| **02-剧情命运谱** | 每卷逐章的确定性事件链（WHAT，不是 HOW） | 上帝视角 |
| **03-伏笔命运谱** | A/B/C 三级伏笔的完整编织网络（含埋设点、暗示点、回收点、影响角色） | 上帝视角 |
| **04-世界观展开谱** | 每卷该揭示什么设定、不该揭示什么 | 上帝视角 |
| **05-卷级注入** | 每卷一份——从以上所有模块提取"当前卷写作 agent 需要知道的一切"= 唯一注入入口 | writer 视角 |

**核心原则**：
- `00~04` → 仅 `destiny_designer` agent 可读写。**绝不注入写作 agent**。
- `05` → 每卷一份。**唯一暴露给写作 agent 的入口**。
- 写作 agent 只看到当前卷的注入包，无法看到后续卷的任何信息。

### 3.2 02-剧情命运谱 强制质量标准

每卷剧情命运谱**必须**包含逐章确定性事件链表。以下为强制门禁：

| 规则 | 内容 |
|------|------|
| ALL_OR_NOTHING | 所有卷必须完成逐章事件链后才允许输出。不接受"骨架级占位" |
| MINIMUM_DETAIL_PER_CHAPTER | 每章至少包含：章号、核心事件（≥20字）、出场角色、新埋设伏笔ID（至少标注一个，标注"无"视为合法）、已有伏笔回收ID |
| VOLUME_END_CHECK | 每卷最后一个事件行必须明确描述卷末状态（主角境界、关系变化、已知信息、下一步方向） |
| ROW_COUNT_CHECK | 事件链表行数 = 该卷预期章数 |
| CROSS_REFERENCE | 每章中涉及的角色名/伏笔ID必须在 01-人物命运谱 或 03-伏笔网络 中有对应条目 |

**事件链格式（精简版·一行的WHAT）**：

```
| 36 | 宗门大比→方尘被迫上台→对宗主亲信弟子（筑基中期）→越级击败 |
      方尘、亲信弟子、宗主 | 实力展示 | F029(宗主注意到方尘) | — |
```

一行约60-100字。以8卷共~310章计，全部事件链总文本量约 15K tokens，在 V4 Pro 上下文窗口中完全可控。

### 3.3 01-人物命运谱 强制字段

每个人物命运谱文件必须包含：

| 节 | 必填内容 |
|----|---------|
| §1 性格底层逻辑 | 核心价值观、性格形成原因（童年/创伤/信念）、行为模式、致命弱点、隐藏渴望 |
| §2 人生各卷状态 | 表：各卷 | 年龄段 | 身份地位 | 核心动机 | 本卷关键行动 | 本卷性格变化 |
| §3 与其他角色的命运交叉点 | 与主角/其他关键角色的交集描述 |
| §4 独立于主角的故事线 | "如果我是一部独立小说，我的故事是这样的..."——从该角色视角写的第一人称叙事 |

## 四、Agent 设计：destiny_designer

### 4.1 定位

| 属性 | 内容 |
|------|------|
| 名称 | `destiny_designer`（命运设计师） |
| 类型 | 全局副本在 `.opencode/agents/destiny_designer.md`，**实际生效副本**在 `project-agents-template/.opencode/agents/destiny_designer.md`（项目创建时自动部署到各项目目录） |
| 模式 | **subagent**（由 chief_editor 通过 `@destiny_designer` 调用。不可为 Primary，因为 chief_editor 自身是 Primary，无法调用另一个 Primary） |
| 模型 | V4 Pro（需要深度推理和长期规划） |
| 触发时机 | chief_editor 初始化 §1.5 检测到注入包不存在 → 自动调用 @destiny_designer（仅首次运行） |
| 运行频率 | **每个项目仅运行一次**（生成完整的上帝之眼目录，后续卷直接复用） |
| 输入 | `base_whitepaper.md` + `project_salt.json` + `仿写衍生总纲领.md` |
| 输出 | 完整的 `上帝之眼/` 目录（见 §3.1） |
| 调用者 | `chief_editor`（在 Phase 2.0 初始化阶段自动触发） |

> ⚠️ 踩坑记录：最初设计为 `mode: primary` 且仅在全局 `.opencode/agents/` 中。导致两个问题——① chief_editor（Primary）无法通过 `@destiny_designer` 调用另一个 Primary；② pipeline 创建项目时从 `project-agents-template/.opencode/agents/` 复制 agents，全局目录中的文件不会被部署。**修正**：mode 改为 subagent + 同步副本到 project-agents-template。

### 4.2 生成流程（8步·严格顺序）

```
步骤1·提取全书卷数（强制）★
  → 依次从仿写衍生总纲领 → project_salt.json 提取总卷数
  → 找不到 → 禁止继续，返回错误（不可默认为1）
  → 找到 → 记录为 {TOTAL_VOLUMES}

步骤2·生成 00-全书命运总谱
  → 三幕结构映射（{TOTAL_VOLUMES}卷）、角色命运坐标系、结局设计

步骤3·生成 01-人物命运谱
  → 每个命名角色独立文件
  → 确保独立动机、性格形成原因、各卷变化弧线

步骤4·生成 02-剧情命运谱
  → 从卷一→卷{TOTAL_VOLUMES}逐卷生成逐章确定性事件链
  → 跨卷同步角色状态
  → ★ 质量铁律：所有卷必须逐章细化，不接受骨架级占位

步骤5·生成 03-全书伏笔网络
  → 提取所有伏笔ID → 标注A/B/C级别 → 埋设卷/暗示卷/回收卷

步骤6·生成 04-世界观展开谱
  → 确定每卷该揭示什么/不该揭示什么

步骤7·生成 05-卷级注入
  → 从 00~04 提取每卷"写作必知信息"
  → 不包含未来卷的任何剧情/伏笔回收/角色结局

步骤8·执行全局完整性自检
  → 4层门禁（文件存在性/逐章细化度/跨文件一致性/叙事一致性）
  → 全部通过 → 输出 + _自检报告.md
```

### 4.3 自检机制

destiny_designer 内置完整的自检流水线。输出前必须通过所有门禁：

```markdown
【自检步骤·强制执行】

## A. 文件存在性
- [ ] 00-全书命运总谱.md 存在
- [ ] 01-人物命运谱/ 文件数 ≥ 总纲领中声明的命名角色数
- [ ] 02-剧情命运谱/ 文件数 = 总卷数
- [ ] 03-伏笔命运谱/全书伏笔网络.md 存在
- [ ] 04-世界观展开谱/世界观展开节奏.md 存在
- [ ] 05-卷级注入/ 文件数 = 总卷数

## B. 逐章细化度（扫描 02-剧情命运谱/ 所有文件）
- [ ] 每卷文件事件链行数 = 该卷预期章数
- [ ] 每行"核心事件"字段 ≥ 20字
- [ ] 无"骨架级占位"或"______"占位符

## C. 跨文件一致性
- [ ] 00 声明的总卷数 = 02 文件数 = 05 文件数
- [ ] 三幕映射中的卷区间与各卷覆盖一致
- [ ] 角色命运坐标系中的角色在 01 中都有对应文件
- [ ] 03 中的伏笔ID在至少一个 02 卷文件中有"埋设"标记
- [ ] 03 中的伏笔ID在至少一个 02 卷文件中有"回收"标记
- [ ] 04 中"本卷可揭示"不与该卷注入包"不可触碰"冲突

## D. 叙事一致性
- [ ] 角色死亡卷之后，02/05 中不再出现该角色
- [ ] 境界递进跨卷线性，无回退
- [ ] 关键物品跨卷追踪无断裂

## 自检结果
- 若全部通过 → 输出文件 + 自检报告到 上帝之眼/_自检报告.md
- 若任一项未通过 → 回到对应步骤修正，不输出
```

**自检失败时的二重保护**：若 destiny_designer 在 token 上限或超时前未能通过自检，返回给调用方（chief_editor）的状态码携带失败原因。chief_editor 在收到失败信号后，**暂停流水线并向用户报告错误**——不允许降级为不确定性生成。失败信息包括具体的自检未通过项和修复建议。

### 5.2 chief_editor 的改动

在 `chief_editor.md` 中进行了两处改动：

**改动1**：核心原则新增第7条（强制门禁）：

```markdown
7. **v3 铁则·命运设计强制**：没有成功生成的注入包，绝不允许进入卷纲规划和章纲生成。
```

**改动2**：初始化 SOP 新增 §1.5（三向分支·无降级路径）：

```markdown
## 1.5 加载本卷注入包（强制·v3 上帝视角命运设计）

1. 检查 上帝之眼/05-卷级注入/卷0X-注入包.md 是否存在

2. 若存在 → 加载注入包 → 启用确定性注入模式 ✅

3. 若不存在 + Phase 1 输入文件就绪（白皮书+盐值+总纲领三者皆存在）：
   → 自动调用 @destiny_designer 生成命运设计书
   → 成功 → 回到步骤2加载注入包
   → 失败 → ★ 暂停流水线，向用户报告错误原因和修复建议

4. 若不存在 + Phase 1 输入文件不全：
   → ★ 暂停流水线，列出缺失文件路径
```

**与设计稿的关键变化**：最初设计稿允许"若注入包不存在→降级为原有模式"。实现时改为强制——没有注入包则暂停流水线，保证所有输出内容都经过命运设计书约束。

### 5.3 plot_planner 的改动（双模式兼容）

在 `plot_planner.md` 中增加"确定性细化模式"分支：

```markdown
## 确定性细化模式（新增·当调用方传入注入包内容时启用）

### 行为变化

调用方传入注入包内容时，plot_planner 进入「确定性细化模式」：
- 不从天零设计卷级阶段 → 直接使用注入包的阶段划分
- 不从天零设计章级事件 → 从注入包事件表读取本章核心事件，设计五段式展开
- 不临时决定爽点类型 → 使用注入包规定的爽点类型，设计实现方式
- 不独立管理伏笔 → 从注入包读取本章需埋设的伏笔ID，登记到章纲

### 输入字段

在原有输入基础上增加：
- 输入4（注入）：调用方传入的本卷注入包内容（字段见 chief_editor §1.5）
- 若未传入 → 降级为「从零设计模式」（向后兼容）
```

## 六、卷级注入与卷纲的关系（消除功能疑惑）

### 6.1 核心结论：WHAT vs HOW

| 文档 | 谁生成 | 层级 | 内容 |
|------|--------|------|------|
| 注入包 | destiny_designer（一次性） | WHAT（确定性约束） | 本章发生什么事件、谁出场、埋什么伏笔、用什么爽点 |
| 卷纲 | plot_planner（每卷一次） | HOW（实现方式） | 五段式结构、场景编排、对话方向、钩子设计、标签覆盖 |

**类比**：注入包是建筑蓝图的结构骨架，卷纲是同一根梁柱上的混凝土浇筑方案。

### 6.2 字段分工

| 字段 | 注入包 | 卷纲 | 关系 |
|------|--------|------|------|
| 核心事件 | ✅ 一行概要 | ✅ 五段式展开 | WHAT→HOW |
| 出场人物 | ✅ 清单 | 引用注入包 | 解耦（建议卷纲直接引用，不重新声明） |
| 爽点类型 | ✅ 规定 | ✅ 定位实现细节 | 约束→实现 |
| 伏笔ID | ✅ 规定 | 引用注入包 | 解耦 |
| 五段式结构 | ❌ | ✅ | 卷纲独有 |
| 章末钩子 | ❌ | ✅ | 卷纲独有 |
| 标签覆盖 | ❌ | ✅ | 卷纲独有 |
| 字数规划 | ❌ | ✅ | 卷纲独有 |

## 附录B：关键设计决策汇总（含实现修正）

| 决策 | 最终选择 | 理由 |
|------|---------|------|
| 注入入口 | 仅 `05-卷级注入/` 一个入口 | 确保写作 agent 不会看到未来卷的剧透 |
| 运行频率 | 仅一次 | 上帝视角是一次性设计，不需要按卷迭代 |
| 模型选择 | V4 Pro | 需要长期规划能力，Flash 不适合设计全N卷的因果链 |
| 自检时机 | destiny_designer 输出前 | 一次不通过=全部不确定，必须整体通过才放行 |
| agent mode | **subagent**（非 Primary） | 调用方 chief_editor 是 Primary，只能调用 subagent。实际运行中验证：mode=primary 导致 chief_editor 无法通过 `@destiny_designer` 调用 |
| 文件部署 | **project-agents-template + .opencode/agents 双副本** | pipeline 创建项目时只从 template 目录复制 agents。仅在全局目录中定义不会被部署到项目 |
| 失败处理 | **暂停流水线**（非降级） | 无注入包=用不确定性方式写作，不符合 v3 的质量标准。设计稿中允许降级，实现时改为强制 |
| 卷数提取 | **强制从总纲领提取，找不到就报错**（非默认1卷） | 甄嬛传 dryrun 验证：总纲领无显式卷数声明时，agent 默认1卷导致只生成1卷注入包。改为找不到卷数则禁止继续 |
| 剧情命运谱质量 | 所有卷必须逐章细化 | 不接受"骨架级"占位，放在 agent 定义中作为生成铁律 |
| pipeline 脚本 | **不需修改** | chief_editor 在 Phase 2.0 初始化阶段自动触发 @destiny_designer，无需改动 `run_pipeline.sh` |
| 卷纲改动 | 增加模式分支，保留原有行为 | plot_planner 在无注入时仍然是"从零设计"（但 chief_editor 已强制要求注入包存在后才会调用 plot_planner） |
