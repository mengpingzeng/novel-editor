# 评分体系 — 爽点公式全层级闭环修复方案

**生成时间**：2026-07-08

---

## 背景

评分体系中"爽点类型专属公式"从白皮书生成后，在 L2-L7 全部丢失。L5 quality_reviewer 的 D3（18分）用"四步闭环"这一未定义概念评分，content_writer 和 quality_reviewer 共用同一模型（自写自审），且 quality_reviewer 缺失章纲输入导致无法验证本章爽点类型。

## 核心设计变更

1. **数据源统一**：L3-L7 爽点验证全部从仿写衍生总纲领 §六 读取类型专属公式（非白皮书）。L2 使用白皮书作为对标物验证总纲领的变异透明度。
2. **粒度梯度**：每层做与该层职责匹配的验证——L2 验变异透明度、L3 验类型存在性、L4 验公式步结构对齐、L5 验执行+变异、L6 验克隆、L7 验保真。
3. **收束笔类型化**：不同爽点类型的收束笔不同（身份打脸→围观震惊、情感守护→被守护者感动），不再使用通用"配角反应镜头"。
4. **变异评分**：L5 对每步判断照搬/变异——照搬扣分，合理变异加分。

### L5 — quality_reviewer.md

**文件**：`project-agents-template/.opencode/agents/quality_reviewer.md`

**改动 1**（P0）：模型从 `team-deepseek/deepseek-v4-flash` 切为 `tokenhub/glm-5.2`。

**改动 2**（P1）：输入清单加入 `versions/{version}/01-大纲/第N章章纲.md`。

**改动 3**：D3（"爽点落地"，18分）全量重构：

```
从章纲读取本章申报的爽点类型 → 从总纲领 §六 读取该类型的专属触发公式 → 逐步检查。
```

| 检查项 | 扣分规则 |
|--------|---------|
| 本章是否包含章纲指定的主要爽点类型 | 缺失 -15 |
| 按该类型的专属公式逐步检查：每步是否存在 | 缺失 1 步扣分（按公式总步数均分 18 分中扣除） |
| 每步：是否有变异 | 照搬公式字面描述 → 扣分；合理变异（同样叙事功能、不同写法）→ 加分 |
| 是否有对应类型的收束笔 | 缺失扣分——不同类型收束笔不同，从公式末步或总纲领定义中获取（身份打脸→围观震惊、情感守护→被守护者感动、金句装逼→围观者被震慑等） |
| 爽点类型与章纲标注不一致 | -3/处 |

**改动 4**：D5（"标签落地"，18分）补充——每个标签需在总纲领"分类标识"中有关联的爽点类型映射。D5 验标签维度的约束（"该有的标签场景有没有"），D3 验爽点公式维度的执行（"标签对应爽点的公式写了没有"）。
