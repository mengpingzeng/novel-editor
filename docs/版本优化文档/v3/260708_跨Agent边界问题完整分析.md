# 跨 Agent 边界问题完整分析

> 分析范围：`master_outline_generator` / `destiny_designer` / `plot_planner` 三个 agent
> 前置讨论：职责边界、注入包来源、plot_planner 存在必要性均已分别分析

---

## 一、Agent 关系总览

```
Phase 1 (automation_manager)
  original_analyst → compliance → style_mapper → facade_generator
    → salt_architect → master_outline_generator → .phase1_done → 停止

Phase 2 (chief_editor, 独立启动)
  → 检查 05-卷级注入 是否存在
  → 不存在 → 调用 @destiny_designer → 生成上帝之眼（00~05）
  → 存在 → 读取注入包
  → 调用 @plot_planner（确定性模式或标准模式）
  → 调用 @content_writer → 调用 @quality_reviewer
```

| Agent | 产出 | 思考维度 |
|---|---|---|
| master_outline_generator | 仿写衍生总纲领.md | 框架融合（继承+叠加） |
| destiny_designer | 上帝之眼/（00~05） | 全局命运设计（故事事件 WHAT） |
| plot_planner | 01-大纲/卷纲 + 章纲 | 叙事形式（HOW 呈现） |

## 二、问题清单

### 第一档：会产生错误输出

#### 问题 1：伏笔 ID 体系分裂

**涉及**：destiny_designer ↔ plot_planner

**现状**：
- destiny 出 A/B/C 级伏笔 ID + 全局网络表（03-伏笔命运谱）
- plot_planner §4 出 F001/F002 自建 ID + 悬念伏/道具伏/成长伏/背景伏分类
- 两套体系无任何映射关系

**确定性模式下的矛盾**：
plot_planner 从注入包收到 destiny 的伏笔 ID（如 `B-07`、`A-02`），在事件段引用它们，但同时 §4 的本地伏笔管理系统照常运行、自建 F-xxx ID。同一章的章纲里，伏笔管理段落的 ID 体系和事件段落的 ID 体系互不认账。

**后果**：
- content_writer 看到两套伏笔编号，不知道引用哪套
- quality_reviewer 无法验证伏笔是否在回收点闭合
- 全书伏笔链路不可追溯

---

#### 问题 2：注入包 §5（爽点预算分配）来源缺失

**涉及**：destiny_designer 自身

**现状**：
destiny 步骤 7 要求"从 00~04 中提取"爽点预算分配（主推类型/次要类型/密度要求），但追溯 00~04 各文件的内容定义：

- 00（全书命运总谱）→ 无爽点预算分配表
- 01（人物命运谱）→ 无关
- 02（剧情命运谱）→ 有"爽点类型"列，但那只是单章标注，不是"预算分配"
- 03（伏笔网络）→ 无关
- 04（世界观展开谱）→ 无关

"爽点预算分配"是需要**计算**的数据（根据全书爽点轮换策略 + 本卷阶段划分 + 阶段密度），不是"提取"的数据。destiny 的 prompt 没有给出预算分配的计算规则。

**后果**：
- 注入包 §5 质量完全取决于模型当次发挥
- plot_planner 冲突规则（§0.4）声明"注入包 > 总纲领 > 盐值"——来路不明的预算会覆盖总纲领和盐值中经过完整计算的设计

---

#### 问题 3：确定性模式下 plot_planner 的双轨检查并行运行

**涉及**：plot_planner 自身

**现状**：
- plot_planner §3（6 阶段角色成长模型）在确定性模式下不关闭
- plot_planner §4（F-xxx 伏笔管理系统）在确定性模式下不关闭

它们的约束仍然生效，与注入包中 destiny 提供的角色状态和伏笔清单**并行检查**。

**冲突场景**：
destiny 设计主角在早期卷中露出实力一角（为后续伏笔埋设），但 plot_planner 的 §3 6 阶段模型把该卷映射到"隐忍蛰伏期"，该阶段的禁止行为含"不可在家族内部公开真实身份"。plot_planner 面临两套对立规则——destiny 说可以做，§3 说禁止——没有裁决机制。

**后果**：
- 确定性模式的优势被 §3/§4 的自治检查稀释
- plot_planner 可能因为自身约束拒绝 destiny 有意设计的越界事件
- 或强行通过导致章纲中出现内部矛盾

---

### 第二档：会导致隐蔽质量退化

#### 问题 4：注入包与源数据（02-剧情命运谱）缺乏同步校验

**涉及**：chief_editor + destiny_designer

**现状**：
chief_editor §1.5 只检查 `05-卷级注入/卷0X-注入包.md` **文件是否存在**，不检查其内容是否与 `02-剧情命运谱/卷0X-剧情.md` 一致。

**触发条件**：
1. destiny 自检不通过，回溯修改了 02 中某章事件，但 05 未被重建
2. 人工调试修改了 02 的事件链但忘记重建 05
3. destiny 在生成 05 时对 02 的转录发生错误（模型幻觉）

**后果**：
- plot_planner 拿到过期/错误的注入数据
- 基于废弃事件设计章纲 → content_writer 写出与全局命运脱节的正文
- 脱节只能在 Phase 3 审稿发现，修复需回溯整条链路

---

#### 问题 5：plot_planner 不标记运行模式

**涉及**：plot_planner 自身

**现状**：
plot_planner 的永久置顶中有模型自检行（">>> 模型自检: team-deepseek/deepseek-v4-flash <<<"），但**没有模式声明行**。标准模式和确定性模式的输出格式完全一样，从章纲文件无法判断其事件来源是"上帝设计"还是"agent 自行推理"。

**后果**：
- human reviewer 拿到章纲无法判断事件的可信度等级
- quality_reviewer 无法区分"确定性事件实现偏差"和"自行发明的事件质量问题"
- 标准模式下 plot_planner 自创了与全局命运冲突的事件，在章纲层面完全看不出来

---

#### 问题 6：master_outline §9 "章节批级剧情单元模板"边界不清晰

**涉及**：master_outline_generator 自身

**现状**：
"章节批级剧情单元模板"这个措辞语义上允许两种理解：
- A）抽象范式："每 5 章：铺垫→酝酿→小高潮→过渡→大高潮"
- B）半具体模板："第 N 章：男主在拍卖会遇到宿敌→第 N+1 章：宿敌派人暗杀→..."

master_outline_generator 的 prompt 没有显式禁止理解 B，没有声明"不可包含具体角色名/事件名/场景名/章节编号"。

**后果**：
如果模型选择理解 B，master_outline 会生成 borderline 具体的剧情内容，侵入 destiny_designer 的设计域。两个 agent 对同一本书产出了不一致的具体剧情。

---

#### 问题 7：master_outline_generator ↔ destiny_designer 无交叉验证

**涉及**：master_outline_generator ↔ destiny_designer

**现状**：
两个 agent 各自独立读取 whitepaper + salt + facade，各自独立产出不同的衍生品。无机制检查：

- master_outline §5 声明的命名角色数 vs destiny 01-人物命运谱 的文件数
- master_outline §6 声明的爽点类型集合 vs destiny 02 事件表中实际分配的爽点类型
- master_outline §7 的节奏参数（大高潮间隔 N 章）vs destiny 计算的总卷数 × 每卷章数是否一致

**冲突场景**：
master_outline 从盐值中解读"大高潮每 25 章一次"，destiny 从同一个盐值计算出"总卷数=8、每卷 20 章"——高潮间隔实际为 20 章。两个衍生品基于同一源数据得出矛盾结论，后续 plot_planner 和 content_writer 在矛盾的约束间无法选择。

---

### 第三档：管道设计与运维问题

#### 问题 8：destiny_designer 是 Phase 2 的单点阻塞

**涉及**：destiny_designer + chief_editor

**现状**：
destiny 不在 Phase 1 框架生成中，而是 chief_editor 在 Phase 2 启动时按需调用。它**一次性生成全书所有卷的 00~05**。如果失败（token 超限、自检不通过），Phase 2 完全阻塞——连第一卷也写不了。

**影响**：
- 对一本 300 章的小说，destiny 的一次调用承担全局设计 + 逐章事件链 + 自检的全部 token 压力
- 没有分卷增量生成的逃生路径
- 失败后用户无任何增量产出

---

#### 问题 9：无增量重建注入包或单卷命运数据的机制

**涉及**：destiny_designer 自身

**现状**：
如果用户发现卷 3 的命运设计有问题（某角色状态不一致），无法只重建卷 3 的 02/05 数据。destiny_designer 总是全量重新生成。

**后果**：
- 局部修改需要全量重跑
- 新的全量生成可能与卷 1、卷 2 已写内容矛盾
- 已完成的写作工作被迫重新对焦

---

#### 问题 10：automation_manager 对 destiny_designer 无感知

**涉及**：automation_manager

**现状**：
automation_manager 的 Phase 1 流程不包含 destiny_designer。它的"全部通过"不代表写作就绪——还有 destiny 这个必经步骤。

**后果**：
- Phase 1 完成后的汇总表显示"白皮书✅ 盐值✅ 总纲✅ → 完成"，但事实上还缺命运设计
- 没有对 destiny 输入文件完整性的预检
- chief_editor 在 Phase 2 启动时遇到缺失的注入包才被动触发，把 destiny 变成了启动时的突发阻塞

## 四、根因分析

10 个问题的根因可以归纳为三类：

### 根因 A：接口契约不完整

问题 #1、#2、#4、#5、#7 的共同原因是：两个 agent 之间的数据传递缺乏显式的契约定义。

- #1：伏笔数据在 agent 间传递，但没有统一 ID 约定
- #2：注入包的爽点预算被要求"提取"，但上游没有产出这个字段
- #4：chief_editor 读取注入包时不校验其与上游的一致性
- #5：plot_planner 的产出不标注其输入模式，下游无法区分可信度
- #7：两个 agent 产出间没有互检机制

**本质上**：agent 之间的输入/输出被当作"被动文件路径传递"，而非"有 schema 验证的数据协议"。每个 agent 的 prompt 都只描述自己如何消费输入、生产输出，但没有定义"输入必须满足的前置条件"和"输出可以对下游做出的保证"。

### 根因 B：一个 agent 内的职责未收敛

问题 #3、#6 的共同原因是：单个 agent 的 prompt 包含了相互矛盾的多套规则。

- #3：plot_planner 同时维护"确定性细化模式"和"标准模式"两套逻辑，模式切换靠布尔参数，两套规则在源代码层面共存
- #6：master_outline 的一个节的措辞模糊到允许两种截然不同的理解

**本质上**：agent 的 prompt 不是"单一职责的声明"，而是"历史需求的堆积"。每次发现问题就往 prompt 追加规则/模式，旧规则没有退役，新规则没有显式声明冲突处理。

### 根因 C：管道编排阶段划分不匹配 agent 依赖

问题 #8、#9、#10 的共同原因是：agent 之间的依赖关系在管道编排中没有得到充分表达。

- #8：destiny 对全量写作的依赖被压入 Phase 2 顶部，没有利用 Phase 1 的可并行性
- #9：没有局部修复的 API，因为管道把 destiny 当成原子操作（全量→成功/失败）
- #10：automation_manager 不知道 destiny 的存在，因为管道阶段划分是按"操作类型"而非"数据依赖"

**本质上**：管道编排（Phase 1 → Phase 2 → Phase 3）是按操作类型划分的，而不是按数据依赖图划分的。destiny 的数据依赖（它需要 master_outline + whitepaper + salt）早已在 Phase 1 结束时满足，但因为它"像是 Phase 2 的事"，被推迟到了 Phase 2 启动顶部的突发阻塞位置。
