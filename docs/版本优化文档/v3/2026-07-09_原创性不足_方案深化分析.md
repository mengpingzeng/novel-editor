# 原创性不足——方案深化分析

**日期：2026-07-09**
**对应主分析：`2026-07-09_原创性不足根因分析与解决方案.md`**
**本文回答三个深化问题：方案一风险规避、plot_planner 创作功能评估、salt_architect 差异度修正**

---

## 一、方案一：`destiny_designer` 增强差异化职责——风险规避

### 风险 1：prompt 过长（已 475 行）

**方案**：不用"内嵌指令"，用"外部文件引用"。

- 让 `master_outline_generator` 在生成总纲时额外输出一个 **`差异化约束表.json`**（独立文件，放在 `00-素材/` 下）
- 内容：原著每章/每卷的事件摘要、本仿写的差异化方向、差异度阈值
- `destiny_designer` 的 prompt 只增加**一行引用**：
  > "读取 `00-素材/差异化约束表.json`，在步骤 4 生成事件链时逐章比对，步骤 8 自检新增 H 项"
- 实质的比对逻辑在外部 JSON 中，prompt 只做引用

**效果**：prompt 从 475 行 → ~480 行（+5 行引用 + H 项检查模板），负担极小。

---

### 风险 2：V4 Pro 模型超时（1260 章逐章比对）

**方案**：差异度检查不做"1260 章一次性比对"，而是**逐卷执行，跟随步骤 4 的自然节奏**。

- `destiny_designer` 步骤 4 本身就是逐卷生成事件链（全量模式下循环 36 卷，重建模式下只做单卷）
- 差异度检查跟着这个节奏：生成完「卷 01」的 30 章 → 立即做一次差异度检查
- 比对范围是单卷 vs 单卷，不是 1260 章 vs 1260 章
- 比对方式不是 NLP 语义分析，而是**事件类型序列比对**（keyword-level，token 消耗极低）：

```
原著卷01事件类型序列: [测试羞辱 → 奇遇 → 导师觉醒 → 教学 → 突破 → 打脸 → 探索]
仿写卷01事件类型序列: [测试羞辱 → 夺宅 → 古丹觉醒 → 血脉揭示 → 突破代价 → 碾压 → 探索天火]
                           ↑ 同类型但变形    ↑ 全新事件       ↑ 同类型但代价体系
```

检测到 ≥3 个连续同类型事件 → 标记预警。

---

### 风险 3：章节映射关系不可靠

**方案**：不需要精确的"1:1 章节映射"，用**卷级模糊对齐**就够了。

- 原著第 1 卷 ~25 章、仿写第 1 卷 ~30 章，覆盖相同的叙事功能区间（开篇→觉醒→成长→高潮）
- 差异度检查不关心"仿写 Ch5 是否对应原著 Ch5"，而是检查"相同叙事功能区间内，双方事件类型序列的同构程度"
- 白皮书 §一 已给出原著的「全书结构地图」（每卷/每章的事件摘要），这个数据已经存在

---

### 额外风险：差异化自检失败导致反复 rebuild

**方案**：H 项检查不设为硬 Gate，改为**分级预警**：

| 重叠比例 | 判定 | 行为 |
|:---:|:---:|:---|
| 0-20% 章与原著同构 | ✅ PASS | 正常放行 |
| 20-40% 章与原著同构 | ⚠️ WARN | 标记事件链，传给 plot_planner 时附加 DIFF_WARNING |
| ≥40% 章与原著同构 | ❌ FAIL | 禁止输出，触发 rebuild_volume |

这样不会因一两章的微弱重叠就阻塞整条管线。

## 二、方案三评估：`plot_planner` 是否应该具有创作功能

### 核心矛盾

```
v2（自由模式）: plot_planner 创造事件 → 原创度 15/15，但可能走偏
v3（确定模式）: destiny_designer 创造事件 → 一致性高，但原创度 10/15
```

v2 和 v3 不是只有两个极端——存在中间态。

### 关键洞察：创作有 3 个等级，不是二元对立

| 等级 | 描述 | 举例 | v2 | v3 | 评估 |
|:---:|:---|:---|:---:|:---:|:---:|
| **L1 创造** | 凭空发明新事件节点 | "加一场原著没有的炼丹大赛" | ✅ | ❌ | **保持给 destiny** |
| **L2 替换** | 事件功能不变，替换具体形态 | "送药材→改为送药材+留苏族暗记" | ✅ | ❌ | **应给 plot_planner** |
| **L3 微调** | 保持功能和形态，调整叙事细节 | "送药场景从早晨→深夜" | ✅ | ✅ | 保持给 plot_planner |

v3 的错误是：**禁止 L2 时把 L2 也一并禁了**。

### L2 替换的边界

**可替换（L2 范畴）**：
- ✅ 事件发生的时空环境（早晨/深夜、室内/户外、晴天/雨天）
- ✅ 触发方式（主动/被动/偶然触发）
- ✅ 配角反应层（增加原著没有的反应细节）
- ✅ 附加线索（在原事件骨架上增加小细节/暗记/伏笔装饰）
- ✅ 叙事角度（纯主角视角 → 第三方视角闪入再切回）

**不可替换（L1 范畴，仍归 destiny）**：
- ❌ 核心事件本身（送药还是送药，不能改成"送秘籍"）
- ❌ 事件的结果输出（不能改变本章的爽点/信息释放目标）
- ❌ 伏笔的埋设和回收（伏笔 ID 和类型不能变）
- ❌ 出场角色（不能无故增减）

### L2 替换的触发条件

不是每章都需要 L2 替换，只在以下情况触发：

1. **destiny_designer 标记了 DIFF_WARNING 的章节**——原著映射度高，需要拉开差距
2. **连续 3 章以上为同类型爽点的章节**——需要替换事件形态避免疲劳
3. **连续 2 章以上场景类型相同的章节**——如"矿洞探索→换不同探索场景"

### 这么做的好处

形成双 Agent 互补：

| | destiny_designer（V4 Pro） | plot_planner（V4 Flash） |
|:---|:---|:---|
| 视角 | 全局一致性优先 | 局部可执行性优先 |
| 职责 | 保伏笔闭合、角色命运连续 | 在每个章节注入微创新 |
| 创作权限 | L1 创造 + L2 替换决策 | L2 替换执行 + L3 微调 |
| 替代关系 | 非替代——互补 | 非替代——互补 |

### 额外风险与缓解

| 风险 | 缓解措施 |
|:---|:---|
| plot_planner 替换后与后续卷冲突 | L2 替换不允许改伏笔和角色状态，不影响跨卷一致性 |
| Flash 模型创意质量不够 | L2 替换有明确模板（替换什么/不替换什么），不是自由发挥。可配"替换方向池"供选择 |
| 替换后情节走偏 | chief_editor 在接收章纲后做"章纲 vs 注入包"合规性校验，检查核心事件和伏笔是否被修改 |

### 评估结论

**应该给 plot_planner L2（替换）能力，但不给 L1（创造）能力。**

这与方案一（destiny_designer 增强差异化职责）是互补关系，不是替代关系：
- `destiny_designer` 在全局层面保差异化（通过自检和约束）
- `plot_planner` 在局部层面注入差异化（通过 L2 替换）

## 三、`salt_architect` 缺少情节创意差异化检查——如何修正

### 先明确边界

`salt_architect` 工作在 **Phase 1（框架层）**，检查的是 `project_salt.json` 本身的质量。它不能检查"写出来的章节是否和原著太像"——那是 Phase 2/3 的事。

所以问题更精确地说：**在 salt 层面，salt_architect 能校验什么来保障后续的差异化执行能力？**

### 当前已有的 14 条规则中与差异化相关的部分

| 规则 | 内容 | 与差异化的关系 |
|:---|:---|:---|
| Rule 1 | base_logic_compliance | 保原著底层逻辑一致（反向作用——不是保差异化的） |
| Rule 2 | deduplication | 对历史盐值去重（间接） |
| Rule 3 | tag_constraint_completeness | 检查标签约束完整性（间接） |
| Rule 5 | pleasure_rotation_validation | 爽点轮换多样性（间接） |

没有一条是直接检查"差异化声明是否充分"的。

### 新增的 4 条检查规则

#### Rule 15：`core_diff_dimension_check`（差异化维度校验）

**目的**：确保 `core_diff` 覆盖全面且有量化参数。

**逻辑**：
- 差异化维度必须 ≥3 个（目前 3 个：角色/节奏/爽点 → ✅）
- 每个维度必须有**量化参数**（不仅是方向声明）：
  - ✅ 好：`"大高潮间隔从 20-30 章改为 35-45 章（偏差≥50%）"`
  - ✅ 好：`"禁用身份打脸（降频至≤3%），新增血脉共鸣型(12%)"`
  - ❌ 差：`"节奏变慢一些"`（不可执行）
- 至少有 1 个维度涉及**情节骨架层面**（不只覆盖人设/文笔）

**失败处理**：❌ FAIL → 须补充量化参数。

---

#### Rule 16：`differentiation_distribution_check`（差异化分布校验）

**目的**：防止差异化策略只集中在"开场"，后续章节缺少差异化指引（这正是 v3 出现的问题）。

**逻辑**：
- 扫描 `core_diff` + `anti_similarity` + 总纲 §9 的差异化声明
- 统计差异化覆盖的时间范围：
  - 仅前 3 章 ❌ FAIL（判定为"开场差异化"）
  - 前 30 章 ⚠️ WARN（需覆盖全书前 20% 才有足够牵引力）
  - 全书（有逐卷或逐阶段的差异化策略） ✅ PASS
- 是否有"差异度衰减防护"机制——如"每卷至少引入一个原著没有的事件类型"

**失败处理**：WARN 触发告警不阻塞，FAIL 则要求补齐分布。

---

#### Rule 17：`dialectical_relationship_check`（原著辨识节点映射校验）

**目的**：检查 salt 是否标注了"原著高辨识度节点的映射方式"。

**逻辑**：
- 从白皮书 §X「原著辨识度清单」（方案五的输出）提取原著核心辨识节点
- 检查 salt 中是否对这些节点的映射方式做了明确标注：

```json
{
  "原著节点'退婚'": "本仿写替换为'夺宅'，差异维度=冲突性质替换（身份羞辱→资源剥夺）",
  "原著节点'异火榜'": "本仿写替换为'天火榜+灵渊血脉感知'，差异维度=金手指获取方式"
}
```

- 如果白皮书有 ≥5 个辨识度节点，但 salt 中标注处理方式的 <3 个 → ❌ FAIL

**前提依赖**：
- 白皮书需要有 §X「原著辨识度清单」（方案五）
- salt 中需要有对应字段记录映射关系

如果两者缺一，Rule 17 跳过（标记"数据不足"）。也就是说，这是一个**渐进式规则**——在白皮书和 salt 都完善后才激活。

---

#### Rule 18：`parallel_events_risk_check`（事件模板同构风险校验）

**目的**：预判 salt 中是否包含了过多与原著类型相同的事件模板。

**逻辑**：
- 白皮书 §七「剧情推进通用模板」中提取了原著的剧情范式（如"测试→羞辱→奇遇→导师→修炼→打脸"）
- salt 的 `volume_rhythm_profile.phases.core_task` 中是否包含了与之同构的阶段描述？
- 如果仿写的节奏阶段与原著的情节范式高度同构（关键词重叠≥60%），则标记 ⚠️ WARN

**注意**：这个检查在工具层面需要一定的语义理解。可行的简化版本：检查关键词重叠率（"修炼""突破""打脸"等词在两个序列中同时出现的比例）。

---

### 职责划分总表

明确四个 Agent 在"差异化"维度上的分工，避免重复和漏检：

| Agent | 职责 | 检查时机 | 检查对象 | 新增内容 |
|:---|:---|:---:|:---|:---|
| **`salt_architect`** | 盐值**元能力**校验 | Phase 1 | `project_salt.json` | Rule 15-18：检查 salt 是否配备了充分的差异化指令基础设施 |
| **`destiny_designer`** | **执行层**差异化自检 | Phase 2 | 生成的事件链 | 步骤 8 H 项：逐章比对差异度，分级预警 |
| **`plot_planner`** | **局部**差异化注入 | Phase 2 | 单章章纲 | L2 替换：在 DIFF_WARNING 章替换事件形态 |
| **`quality_reviewer` / `signing_reviewer`** | **结果层**差异度验证 | Phase 2+3 | 正文终稿 | 原创改写度评分（已有，不需新增） |

这条链路的设计原则是：**上游验证"能不能做"，中游验证"有没有做"，下游评价"做得好不好"**。

## 四、三个问题的关联性

三个问题不是孤立的，它们形成一条完整的差异化增强链：

```
salt_architect (Rule 15-18: 确认盐值有差异化能力)
       ↓
destiny_designer (步骤8 H项: 执行全局差异化自检)
       ↓
plot_planner (L2替换: 在局部注入差异化微创新)
       ↓
quality_reviewer / signing_reviewer (验证结果)
```

- 如果 salt_architect 发现 salt 的 `core_diff` 只有方向声明没有量化参数（Rule 15）→ 在 Phase 1 拦截，省去下游一堆问题
- 如果 destiny_designer 发现某卷事件链与原著重叠 30%+（H 项 WARN）→ 给 plot_planner 打标记
- 如果 plot_planner 收到 DIFF_WARNING 标记 → 启动 L2 替换，在该章注入差异化
- 如果质量端 reviewer 仍发现差异化不足 → 说明阈值太高，调低 H 项的阈值

这一套不是"一个方案替代另一个方案"的关系，而是一整套**分层防御体系**。
