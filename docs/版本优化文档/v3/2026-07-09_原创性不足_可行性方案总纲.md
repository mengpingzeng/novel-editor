# 原创性不足——可行性方案总纲

**日期：2026-07-09**
**基于三轮讨论的综合整理：根因分析 → 方案深化 → 范围与重建机制探讨**

---

## 一、问题回顾

签约审稿维度 9（原创改写度）仅 10/15，扣 5 分。主干情节与原著的映射关系过于直接：
- 前 3 章的每节核心事件在原著对应章节均有原型
- 差异度从 Ch1(95) → Ch5(86) 呈下行趋势
- 差异化策略仅覆盖"开场"，对后续章节缺乏持续约束

v3 的核心矛盾：**工程化水平提升 vs 创作差异化下降**。
v2 在两约束下原创改写度 15/15 满分，v3 高精约束下反而跌至 10/15。

---

## 二、根因定位（一句话版）

```
destiny_designer 掌管了全书 1260 章的"事件 WHAT"设计权，
但其 prompt 中没有任何"差异化自检"职责。
它在 v3 的确定性模式下生成事件链时，
自然回归到了原著的事件类型惯性。
下游的 plot_planner 被锁定为"只细化不创造"，
content_writer 被锁定为"只执行不发明"，
整条管线在原创性维度上变成了"高精度复印机"。
```

### 五层根因

| 层 | 根因 | 严重程度 |
|:---|:---|---:|
| L0 职责 | `destiny_designer` 的 prompt 没有差异化自检职责（步骤 8 缺 H 项） | **根本原因** |
| L1 框架 | `core_diff` 是框架级声明，没有逐卷/逐阶段的差异化约束 | 次因 |
| L2 锁定 | v3 的"确定性细化模式"剥夺了 plot_planner 的所有创意弹性 | 次因 |
| L3 静态 | `salt_architect` 的 14 条规则中无任何差异化元能力检查 | 补充 |
| L4 后置 | Phase 2 无差异度预检，Phase 3 才发现但无法返工 | 流程 |

```
       │           └──────────────────┘
       │                      ↑
┌──────┴──────┐    ┌──────────┴────────┐
│ master       │    │ plot_planner      │

│  (36卷×1行)  │    │  替换事件形态      │
└───────────────┘    └──────────────────┘
       │
┌──────┴──────┐
│ salt_        │

```

五层不是"选一个"的关系，是**每层解决一个不同的问题**：

| 层级 | Agent | 解决的问题 | 角色 |
|:---:|:---|:---|---:|
| L1 | `original_analyst` | 上游白皮书没有标注"哪些是原著的原著辨识度指纹" | 提供"雷区地图" |
| L2 | `master_outline_generator` | 差异化策略只覆盖开场，没有覆盖全书 | 提供"每卷差异化宪法" |
| L3 | `salt_architect` | 盐值没有差异化元能力的完整性校验 | 在 Phase 1 拦截不合格盐值 |
| L4 | `destiny_designer` | 事件链生成后没有差异化自检 | 在"事件输出前"做拦截 |
| L5 | `plot_planner` | 确定模式剥夺了局部创意弹性 | 在"章纲级"注入差异化微创新 |

## 四、各层方案详述

### L1: `original_analyst` — 新增 §X 原著辨识度清单

**改动**：在白皮书末尾新增一节 §X「原著辨识度清单」

**内容**：

```markdown
## §X 原著辨识度清单

以下节点是原著中最具辨识度的情节指纹——
若仿写不加以变形直接复用，读者/编辑会立刻察觉。

### X.1 标志性事件（不可原样复用的高辨识度节点）
| 原著事件 | 辨识度等级 | 识别维度 | 建议变形方向 |
|----------|:---------:|---------|-------------|
| 退婚/休书 | 极高 | 这是原著开篇的核心记忆点 | 冲突类型替换（身份羞辱→资源剥夺） |
| 三十年之约 | 极高 | 这是原著中期的核心动力 | 时间驱动力替换（复仇约期→生存守护期） |
| 异火榜排名 | 高 | 这是原著金手指体系的核心 | 体系替换（异火榜→天火榜+血脉阶） |
| 黑袍炼药师大赛 | 高 | 这是原著中典型的装逼场景 | 场景类型替换（大赛→遗迹试炼） |
| 萧薰儿古族身份 | 中 | 这是原著女主的身份隐藏线 | 身份线替换（古族→远古苏族+天火守护者） |
| ... | ... | ... | ... |

### X.2 高频套路模板（易被无意识复用的叙事模式）
- "废柴被嘲笑→奇遇→打脸→舆论反转"四段式（每周≥3次使用 → 判定为过度使用）
- "导师沉睡→主角遇险→导师苏醒解围"救场模式
- "大赛/拍卖会/秘境"三件套场景轮换
```

**价值**：
- original_analyst 是唯一定向读过原著全文的 agent，它最有资格做这个判断
- 下游所有 agent 自动受益（destiny_designer → 规避处理、quality_reviewer → 自动检测锚点）
- 即使其他层不改，有了这个清单，destiny_designer 生成事件时至少知道"哪些是雷区"

**工作量评估**：
- original_analyst 的 prompt 增加 ~50 行（模板 + 示例 + 输出格式）
- 执行时约增加 5-10% 的 token 消耗（多一步辨识度分析）

### L2: `master_outline_generator` — 新增卷级差异化约束表

**改动**：在生成总纲的流程中，额外输出一张「差异化约束表.json」

**位置**：`{version}/00-素材/差异化约束表.json`（独立文件，不塞进总纲正文）

**结构**：

```json
{
  "version": "v3",
  "base_novel": "斗破苍穹",
  "derivative_novel": "灵渊天火录",
  "volume_differentiation": [
    {
      "volume": 1,
      "original_core_conflict": "天才陨落→退婚羞辱→戒指奇遇→药老觉醒",
      "derivative_core_conflict": "天才陨落→夺宅→古丹觉醒→血脉封印揭示",
      "differentiation_dimensions": ["冲突类型替换（羞辱→剥夺）", "金手指触发方式（被动→血脉绑定）", "代价体系植入"],
      "high_risk_original_nodes": ["退婚休书", "三十年之约", "药老戒指"],
      "required_min_overlap_rate": 0.4,
      "required_new_event_types": ["代价揭示", "血脉共鸣"],
      "notes": "本卷需建立全套代价体系——每次突破必有可感知的代价，这是与原著的核心理念差异"
    },
    {
      "volume": 2,
      "original_core_conflict": "学院修行→炼药初显→势力接触",
      "derivative_core_conflict": "矿洞异变→血脉试炼→莫家全面反击",
      "differentiation_dimensions": ["地图替换（学院→矿区遗迹）", "主线替换（炼药大赛→血脉觉醒测试）"],
      "high_risk_original_nodes": ["迦南学院入学", "炼药师品级测试", "萧薰儿古族暗示"],
      "required_min_overlap_rate": 0.3,
      "required_new_event_types": ["势力经营", "遗迹探索"],
      "notes": "本卷需确保远古苏族这条线开始浮现，替代原著的古族线"
    },
    ...
  ],
  "global_constraints": {
    "max_consecutive_similar_events": 3,
    "required_new_event_types_per_volume": 1,
    "opening_differentiation_bonus": "前5章差异度应≥30%，10章后应≥50%",
    "forbidden_direct_mappings": ["退婚→任何形式的婚约解除", "三十年之约→任何形式的复仇约期", "异火榜排名→任何形式的天地灵物排行榜"]
  }
}
```

**关键设计原则**：

1. **卷级粒度**（36 行），不是章级（1260 行）。`master_outline_generator` 的信息量足够支撑卷级约束，但不足以支撑章级（它不知道 destiny 会怎么设计具体事件）。
2. **约束是禁止性的**（"不要做 X"），不是建设性的（"去做 Y"）。建设性的交给 destiny_designer。
3. **`required_min_overlap_rate`** 是卷级差异度阈值——超过这个比例就触发 WARN 或 FAIL。
4. **`required_new_event_types`** 强迫每卷至少引入一个原著没有的事件类型，防止全卷都在原著事件类型范围内打转。
5. 全局的 `forbidden_direct_mappings` 列表是硬红线——destiny_designer 如果直接映射了这些节点，自检时直接 FAIL。

**工作量评估**：
- `master_outline_generator` 的 prompt 增加 ~80 行（输入读取 + 输出格式 + 跨卷约束逻辑）
- 输出文件大小：~3000 字（36 卷 × 每卷 80 字 + 全局约束）

### L3: `salt_architect` — 新增 Rule 15-18 差异化元能力校验

**改动**：在 salt_architect 的 14 条规则基础上新增 4 条。

**新增规则明细**：

#### Rule 15: `core_diff_dimension_check`

- **目的**：确保 `core_diff` 有 ≥3 个差异化维度，且每个维度有量化参数
- **检查逻辑**：
  - 维度数量 ≥3 → PASS，否则 FAIL
  - 每个维度必须有可执行的量化参数（不仅是方向性声明）
    - ✅ `"大高潮间隔从 20-30 章改为 35-45 章"` → 通过
    - ❌ `"节奏变慢一些"` → 不通过
  - 至少有 1 个维度涉及"情节骨架"层面 → 否则 WARN
- **失败处理**：FAIL → 盐值无法通过校验，须补充

#### Rule 16: `differentiation_distribution_check`

- **目的**：防止差异化策略只集中在"开场"
- **检查逻辑**：
  - 从 `core_diff` + `anti_similarity` + 总纲 §9 提取差异化声明的覆盖范围
  - 仅前 3 章 → ❌ FAIL
  - 前 30 章 → ⚠️ WARN
  - 全书覆盖（有逐卷或逐阶段的差异化策略）→ ✅ PASS
- **失败处理**：WARN 触发告警但可由总编覆盖，FAIL 要求补齐

#### Rule 17: `dialectical_relationship_check`（条件激活）

- **目的**：检查 salt 是否标注了原著的"高辨识度节点映射方式"
- **前提**：白皮书存在 §X（L1 的输出）且 salt 有对应字段 → 才激活
- **检查逻辑**：白皮书 §X 的 ≥5 个辨识节点中，salt 标注了处理方式的 ≥3 个 → PASS，否则 FAIL
- **前提不满足时**：跳过，标记"数据不足"

#### Rule 18: `parallel_events_risk_check`

- **目的**：预判盐值的节奏阶段是否与原著模板高度同构
- **检查逻辑**：
  - 白皮书 §七 提取原著的情节范式类型序列
  - salt 的 `volume_rhythm_profile.phases.core_task` 提取仿写节奏阶段类型序列
  - 关键词重叠率 ≥60% → ⚠️ WARN
- **失败处理**：WARN 触发告警（提示「节奏框架与原著同构风险较高，建议增加差异化阶段设计」）

**工作量评估**：
- `salt_architect` 的 prompt 增加 ~60 行（4 条规则的定义 + 输出格式 + 通过标准）
- 执行时约增加 10% 的 token 消耗（读取白皮书 §一 和 §七 做对比）

### L4: `destiny_designer` — 步骤 8 H 项差异度自检 + Phase 级 rebuild

这是**整个方案的核心改动**——destiny_designer 是事件链的生产者，在它这里增加自检是最直接的拦截点。

#### 4.1 新增 H 项差异度自检

**添加位置**：步骤 8 自检清单末尾

**检查内容**：

```markdown
## H. 差异化自检（新增）

**数据来源**：
- 00-素材/差异化约束表.json（来自 L2 master_outline_generator）
- 白皮书 §X 原著辨识度清单（来自 L1 original_analyst）
- 本卷生成的事件链（步骤 4 的输出）

**H1 事件重叠度检查**（逐章比对）：
- 逐章读取本卷事件链的"核心事件"字段
- 与白皮书 §一 中原著对应阶段/卷的事件摘要做事件类型比对
- 重叠定义：事件类型相同 + 角色功能相同 + 结果方向相同 → 计为 1 次重叠
- 统计重叠占比 = 重叠章数 / 本卷总章数

**H2 差异化衰减检查**（趋势分析）：
- 对比前 5 章与后 5 章的重叠率
- 若后 5 章重叠率 > 前 5 章重叠率 + 10% → 判定为"差异化衰减"→ WARN

**H3 core_diff 落地检查**：
- 从 差异化约束表.json 中读取本卷的 required_new_event_types
- 在本卷事件链中检查这些事件类型是否真实出现
- 缺 ≥1 个 → "⚠️ 本卷缺认可新事件类型：{list}"

**输出结果**：
| 章号 | 核心事件摘要 | 事件类型 | 原著对应事件 | 重叠判定 |
|------|------------|---------|-------------|:-------:|
| Ch1  | 灵力测试三段→全城嘲讽 | 测试羞辱 | 测试羞辱（原著 Ch1） | ⚠️ 同类型 |
| Ch2  | 莫寒夺宅→古丹觉醒 | 资源冲突 | 退婚羞辱（原著 Ch1） | ✅ 变形 |
| Ch3  | 丹尘子觉醒→血脉封印揭示 | 导师觉醒 | 药老觉醒（原著 Ch2） | ⚠️ 同类型 |
| ...  | ... | ... | ... | ... |

**判定标准**（与 差异化约束表.json 的 required_min_overlap_rate 比较）：
- 重叠占比 < 0.3 → ✅ PASS
- 重叠占比 0.3-0.4 → ⚠️ WARN（标记注入包，传给 plot_planner 时附加 DIFF_WARNING）
- 重叠占比 ≥ 0.4 → ❌ FAIL（触发 rebuild）

**效果**：
- 30% 是宽松阈值——按 v3 的实际数据（Ch1 15、Ch5 86），前 5 章平均重叠约 10-15%，远低于 30%
- 40% 是严格阈值——如果一卷中有 40% 的章 + 原著同构，说明整卷骨架高度相似，需要重建
- WARN 区间是"可优化但不需要重建"——plot_planner 的 L2 替换在这里发挥作用
```

#### 4.2 Phase 级 rebuild 机制

**背景**：当前 `destiny_designer` 支持 `rebuild_volume = {N}`，但最小重建单元是整卷（~30 章）。如果只有 1 个 phase 重叠度高（如 9 章中的 5 章），重建整卷 30 章是不划算的。

**新增参数**：`rebuild_phase = {phase_index}`

```
传入：
  rebuild_volume = 1        // 卷 1
  rebuild_phase = 0         // 仅重建 phase 0（Ch1-Ch9）

行为：
  1. 读取 00-素材/差异化约束表.json 中卷 1 的约束
  2. 读取 01-人物命运谱/ 中各角色的本卷初状态
  3. 读取 03-伏笔命运谱/ 中"埋设卷=1 且 回收卷=1"的伏笔
  4. 仅重建 Phase 0（Ch1-Ch9）的事件链
  5. 更新 02-剧情命运谱/卷01-剧情.md 中 Phase 0 的事件表
  6. 更新 05-卷级注入/卷01-注入包.md 中对应段落
  7. 执行步骤 8 自检（含 H 项），仅检查本 Phase

  不修改：
    - Phase 1-3 的事件表
    - 03-伏笔命运谱/ 中跨卷伏笔
    - 01-人物命运谱/ 中跨卷角色状态
    - 00/04 全局文件
```

**Phase 级重建的前提条件**（全部满足才准执行）：

| 条件 | 说明 | 保护措施 |
|:---|:---|---:|
| Phase 划分固定 | 盐值 `volume_rhythm_profile.phases` 的定义不会在重建中改变 | 重建时读取 phases 定义，不做修改 |
| 输入状态不变 | 新 Phase N 的开始状态 = 旧 Phase N-1 的结束状态（旧 Phase N-1 没有重建） | 拼接校验：重建后检查本 phase 的输入状态与上一 phase 的输出状态是否一致 |
| 输出状态不变 | 新 Phase N 的结束状态 = 旧 Phase N 的结束状态（给 Phase N+1 的接口不改变） | 强制性约束——重建时不可改变本 phase 的结束状态。若需改变，则降级为全卷 rebuild |
| 跨 phase 伏笔保护 | 重建 phase 时，被 03 标注为"埋设卷=本卷、回收卷>本卷"的伏笔必须保留 ID 不变 | 重建后做伏笔连续性校验：03 中所有"埋设卷=本卷"的伏笔 ID 在新事件链中是否仍存在 |

**不符合前提条件时**——如果 H 项检查发现的差异化问题涉及跨 phase 的伏笔或角色状态 → 不能做 phase 级重建，降级为全卷 rebuild。

**省时评估**：

| 场景 | 重建范围 | 省时比例 | 适用条件 |
|:---|:---|---:|:---|
| H 项 FAIL 仅因 1 个 phase | 1 个 phase（如 9/30 章） | ~70% | 该 phase 不涉及跨 phase 伏笔和角色状态变更 |
| H 项 FAIL 涉及 2 个 phase | 2 个 phase（如 18/30 章） | ~40% | 同 |
| H 项 FAIL 且涉及跨 phase 元素 | 全卷（30 章） | 0%（降级） | 跨 phase 伏笔或角色状态变化时 |
| H 项 WARN（不需要 rebuild） | 不做 rebuild | — | plot_planner 的 L2 替换覆盖 |

**工作量评估**：
- `destiny_designer` 的 prompt 增加 ~100 行（H 项检查 + Phase 级重建逻辑 + 拼接校验）
- 这是所有改动中最大的一个，但也是最关键的一个

### L5: `plot_planner` — 注入 L2 替换能力（局部差异化微创新）

**改动**：在 `plot_planner` 的「确定性细化模式」中，新增一个例外规则。

**核心原则**：**"保持核心事件不变，允许替换事件的具体形态"**

#### 5.1 触发条件

plot_planner 在以下情况获得 L2 替换权限：

1. **调用方传入 DIFF_WARNING 标记**：当 chief_editor 在注入包中检测到 `"diff_warning": true`（由 destiny_designer 的 H 项 WARN 产生）时，传给 plot_planner 时附加 `diff_warning_chapters = [N, M, ...]`
2. **连续 3 章以上同类型爽点**：plot_planner 在规划章纲时检测到连续 3 章主爽点类型相同 → 自动触发 L2 替换（不等待外部标记，这是内部的差异化防护）
3. **连续 2 章以上场景类型相同**：如"矿洞探索→还是探索" → 触发 L2 替换

触发条件 1 是"外部指令"，条件 2 和 3 是"内部自检"——plot_planner 的批量检查清单中已有爽点轮换检查，可以自然扩展。

#### 5.2 替换边界（L2 vs L1）

| 可替换（L2 范畴） | 不可替换（L1 范畴，仍归 destiny） |
|:---|:---|
| ✅ 事件发生的时空环境（早晨→深夜、室内→户外） | ❌ 核心事件本身（送药不能改成送秘籍） |
| ✅ 触发方式（主动→被动触发→偶然触发） | ❌ 事件的结果输出（不能改变爽点/信息释放目标） |
| ✅ 配角反应层（增加原著没有的反应细节） | ❌ 伏笔的埋设和回收（伏笔 ID 和类型不能变） |
| ✅ 附加线索（在原事件上增加小细节/暗记） | ❌ 出场角色（不能无故增减） |
| ✅ 叙事角度（纯主角视角→第三方闪入再切回） | ❌ 注入包声明的"不可触碰内容" |

#### 5.3 替换方向池

为了让 Flash 模型的 L2 替换有据可依，可以在 `plot_planner` 的 prompt 中提供替换方向池：

```markdown
### L2 替换方向池（仅 DIFF_WARNING 章可用）

当需要替换事件形态时，从以下方向中选择至少 1 个：

| 方向 | 说明 | 示例 |
|:---|:---|---:|
| 环境翻转 | 将公开场景改为私密场景，或反之 | 测试广场羞辱→改为私下对峙羞辱 |
| 触发翻转 | 将被动触发改为主动触发，或反之 | 被动捡到古丹→改为血脉感应引导找到古丹 |
| 配角反应增强 | 在原事件中增加一个多层级的配角反应 | 围观者→改为三个不同立场的围观者各有反应 |
| 附加线索植入 | 在原事件骨架上叠加一条原著中没有的线索 | 送药材→药材篮底留下远古苏族暗记 |
| 代价前置 | 在爽点到来前展示代价，而不是爽点后才展示 | 突破成功后白发→突破过程中白发逐现 |
| 视角切换 | 用旁观者视角开头再切回主角视角 | 从莫寒视角看苏夜的测试结果 |
```

替换方向池不是让 plot_planner 自由发挥——它需要从中选一个方向，并按照该方向的具体模板执行替换。方向池本身也可以从盐值的 `core_diff` 或 `presentation_variants` 中动态扩展。

#### 5.4 替换后的安全性检查

plot_planner 完成 L2 替换后，在章纲中需要标注：
- 本节使用了 L2 替换：方向 = _____
- 替换范围：__________（具体描述改了哪里）
- 未修改项：核心事件 ☑、伏笔 ☑、出场角色 ☑

chief_editor 在接收章纲后做一次合规性校验：
- ✅ 核心事件未变
- ✅ 注入包声明的伏笔 ID 在章纲中仍出现
- ✅ 注入包声明的出场角色在章纲中仍出现

**工作量评估**：
- `plot_planner` 的 prompt 增加 ~80 行（L2 替换定义 + 替换方向池 + 安全性检查）
- `chief_editor` 的 prompt 增加 ~20 行（合规性校验）

## 五、方案对比总表

| 方案 | 改动 agent | 改动量 | 解决的问题 | 独立效果 | 相互依赖 |
|:---|:---|---:|:---|---:|:---|
| **L1** original_analyst +§X | 1 个 | ~50 行 | 上游无"雷区地图" | 中 | 无依赖，可独立实施 |

| **L3** salt_architect Rule 15-18 | 1 个 | ~60 行 | 盐值无差异化元能力检查 | 低（只拦 Phase 1，不拦 Phase 2） | 依赖 L1（Rule 17 有条件激活），L2（Rule 16/18 需要约束表） |
| **L4** destiny_designer H 项 + Phase rebuild | 1 个 | ~100 行 | **事件链生成后无差异化自检** | **极高（这是最核心的改动）** | 强依赖 L2（H 项检查需要约束表作为比对基准） |
| **L5** plot_planner L2 替换 | 2 个（+chief_editor） | ~100 行 | 确定模式剥夺了局部创意弹性 | 中高（解决的是"微调"而不是"判别"） | 依赖 L4（DIFF_WARNING 标记来自 L4 的 H 项） |

### 关键依赖链

```
L1 (辨识度清单) ──→ L2 (约束表, 利用辨识度增强精度)
                     ├──→ L3 (Rule 17/18, 需要约束表和白皮书)
                     └──→ L4 (H 项, 需要约束表作为比对基准)
                              └──→ L5 (DIFF_WARNING 标记来自 H 项)
```

### 最小可行改动集合

如果只做最小改动获得最大收益：

```
L4 (destiny_designer H 项 + Phase rebuild) + L2 (master_outline 约束表)
```

解释：

- L4 是**最核心的拦截点**——它在事件输出前做自检，是整个方案中"杠杆率最高"的改动
- L2 是 L4 的必要前置——没有约束表作为比对基准，H 项自检缺乏标准
- L1 可以后补——没有 §X 清单时，约束表可以用更粗糙的原生比对（只用白皮书 §一的事件摘要）
- L3 可以后补——没有 Rule 15-18 时，盐值质量靠人工把关
- L5 可以后补——没有 L2 替换时，DIFF_WARNING 可以通过人工介入处理

**但用户强调重视 L5（plot_planner 创作功能）**：
如果恢复 plot_planner 的 L2 替换是核心诉求，那么最小改动集合变为：

```
L4 + L2 + L5
```

L4 负责"判断是否太像"（判别层），L5 负责"如何改得不一样"（执行层）。两者一个是"质检"，一个是"修正"，缺一则空。

## 七、风险与应对总表

| 风险 | 涉及层 | 缓解措施 |
|:---|:---:|:---|
| H 项判断标准过大（漏报）或过严（误报） | L4 | 分级预警（PASS/WARN/FAIL），WARN 不阻塞管线，只做标记 |
| Phase 级重建后跨 phase 伏笔断裂 | L4 | 拼接校验 + 降级机制（不符合条件时自动降级为全卷 rebuild） |
| plot_planner L2 替换后与后续卷冲突 | L5 | L2 替换不允许改变伏笔和角色状态，不影响跨卷一致性 |
| Flash 模型替换质量不足 | L5 | 替换方向池提供结构化模板，不是自由发挥 |
| Layer 间配合数据格式不一致 | L2→L4→L5 | 统一用 JSON 作为层间数据交换格式，各层只读写固定字段 |
| H 项检查增加 destiny_designer 的 token 消耗 | L4 | 事件类型序列比对是 keyword-level，token 消耗极低（比对 30 章 vs 30 章事件类型字符串） |
| 差异化约束表.json 的维护成本 | L2 | 36 行/本书，由 master_outline_generator 自动生成，不增加人工维护 |

## 八、最终结论

```
原始问题：写出来的内容和原著过于相似。

根本原因：
  destiny_designer 掌握了全部事件设计权，
  但其 prompt 没有差异化自检职责，
  v3 的确定性模式让下游无法纠正。

解决方案：
  五层防御链，核心是 L4（destiny_designer H 项自检）+ L2（卷级约束表）+ L5（plot_planner L2 替换）。
  三层互为依赖：L2 提供标准 → L4 执行判断 → L5 执行修正。
  最小可行改动：L4 + L2（先能"发现问题"），再补 L5（再能"解决问题"）。

设计原则：
  1. 职责归位——谁决策事件谁负责差异化（destiny_designer）
  2. 分级预警——不是二元的 through/fail，是 PASS/WARN/FAIL
  3. 有限修正——plot_planner 有 L2 替换权但没有 L1 创造权
  4. 最小重建——Phase 级 rebuild，省时 60-80%
  5. 渐进实施——先核心后外围，不改管线整体结构
```
