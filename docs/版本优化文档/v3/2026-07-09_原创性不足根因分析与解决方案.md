# 原创性不足根因分析与解决方案

**日期：2026-07-09**
**分析对象：v3 管线（斗破苍穹 → 灵渊天火录）**
**核心问题：仿写内容与原著情节骨架过度相似，签约审稿维度 9（原创改写度）仅 10/15（-5 分）**

---

## 一、问题现象

根据 `总结性审核报告-v3.md` 和 `审稿报告-v3.md`：

> "全书主干情节'测试→祖宅被夺→捡古丹→导师觉醒→突破→碾压→探索天火'与斗破苍穹原著的映射关系过于直接。前三章的每一节核心事件在原著的对应章节中均可找到位置相近的原型。"

> "从 CH2 开始，差异化约束的力度递减...总纲的差异化策略主要覆盖了'开场'场景，而未对后续章节的'情节骨架差异化'提供持续约束。"

> "章节差异度从 Ch1(95)→Ch5(86) 呈下行趋势"

**核心矛盾**（审核报告自述）：

> "v3 的核心矛盾是工程化水平提升 vs 创作差异化下降。高精度约束体系在提高执行一致性的同时，也限制了 agent 的创意自由。v2 在较宽松的约束下反而拿到了原创改写度满分——这暗示存在一个'约束精度 vs 创意空间'的最优均衡点。"

---

## 二、责任链追溯：谁在管原创性？

### 2.1 各 agent 对「事件 WHAT」的控制权

| Agent | 创造 WHAT（事件） | 设计 HOW（执行方式） | 是否有差异化职责 |
|-------|:---:|:---:|:---:|
| `original_analyst` | — | — | 提取方法论（拆法不拆事），不设计事件 |
| `master_outline_generator` | 框架级策略（开场差异化、爽点轮换、节奏参数） | — | 有 §9.5 差异化声明，但**仅覆盖开场** |
| **`destiny_designer`** | **每章核心事件、角色命运、伏笔网络** | — | **无**（自检无差异化检查） |
| `plot_planner`（确定模式） | ❌ 明确被告知"不创造，只细化" | 五段式展开、钩子设计、爽点定位 | **无**（仅有执行层检查如爽点轮换/钩子类型） |
| `content_writer` | ❌ 严格遵循章纲 | 对白、描写、文笔 | **无**（70/20/10 句式变异仅限文笔层面） |
| `quality_reviewer` | — | — | **无**（11 个检查维度无一涉及原著情节重叠度） |

### 2.2 关键发现

- `destiny_designer` 拥有全管线最大的事件创造权（决定了 1260 章的 WHAT），但它的 prompt（475 行）中**完全没有**「差异化/原创性」职责要求。
- `plot_planner` 在 v3 的"确定性细化模式"下被剥夺了创造事件的能力，无法充当创意安全阀。
- `quality_reviewer` 在 Phase 2 每章都打过分，但从未发现"这章和原著太像"——直到 Phase 3 的 `signing_reviewer` 才发现问题（但那时已成终稿）。

---

## 三、根因分析

### 根因 #1：`destiny_designer` 缺失「差异度自检」（职责漏洞·最关键）

`destiny_designer.md` 的步骤 8 自检清单包含 7 大类检查：

- ✅ A. 文件存在性
- ✅ B. 逐章细化度
- ✅ C. 逐行完善度
- ✅ D. 跨文件一致性
- ✅ E. 叙事一致性
- ✅ F. 注入包完整性
- ✅ G. 总纲一致性（角色/爽点/节奏/标签）

**完全缺失**：
- ❌ 每章核心事件与原著的「情节重叠度」检查
- ❌ 连续 N 章主干事件与原著「同构」的预警机制
- ❌ 对 salt 中 `core_diff` 策略的逐章落地验证

这是 `destiny_designer` prompt 设计中最关键的缺失项。当这个 agent 回归到原著情节惯性时（它在生成 1260 章的逐章事件时天然会受白皮书中的方法论倾向影响），整个下游管线无法自救。

### 根因 #2：`core_diff` 只在框架级定义，缺少逐章/逐卷的差异化细则

`project_salt.json` 的 `core_diff` 定义了三维差异化：
- 角色动机：证明自己 → 守护家族
- 节奏：20-30 章大高潮 → 35-45 章
- 爽点：禁用身份打脸，新增血脉共鸣/势力经营/代价揭示

这些是**框架级策略**，只有「方向性声明」。实际落地到 1260 章的逐章事件设计时，没有具体到「第 X 章的原著对应节点是什么，本仿写必须偏离多少」的粒度。而 `destiny_designer` 需要更具体的约束才能有效执行。

### 根因 #3：v3 "确定性细化模式"剥夺了 plot_planner 的创意安全阀作用

在 v3 的确定模式下，`plot_planner` 被明确告知：

> "你的创作自由度从'可以发明事件'收窄为'在给定事件骨架上设计最好的实现方式'"

对比 v2（没有 destiny_designer）：
- v2：plot_planner 拥有创造事件的自由 → 原创改写度 15/15 满分
- v3：事件被 destiny 锁定，plot_planner 无法偏离 → 原创改写度 10/15

当 destiny_designer 给出的 events 已有高重叠度时，整个下游管线被锁死了。

### 根因 #4：`salt_architect` 的 14 条校验规则中没有「情节差异度」检查

`salt_architect` 有 14 条规则，覆盖了合规性、去重、标签完整性、爽点轮换、钩子传递、反派弧线等。但没有任何一条检查「salt 中的差异化声明是否足以支撑全书的逐章差异化需求」。框架层通过了盐值校验，但盐值本身的差异化深度不够。

### 根因 #5：Phase 2 没有原创性预检，Phase 3 才发现但已无法返工

Phase 2 写了 5 章正文，每章通过 `quality_reviewer` 打分，但没有一个环节检测到"这 5 章和原著太像了"。等到 Phase 3 的 `signing_reviewer` 才发现问题——但那时已经是"终稿"，只能打分，不能返工。

---

## 四、解决方案分析

### 方案一：增强 `destiny_designer` 的差异化职责（最小改动，最高杠杆）

**地位**：最直接的解法。destiny_designer 管了所有事件的 WHAT，在它这里修效果最大。

**改动内容**：

1. **新增输入 7**：传入「原著章节事件摘要」——由 `original_analyst` 的白皮书 §一 提取。让 destiny_designer 在设计每章事件时能**对照**原著对应位置。

2. **在步骤 4（生成逐章事件链）中新增「差异度约束」**：
   - 每章核心事件不得与原著对应章节的事件"同构"（同构 = 事件类型相同 + 角色功能相同 + 结果方向相同）
   - 每 5 章为一批次，连续 3 章以上的核心事件不得与原著呈"一一映射"关系
   - 对盐值 `core_diff` 中声明的三维差异（角色动机/节奏/爽点），逐章验证是否落地

3. **在步骤 8（自检）中新增 H 项「差异化自检」**：
   - H1：逐章比对——每章核心事件与原著对应节点的重叠度评级（无/低/中/高），≥3 章为"高"则禁止通过
   - H2：差异化衰减检查——前后 5 章差异度是否存在下行趋势（参考 v3 的 95→86 教训）
   - H3：core_diff 策略的逐章落地覆盖率

**优点**：
- 改动最集中，只改一个 agent
- 解决的是根本矛盾（destiny 管 WHAT 但不管差异化）

**风险**：
- destiny_designer 的 prompt 已经 475 行，增加差异化职责后会加重
- V4 Pro 模型能否在"生成 1260 章事件"的同时兼顾"逐章比对"是未知数——可能超时
- 如果比对逻辑依赖于"寻找原著对应章节"，需要可靠的章节映射关系

---

### 方案二：新增「差异化校验 agent」，夹在 destiny 和 plot_planner 之间

**地位**：不在 destiny_designer 身上加负担，用独立 agent 做专项检查。

**流程变为**：
```
destiny_designer 生成事件链
  → 【新】differentiation_checker 逐章校验差异度
    → 差异度 ≥ 阈值 → 放行到 plot_planner
    → 差异度 < 阈值 → 标记"需重设计"，触发 destiny_designer rebuild_volume
```

**新 agent 职责**：
- 读取白皮书中的原著事件骨架
- 读取 destiny_designer 的事件链
- 逐章比对，输出「差异度矩阵」
- 对不达标的章节给出「差异化重设计建议」（事件类型替换方向、角色动机反转方向、环境/背景区隔方向）
- 建议必须具体（不能是"这里需要更不同"这种泛泛之谈）

**优点**：
- 单一职责，不过度增加 destiny_designer 的复杂度
- 输出的是"建议"，不破坏 destiny 的增量重建机制
- 可以在 Phase 2 写作前做（不等 Phase 3 才发现）

**缺点**：
- 增加一个新 agent + 一个新 pipeline step，系统复杂度上升
- 如果建议太泛（"这里需要更不同一些"），等于没用——需要 prompt 设计得非常具体
- 差异度判断本身需要相当的文学理解力，Flash 级别模型可能不够

---

### 方案三：给 `plot_planner` 恢复「创意偏离」的有限权力（结构层修复）

**地位**：恢复 v2 时代 plot_planner 的创意弹性，但加保险。

**改动内容**：

1. 在 `chief_editor` 加载注入包后，增加一步「差异度预判」：将注入包的事件链与白皮书中的原著事件骨架做模糊比对（用章节关键词匹配即可）。如果发现 ≥3 章的关键事件与原著同构，在传入 plot_planner 时附加 `DIFF_WARNING` 标记。

2. 在 `plot_planner` 的「确定性细化模式」中新增例外规则：
   > "若调用方在注入包中标记了某章为 `DIFF_WARNING`（差异度预警），则该章你可以将注入包的核心事件作为'方向'而非'硬约束'——可以在保持事件功能（爽点类型/角色出场/伏笔）不变的前提下，替换事件的具体形态。"

3. plot_planner 替换事件形态时，从 salt 的 `core_diff` 中抽取备选方向。

**优点**：
- 利用 plot_planner 已有的"设计 HOW"能力，恢复 v2 时代的创意弹性
- 但加了保险（只对标记章生效）

**缺点**：
- 引入不确定性——如果 plot_planner 修改了事件，可能与后续卷的 destiny 设计冲突（跨卷伏笔/角色状态引用可能断裂）
- plot_planner 是 Flash 模型，事件级创意生成的质量存疑

---

### 方案四：增强 `core_diff` 的颗粒度（框架层修复）

**地位**：从根源解决——让盐值/总纲中的差异化约束足够细，使 destiny_designer 有据可依。

**新增内容**（在 salt 或总纲中）：

1. **`per_chapter_differentiation`**：对前 N 章（如 50 章），逐章标注本仿写与原著的核心事件偏离方向：
   ```json
   {
     "ch_1": {"原著名事件": "测试+嘲讽+休书", "本仿写事件": "测试+嘲讽+夺宅", "差异维度": "冲突性质替换"},
     "ch_2": {"原著名事件": "药老觉醒+传授焚决", "本仿写事件": "丹尘子觉醒+血脉封印揭示", "差异维度": "导师触发方式差异+信息释放策略差异"},
     ...
   }
   ```

2. **`forbidden_event_types`**：沿袭 salt 中已有的 `forbidden_patterns` 机制，增加"禁止与原著的章节事件同构"的禁止清单。

3. **`differentiation_escalation`**：定义差异度应随章节数递增的策略——前 5 章差异 ≥30%，10 章后 ≥50%，30 章后应建立独立的剧情生态。

**优点**：
- 从根源解决问题——如果框架层的差异化约束足够细，destiny_designer 执行时有据可依
- 与盐值现有体系兼容（已有 `forbidden_patterns`、`tag_constraints` 等类似机制）

**缺点**：
- 谁来生成 `per_chapter_differentiation`？人工撰写等于手工设计全书，丧失自动化意义
- 如果让 `master_outline_generator` 来生成，它目前已经 94/S 高分，再加工作量可能降低其他维度的质量

---

### 方案五：增强 `original_analyst` 输出，增加「原著辨识度清单」

**地位**：改动在最上游，让白皮书直接提供"踩雷预警"，下游自动受益。

**新增白皮书章节**：

- **`§X 原著辨识度清单`**：标注原著中不可复用的高辨识度情节节点（如"退婚""三十年河东三十年河西""异火榜排名""黑袍炼药师大赛"等），并标注"若仿写使用同类节点，必须做何种程度的变形"。

- 这些标注给 `destiny_designer` 提供"踩雷预警"，也给 `quality_reviewer` 提供自动检测的锚点。

**优点**：
- original_analyst 是唯一定向读过原著全文的 agent，最有资格说"这些是原著的指纹"
- 改动上游，下游所有 agent 自动受益，不需逐个修改

**缺点**：
- original_analyst 需要额外执行一步分析，增加其 token 消耗
- 辨识度判断可能过于保守（"太像原著"的阈值不好定）

## 五、综合推荐：三层防护方案

| 层级 | 方案 | 作用 | 复杂度 |
|------|------|------|:---:|
| **L1 上游增强** | 方案五：original_analyst 输出「原著辨识度清单」 | 让 destiny_designer 知道"哪些是雷区" | ★★☆ |
| **L2 中游增强** | 方案一（轻量版）：destiny_designer 自检增加 H 项差异度检查 | 在事件链生成后、注入包输出前做第一次拦截 | ★★★ |
| **L3 下游门禁** | 方案二：新增 differentiation_checker agent | 在注入包传给 plot_planner 前做第二次拦截 | ★★★☆ |

### 最小可行改动（最快落地）

如果只要最少的改动获得最大收益：

**方案五**（original_analyst 加一节 §X：原著辨识度清单）+ **方案一轻量版**（destiny_designer 自检加 H 项）

只改两个 agent 的 prompt，不改流程结构，不改 pipeline 脚本。改动集中在最上游的两个 agent——一个制造差异化参考（original_analyst），一个执行差异化落地（destiny_designer）。

方案三（plot_planner 恢复创意权力）和方案四（core_diff 颗粒化）作为中期优化，不优先做——它们解决的是同一问题的不同侧面，先跑通最薄的三层再考虑。

## 六、补充：v3 与 v2 的关键差异回顾

| 指标 | v2（凡人修仙传） | v3（斗破苍穹） | 变化 |
|------|:---:|:---:|:---:|
| 签约审稿 维度 9（原创改写度） | **15/15 满分** | **10/15 扣 5 分** | **-5** |
| 章纲均分 | 96.8 | 91.2（Ch5=86） | -5.6 |
| 卷纲 | 97.0 | 85.0 | -12.0 |
| 签约总分 | 90.0 | 83.0 | -7.0 |
| 综合加权 | 85.36 | 86.42 | +1.06 |

v3 在工程化上的进步是真实的（总纲 salt 9 万字符、零模糊指令体系、品质验证前移），但这些进步在"原创性"这个维度上产生了反效果：

> "高精度约束体系在提高执行一致性的同时，也限制了 agent 的创意自由。"

v2 没有 `destiny_designer`，`plot_planner` 自己创造事件——可能走偏，也可能走出原著引力。v3 用确定性取代了不确定性，代价是也取代了创意偏离的可能性。解决方案需要在"确定性"和"差异化"之间找到新的平衡。
