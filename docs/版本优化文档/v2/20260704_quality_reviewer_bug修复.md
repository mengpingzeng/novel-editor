```
Step 7: 卷纲规划 + 评分 (GLM 5.2)

  ├── 8a: 章纲 (plot_planner)
  ├── 8a-评分: 章纲评分 (chapter_outline_reviewer, GLM 5.2)
  ├── 8b: 正文初稿 (content_writer)
  ├── 8c: 质检 (quality_reviewer)
  ├── 8d: 记录章节名到 novel_metadata.json

Step 10: 审核报告
```

### 1.2 Step 8c 当前行为 (book_factory.md:355-368)

```

评分 < 60 → 日志标记 ⚠(分数) — 未通过，继续写下一章
```

**核心问题：未通过不触发任何重写动作，直接跳过进入下一章。**

## 三、优化目标

### 3.1 核心需求

当 `quality_reviewer` 对某章评分为 <60 分（未通过）时，将该章的审核结果（纪要）回传给 `content_writer`，让其基于反馈重写该章，而非直接跳过。

### 3.2 约束条件

1. **最多重写 3 次**（初写 1 次 + 重写 2 次）
2. **每次重写只基于最新一份纪要**，不累积历史纪要（避免输入信息爆炸）
3. 循环结束后取分数最高版本为终稿
4. 所有版本 > 3 次或连续退化则终止并告警
5. 每轮重写独立记录到 execution-log

#### 4.2.3 新增重写提示词段落

在现有写作指导后追加一个独立的「重写模式」章节：

```
## 附：重写模式（mode=rewrite）

当 mode=rewrite 时，除常规输入外，你还需读取 ./03-纪要/第N章纪要.md。

重写原则：
1. 以现有初稿为基础修改，而非从零重写（除非总分 <40）
2. 逐条对照纪要中的「逐条修改建议」修正
3. 优先解决 D2（文风仿写）和 D3（爽点落地）维度的扣分项
4. 保持章纲核心事件不变，不得自行扩展支线
5. 重写后重新自检：字数预估|忠于章纲|标签覆盖|合规无风险
6. 总分 <40 时：判定为结构性失败，从章纲重新撰写全章
```

#### 4.2.4 重写模式下的输入优化

重写时只读「最近 1 章纪要（前章）+ 当前章纪要」，从 3 份减少到 2 份，避免上下文膨胀。

### 4.3 book_factory.md 具体改动

#### 4.3.1 Step 8c 重构为循环

将当前单次调用的 Step 8c 替换为：

```
#### 8c. 质检 + 重写循环（最多 3 轮）

1. 初始化：retry = 0, best_score = 0, best_version = null
2. 调用 @content_writer mode=fresh → 输出 第N章-初稿-v1.md
3. LOOP:
   a. 调用 @quality_reviewer → 读取 第N章-初稿-v{retry+1}.md → 输出 第N章纪要-v{retry+1}.md
   b. 解析返回的分数 score
   c. 记录日志: | {now} | 第N章质检(第{retry+1}轮) | quality_reviewer | team-deepseek/deepseek-v4-flash | ✅({score}分) |
   d. IF score > best_score → best_score = score, best_version = retry+1
   e. IF score ≥ 60 → BREAK (通过)
   f. IF retry ≥ 2 → BREAK (已达最大重试次数)
   g. IF 连续 2 轮评分下降 → BREAK (退化终止)
   h. retry++
   i. 调用 @content_writer mode=rewrite → 输入含 第N章纪要-v{retry}.md → 输出 第N章-初稿-v{retry+1}.md
   j. GOTO 3a
4. 循环结束 → 取 best_version 对应的初稿复制为 第N章-终稿.md
5. 纪要处理：取 best_version 对应的纪要作为终稿纪要，删除中间版本
6. IF best_score < 60 → 日志标记 ⚠({best_score}分) — 未通过(已重写{retries}次)，继续
7. 追加最终日志: | {now} | 第N章重写完成 | - | - | ✅(最佳{best_score}分，第{best_version}轮) |
```

#### 4.3.2 文件名规范

| 文件类型 | 初写 (v1) | 重写第1轮 (v2) | 重写第2轮 (v3) |
|---------|-----------|---------------|---------------|
| 正文 | `第N章-初稿-v1.md` | `第N章-初稿-v2.md` | `第N章-初稿-v3.md` |
| 纪要 | `第N章纪要-v1.md` | `第N章纪要-v2.md` | `第N章纪要-v3.md` |
| 终稿 | — | — | `第N章-终稿.md`（循环结束复制） |

#### 4.3.3 execution-log 格式更新

新增重写轮次的日志记录，示例：

```
| 2026-07-04 12:00:13 | 第1章初稿(v1) | content_writer | team-deepseek/deepseek-v4-flash | ✅(2050字) |
| 2026-07-04 12:00:14 | 第1章质检(第1轮) | quality_reviewer | team-deepseek/deepseek-v4-flash | ⚠(56分) — 未通过 |
| 2026-07-04 12:00:15 | 第1章重写(v2) | content_writer | team-deepseek/deepseek-v4-flash | ✅(2100字) |
| 2026-07-04 12:00:16 | 第1章质检(第2轮) | quality_reviewer | team-deepseek/deepseek-v4-flash | ✅(73分) — 通过 |
| 2026-07-04 12:00:17 | 第1章重写完成 | - | - | ✅(最佳73分，第2轮) |
```

## 五、潜在问题与对策（设计者未注意到的点）

### 5.1 文件名覆盖 vs 多版本留存

**问题**: `content_writer` 当前固定输出 `第N章-初稿.md`。直接覆盖写入会丢失初版，后续无法取回最高分版本。

**对策**: 每轮写入不同版本号文件（`第N章-初稿-v1.md` ~ `v3.md`），循环结束后取最高分版本复制为终稿，中间版本可保留供排错。

### 5.2 content_writer 的「全文重写」vs「局部修订」边界

**问题**: 低分情况有两种：
- 总分 <40：通常所有维度都崩了，需要从章纲重新出发，全文重写
- 总分 50-59：仅个别维度扣分严重，应按纪要逐条局部修订即可

如果一律全文重写，50-59 分的章节可能越改越差。

**对策**:
1. 在 `content_writer` 重写模式中增加判定：总分 <40 → 全文重写；≥40 → 局部修订
2. 或者由 `quality_reviewer` 的返回中增加一个 `重写模式: 全文/局部` 字段，由 `book_factory` 解析后传入

### 5.3 quality_reviewer 跨章检查的纪要版本污染

**问题**: `quality_reviewer` 的输入 3 是「最近 3 章的纪要文件」。当前章重写时生成了 v1/v2/v3 多份纪要，`quality_reviewer` 在检查后续章时可能误读当前章的中间版纪要。

**对策**: `book_factory` 在调用 `quality_reviewer` 做跨章检查时，必须过滤掉非终稿版本的纪要文件（只保留每章的终稿版本或最高分版本）。

### 5.4 上下文累计风险

**问题**: `content_writer` 已有 3 个输入（总纲领 + 章纲 + 2 章纪要），重写时新增当前章纪要（~200行），单次调用输入约 30-50KB。3 轮重写不会递归膨胀（每轮只传最新一份纪要），但仍需关注总纲领的长度。

**对策**:
1. 重写时建议 `content_writer` 将「2 章纪要」降为「1 章纪要」，从 4 份输入控制回 3 份
2. 总纲领建议只读取关键章节（平台适配 + 分类标识 + 文风句式），不全文加载
3. 当前 `input_monitor` 已经在记录每步的输入大小，可利用该数据监控阈值告警

### 5.5 改写循环的「退化」风险

**问题**: AI 反复改写同一章可能出现不可预期的退化：
- 越改越差（新引入 G1-G11 违规）
- 文案疲劳（语言变得生硬机械）
- 章纲偏离（潜意识中产生偏移）

**对策**:
1. 每轮保留分数记录，取最高分版本为终稿（而非最后一版）
2. 如果连续 2 轮评分下降，提前终止循环
3. 终稿仍然 <60 时，日志标记但不阻断流水线（保持现有行为）

### 5.6 评分标准的一致性

**问题**: `quality_reviewer` 依赖 Grep 工具扫描 + 主观评分，同一章节的多轮评审可能因 AI 的随机性产生评分波动（同一版本可能给 56 分和 62 分）。

**对策**:
1. `quality_reviewer` 的 temperature 已经是 0.3（较低），有助于评分一致性
2. 设置评分波动容忍带：±5 分内的波动视为持平，不计入「连续下降」判定
3. 保守策略：任何一轮 ≥60 即视为通过，不要求维持

### 5.7 跨章一致性在重写场景下的退化

**问题**: `content_writer` 重写一章时，只基于最新一份纪要的反馈。但如果重写大幅修改了剧情走向（如改变了某角色的态度、某物品的状态），可能与已通过的后续章产生矛盾。

**对策**: 当前流水线是逐章顺序生成（8a→8b→8c→8d→8e，第N章通过后才进入第N+1章），所以在第N章重写时，后续章尚未生成，不存在已通过章节的矛盾问题。但需确保 `content_writer` 重写时的「最近两章纪要」指向的是前 N-2 和 N-1 章的**终稿纪要**，而非中间版本。

### 5.8 重写时间成本

**问题**: 每章 3 轮重写 = 6 次 agent 调用（3 写 + 3 审），是目前的 3 倍时间。

**对策**:
1. 设定超时阈值，单章总耗时超过 N 分钟则跳过剩余重试
2. 可在 `book_factory` 中增加全局开关：`MAX_REWRITE_RETRIES`，可配置为 0/1/2
3. 通过 `execution-log.md` 的总耗时字段监控整体效率
