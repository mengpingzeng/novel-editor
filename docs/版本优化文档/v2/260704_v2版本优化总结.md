## 详细变更

### 1. 自动化处理日志时间修复

**问题**：`book_factory.md` 中 `{now}` 占位符被 AI 逐行递增 1 秒（如 12:00:00, 12:00:01...），但实际每步耗时远超此值。

**修复**：
- `book_factory.md` 核心原则新增："时间戳必须通过 bash `date '+%Y-%m-%d %H:%M:%S'` 获取真实系统时间，禁止编造"
- 同时将 `execution-log.md` 改名为 `自动化处理日志.md`（涉及 `book_factory.md` 和 `automation_manager.md`）
- 已生成的 dryrun 日志同步改名

---

### 2. "「」" → """ 去 AI 感

**问题**：agent 定义文件中大量使用中文引号 「」表示对话/概念引用，AI 感强。

**修复**：全局替换 28 个文件中的 83 处 「」为英文双引号 ""，覆盖 `.opencode/agents/` 和 `project-agents-template/.opencode/agents/`。

---

### 3. novel_metadata.json 迁移到"发布/"目录

**问题**：`novel_metadata.json` 以 `workspace/books/{name}/novel_metadata.json` 写在版本目录外。

**修复**：
- 路径改为 `workspace/books/{name}/versions/{version}/发布/novel_metadata.json`
- Step 0 初始化新增 `发布/` 目录创建
- `发布/` 与 `00-素材`、`01-大纲`、`02-正文`、`03-纪要` 同级，不带数字前缀

---

### 4. quality_reviewer 重写循环

**问题**：`book_factory` Step 8c 中，评分 <60 分时直接跳过进入下一章，不触发任何重写动作。

**修复**：
- **content_writer**：新增 mode 参数（`fresh` / `rewrite`），rewrite 模式下读入本章质检纪要作为反馈；新增"重写模式"章节
- **质检循环**：从单次调用重构为最多 3 轮（1 初写 + 2 重写），取最高分版本为终稿
  - 评分 ≥60 → 通过，跳出循环
  - 连续 2 轮下降（容忍带 3 分）→ 退化终止
  - 字数不合格（0 分）→ 特殊日志标记
- **文件版本化**：`第N章-初稿-v1.md` ~ `v3.md`，循环结束取最佳版本复制为终稿
- quality_reviewer 本身无需改动（输出已满足需求）

---

### 5. quality_reviewer 字数约束严格化

**问题**：旧 D1 最多扣 5 分（权重仅 5），字数完全失控也能靠其他维度拼到 60 分。

**修复**：
- D1 从阶梯式扣分改为三档：
  - `< 1000` 或 `> 3000` → **整篇 0 分**（硬性不合格，立即终止审核，不出具维度评分）
  - `[1000, 1500)` / `(2500, 3000]` → 线性过渡公式 `5 × (字数 - 1000)/500` 或 `5 × (3000 - 字数)/500`
  - `[1500, 2500]` → 满分
- 阈值改为固定绝对值，不再从总纲领的"平台适配"读取
- 新增 Flash 执行铁律第 11 条："字数不合格一票否决"

---

### 6. OpenCode Agents 三层解耦

**问题**：30+ agent 全部混在 `.opencode/agents/` 下；`book_factory` 单体承载全流程（上下文从 50K 膨胀至 225K+）；评分 reviewer 与写作 agent 无物理隔离。

**架构变更**：

```
Phase 1 框架层（novel-editor/.opencode/agents/ — 17 个 agent）
  automation_manager（精简版）→ 仅负责：白皮书 + 赛道 + 门面 + 盐值 + 总纲
  完成后输出 Phase 2 启动命令，停止。

Phase 2 内容层（workspace/books/<项目>/.opencode/agents/ — 5 个 agent）
  chief_editor → plot_planner + content_writer + quality_reviewer + data_operator
  独立上下文启动：opencode run --dir workspace/books/<书名>/
  完成后写入 .phase2_done

Phase 3 评价层（workspace/reviewer/.opencode/agents/ — 8 个 agent）
  reviewer_orchestrator（新建）→ 6 个 reviewer + pipeline_auditor + signing_reviewer
  独立上下文启动：opencode run --dir workspace/reviewer/
  完成后写入 .phase3_done
```

**关键变更**：
- **新建**：`workspace/reviewer/` 目录、`reviewer_orchestrator.md`、`pipeline_auditor.md`
- **迁移**：6 个 reviewer（signing/whitepaper/master_outline/volume_outline/chapter_outline + input_monitor）→ `workspace/reviewer/.opencode/agents/`
- **废弃**：`book_factory.md`（拆分为三阶段各自的编排者）
- **删除**：根目录 `plot_planner.md`、`content_writer.md`、`quality_reviewer.md`（以 `project-agents-template/` 版本为准）
- **精简**：`automation_manager.md`（移除 writing/reviewing/done 阶段）
- **重写**：`chief_editor.md`（新增 .phase2_done 写入 + input_monitor 数据采集）
- **上下文膨胀**：Phase 2 从 ~225K 降至 ~60K

### 7. pipeline_auditor 流程架构评分系统

**问题**：现有评分仅评估各层产物（白皮书/总纲/卷纲/章纲/内容），缺少对"整个流程是否提供了足够信息支撑 content_writer 写出好章节"的评估。

**新增 agent**：`workspace/reviewer/.opencode/agents/pipeline_auditor.md`

**核心方法论**：从章内容出发，逆向溯源每条信息的上游来源（章纲 → 卷纲 → 总纲 → 白皮书），标记"信息断裂点"。

**四维评分**：

| 维度 | 权重 | 检查项 |
|------|:--:|-------|
| A 开篇能力 | 25 | 白皮书是否提取了开场钩子方法论？总纲是否有差异化开场策略？前 3 章章纲是否有具体冲突？钩子类型是否无重复？ |
| B 中间丰富度 | 35 | 卷纲是否有明确的中间危机？章纲间伏笔延续率是否 ≥ 70%？爽点是否有渐进升级？人物关系是否在各章有推进？ |
| C 结尾防烂尾 | 20 | 总纲是否定义了结局高潮类型？终卷是否有 ≥ 3 章铺垫？后 20% 章节高潮分量是否明显递增？ |
| D 信息可执行率 | 20 | 章纲中模糊指令占比是否 < 10%？content_writer 五要素（角色动作/场景/冲突/字数/节奏）覆盖率？章纲要素可溯源性？ |

**集成位置**：Phase 3 `reviewer_orchestrator` 单书评分流程中，位于签约审稿之后、最终汇总之前。

## 一键执行脚本

为方便串联三阶段，提供了 `scripts/run_pipeline.sh`：

```bash
bash scripts/run_pipeline.sh dryrun    # 甄嬛传 × 3章 全流程自测
bash scripts/run_pipeline.sh step      # 全部 active_books 全流程
```

**step 模式执行流**：每本书独立跑 Phase 1→2→3，完成后再处理下一本。

```
书A: Phase1(框架) → Phase2(写书) → Phase3(审稿)
书B: Phase1(框架) → Phase2(写书) → Phase3(审稿)
...
跨书总结
```

**监控机制**：
- `opencode run --auto` 同步阻塞，直到 agent 完成全部子任务才返回
- 返回后轮询 `.phase*_done` 标记文件（每 15s，最多等 5min，防异步落盘）
- 中断后重跑自动跳过已完成的书籍（通过标记文件判断）
- 无超时限制，可跑数十小时
