# 盐值跨版本校验修复（2026-07-05）

## 现象

执行 `bash scripts/run_pipeline.sh step` 时，凡人修仙传 Phase 1 报错失败：

```
[14:25:02] ❌ 失败:  凡人修仙传
```

## 排查过程

### 1. 定位失败点

`run_pipeline.sh step` 对每本书调用 `run_phase1_for_book()` → `automation_manager` → `salt_architect`。凡修 Phase 1 失败后没有生成 `.phase1_done` 标记。

### 2. 发现中间问题

凡修的 `workspace/books/凡人修仙传/versions/v1/project_salt.json` 存在两处问题：

- **JSON 格式损坏**：`core_diff` 字段中嵌入了 4 个 ASCII 双引号 `"`（应使用中文弯引号 `""`），导致 `json.load()` 抛出 `JSONDecodeError`
- **缺少 `.phase1_done` 标记**：凡修在 V1 管线中已完整生成全部 Phase 1 产物（白皮书、赛道映射、门面候选、盐值、总纲）和 5 章正文，但 V1 的 `book_factory` agent 不写 `.phase1_done` 标记，导致 V2 管线误判为"未完成"

### 3. 透视架构问题

修复 JSON 和补标记只是表面处理。深层问题是 `salt_architect` 的校验规则强行跨版本关联，违背了版本隔离原则。

## 根源分析

### `salt_architect` 的原始设计

```markdown
输入4：同原作下所有历史盐值文件路径（用于去重对比）

校验规则 2a：与同原作下已有的历史盐值对比（重合度 ≤60%）
校验规则 2b：与同原作下已有的历史盐值的 tags 对比（重合度 ≤50%）
校验规则 4： 与同原作下已有的历史盐值的 book_title 对比（相似度 ≤60%）
校验规则 7c：与同原作下已有历史盐值的 opening_rotation 对比（重合度 ≤60%）
```

`global_manager:130` 也明确传入：
```markdown
传入同原作下所有历史盐值路径（workspace/books/{原作名}_salt_*/project_salt.json）
```

### 设计意图 vs 实际效果

| 设计意图 | 实际效果 |
|---------|---------|
| 同原作生成多个仿写版本时，确保各版本完全差异化 | 所有版本共享一个全局盐值池，v2 必须和 v1 比对 |
| 防止"改了个寂寞" | v1 的任何问题都会污染 v2（如损坏的 JSON） |
| 适用于同一本书的多次变异迭代 | 不适合"推翻 v1 从零生成 v2"的场景 |

### 矛盾点

版本递增的语义本身是：

> 上一版本有瑕疵 → 优化规则/流程 → 从零生成新版本

如果 v2 还要强制参考 v1 的盐值做"差异化校验"，等于承认 v1 仍然是参照系，v2 不能偏离太多——这和"推翻重来"矛盾。

此外，v1 是用旧管线（`book_factory`）生成的，无 `.phase1_done` 标记、无 `.opencode/agents/` 目录、JSON 可能存在格式问题。强制关联会导致旧数据拖累新管线。

## 修复方案

### 核心原则：版本完全隔离

```markdown
每个版本目录（v1/v2/v3）是独立的工作空间：
- 互不参考、互不依赖
- 所有校验仅在本版本目录内执行
- 禁止读取其他版本目录的任何文件
- 差异化校验仅对同一目录下重复生成有效
- 首次生成（本目录无历史盐值）→ 跳过去重
```

### 修改文件清单

| 文件 | 修改内容 |
|------|---------|
| `.opencode/agents/salt_architect.md` | 新增"版本隔离原则"置顶；规则 2a/2b/4/7c 改为"本版本目录下"；新增规则 2c（无历史盐值时跳过）；删除输入4（跨版本盐值路径） |
| `.opencode/agents/global_manager.md` | `workspace/books/{原作名}_salt_*` → `output_path 所在目录` |
| `workspace/books/凡人修仙传/versions/v1/project_salt.json` | 修复 4 个 ASCII 双引号 → 中文弯引号 |
| `workspace/books/凡人修仙传/.phase1_done` | 手动创建 |
| `workspace/books/凡人修仙传/.opencode/agents/` | 补复制 project agents |

### 修改前后对比（`salt_architect.md`）

```
- 输入4：同原作下所有历史盐值文件路径（用于去重对比）
+ 【版本隔离原则·最高优先级】（新增大段）
+ 2c. 若本版本目录下无历史盐值（首次生成）→ 跳过规则 2a 和 2b

- 2a. 与同原作下已有的历史盐值对比
+ 2a. 与本版本目录下已有的历史盐值对比

- 2b. 与同原作下已有的历史盐值的 classification.tags 对比
+ 2b. 与本版本目录下已有的历史盐值的 classification.tags 对比

- 标题差异化：与同原作下已有的历史盐值的 book_title 对比
+ 标题差异化：与本版本目录下已有的历史盐值的 book_title 对比。无则跳过

- 7c. 与同原作下已有历史盐值的 opening_rotation 对比
+ 7c. 与本版本目录下已有历史盐值的 opening_rotation 对比。无则跳过
```

## 经验总结

1. **版本号递增 = 隔离**。新版不应依赖旧版数据。如果需要参考，应该在调用侧显式指定，而非 agent 内部隐式搜索全局目录。
2. **标记文件是一等公民**。`run_pipeline.sh` 用 `.phase1_done` 判断是否可以跳过，agent 也应当用它判断状态，而不是根据目录结构猜测。
3. **旧管线产物的兼容性**。V1→V2 迁移时，旧产物可能缺少 V2 管线依赖的文件（如 `.phase1_done`、project agents），应有迁移脚本或防御性检查，而非让管线默默失败。
4. **JSON 编码规范**。agent 生成 JSON 时应避免在字符串值中直接嵌入未转义的 ASCII 双引号。建议 `json.dump(ensure_ascii=False)` 之前对所有字符串做一次 `"` 字符检测。
