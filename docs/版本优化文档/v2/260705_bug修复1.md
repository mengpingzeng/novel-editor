**日期**: 2026-07-05  
**影响范围**: Phase 2 (写书阶段)  
**状态**: ✅ 已修复

---

## 一、Bug 现象

运行 `bash scripts/run_pipeline.sh dryrun` 时：

1. Phase 2 在第 3 章质检环节停止，未能完成全部 3 章的生产
2. `versions/v2/发布/novel_metadata.json` 文件格式不符合 `novel_metadata.py` 定义的标准 schema
3. `versions/v2/01-大纲/` 下缺少第 2 章和第 3 章的章纲文件（仅有第 1 章）

### 运行日志片段（卡住位置）

## 二、根因分析

### 2.1 上下文溢出（主因）

**旧架构**：`run_pipeline.sh` 的 Phase 2 通过一次 `opencode run` 调用启动 `chief_editor` agent，该 agent 在**单个 session** 中处理全部 3 章的所有子任务（卷纲 → 每章的章纲 → 初稿 → 质检循环 → 记录）。

```bash
# 旧代码
run_phase "Phase 2 dryrun" \
    --dir "$dryrun_book_dir" \
    --agent chief_editor \
    "全自动执行第1~3章生产"     # ← 一个 session 跑 3 章
```

每次子 agent 调用（plot_planner、content_writer、quality_reviewer）的输出都累积在 chief_editor 的对话上下文中。

**上下文膨胀时间线**：

| 时刻 | chief_editor 上下文中的内容 | 累计估算 |
|------|--------------------------|---------|
| 启动 | chief_editor.md SOP (~480行) | ~20K |
| +读文件 | 仿写衍生总纲领.md(623行) + 卷纲(大文件) | ~70K |
| 第1章章纲 | plot_planner 输出 (~19K) | ~89K |
| 第1章初稿 | content_writer 输出 (~3K) | ~92K |
| 第1章质检 | quality_reviewer 完整报告 (~5K) | ~97K |
| 第2章章纲 | plot_planner 输出 | ~115K |
| 第2章初稿v1 | content_writer 输出 | ~118K |
| 第2章质检v1(59.68分) | 质检报告 | ~124K |
| 第2章初稿v2 | 重写(rewrite) | ~127K |
| **第2章质检v2** | 第二次质检报告 | **~133K** |
| 第3章章纲 | plot_planner 输出 | ~150K |
| 第3章初稿 | content_writer 输出 | ~153K |
| 第3章质检 | **半途崩溃** | **~160K+** |

**→ 到第 3 章质检时，上下文已超过 150K 字符，模型无法继续执行后续步骤（解析分数、判断重写、记录章节名）。**

关键恶化因素：第 2 章需要 2 轮重写（v1=59.68 分 → v2=75 分），导致上下文翻倍。

### 2.2 章纲文件丢失（副产物）

通过文件修改时间排序分析：

```
23:59:31  第1章章纲.md           ← 唯一成功写盘的章纲
00:21:16  第2章-初稿-v1.md       ← 章纲未写盘
00:44:26  第3章-初稿-v1.md       ← 章纲未写盘
```

**章纲不是被删除的，是从未被写入磁盘。**

早期（第 1 章），context 干净，chief_editor 能完整执行 SOP 的每一步（包括写章纲文件）。到第 2、3 章，上下文已膨胀到 130K+，模型开始跳过"保存章纲文件"这一步——plot_planner 返回了章纲内容到上下文中，content_writer 可以从中读取来写初稿，但文件落盘步骤被省略。日志中的 "✅" 是 subagent 调用完成即标记，不保证文件落盘。

### 2.3 novel_metadata.json 格式错误

| 对比项 | 旧 v2 文件（错误） | 标准格式（正确） |
|--------|------------------|-----------------|
| 书名 | `"book_title": "替身谋心"` (字符串) | `"title": ["书名1", "书名2", ...]` (数组，≥5个) |
| 章节 | `"chapters": [{...}]` (对象数组) | `"chapter_names": ["章名1", "章名2"]` (字符串数组) |
| 缺少字段 | — | `source`, `genre`, `protagonist`, `setting`, `shadow_intensity` 等 |

**原因**：chief_editor 的旧 SOP 只在 Step 3d 写了 `add-chapter` 调用，没有初始化步骤。`novel_metadata.py add-chapter` 要求目标文件已存在且格式正确。文件不存在时，openocode agent 自行创建了一个临时 JSON 而非使用 Python 脚本的标准 `create` 命令。

## 三、修复方案

### 3.1 架构改造：Scheme B（每章独立 session）

**核心变更**：将 Phase 2 从一个巨型 session 拆分为：

```
Session 0:  初始化 + 卷纲生成     (1 次调用)
Session 1:  第 1 章：章纲+初稿+质检  (第 1 次调用)
Session 2:  第 2 章：章纲+初稿+质检  (第 2 次调用)
Session 3:  第 3 章：章纲+初稿+质检  (第 3 次调用)
              ↓
         shell script 写 .phase2_done
```

每次 session 的上下文仅包含**当前一章**的内容，总量 < 50K 字符，远低于溢出阈值。

### 3.2 修改清单

#### 文件 1: `scripts/run_pipeline.sh`

**`run_book_pipeline()` Phase 2 块**：从一次 `run_phase "全自动执行第1~N章"` 改为：

```bash
# Phase 2.0: 初始化 + 卷纲（独立 session）
run_phase "Phase 2.0 - $name" \
    --dir "$book_dir" --agent chief_editor \
    "初始化项目并生成第1卷卷纲"

# Phase 2.1+: 每章独立 session（支持断点续跑）
for ((ch=1; ch<=chapters; ch++)); do
    if [ -f "$book_dir/versions/$ver/02-正文/第${ch}章-终稿.md" ]; then
        warn "跳过第${ch}章（终稿已存在，断点续跑）"
        continue
    fi
    run_phase "Phase 2.$ch - $name 第${ch}章" \
        --dir "$book_dir" --agent chief_editor \
        "执行第${ch}章生产"
done

# 写完成标记（shell 直接写入，不再由 agent 负责）
python3 -c "..."  # 写入 .phase2_done
```

**`dryrun_all()` Phase 2 块**：同样改造，3 章循环 + 断点续跑。

**断点续跑机制**：若终稿文件已存在则跳过该章。Phase 2 崩溃后重新运行 `bash scripts/run_pipeline.sh dryrun` 即可从断点继续。

#### 文件 2: `workspace/_dryrun/books/甄嬛传/.opencode/agents/chief_editor.md`

**结构重组为双模式**：

```
§一  模式判断（根据指令关键词）
§二  Init 模式：初始化 + 卷纲生成
      2.1 读取核心配置
      2.2 创建目录结构
      2.3 创建/追加自动化处理日志
      2.4 初始化 novel_metadata.json（调用 novel_metadata.py create）
      2.5 生成卷纲（调用 plot_planner）
§三  Chapter 模式：单章生产
      3a 章纲生成 → 确认落盘
      3b 正文初稿
      3c 质检+重写循环（同旧逻辑）
      3d 记录章节名（调用 novel_metadata.py add-chapter）
      3e 更新 progress.json
```

**关键改进**：
- 新增 `§2.4`：使用 `novel_metadata.py create` 初始化文件
- 移除 `§四(卷完成处理)` 和 `§五(完成)`——这些由 shell script 负责
- `§三` 从多章循环改为单章处理

#### 文件 3: `workspace/_dryrun/books/甄嬛传/versions/v2/发布/novel_metadata.json`

用标准格式完全重写，与 `novel_metadata.py` 兼容。当前内容反映已完成 2 章的状态：

```json
{
  "title": ["入宫为替身，本宫要上位", "开局觉醒，掀了这后宫棋局", ...],
  "source": {"title": "后宫·甄嬛传", "author": "", "year": 2020, "public_domain": true},
  "genre": "女频言情",
  "word_count_target": 6000,
  "total_chapters": 3,
  "chapters_completed": 2,
  "chapter_names": ["替身揭露·暗网觉醒", "首次交锋·情报萌芽"],
  "protagonist": "甄嬛",
  "setting": "架空古代后宫",
  ...
}
```

验证：`novel_metadata.py add-chapter` 已测试通过。

## 五、影响评估

| 方面 | 旧架构 | 新架构 |
|------|--------|--------|
| Session 数 | 1 次 opencode run | 1 + N 次 opencode run |
| 单 session 上下文 | 150K+ (第3章时溢出) | < 50K (仅当前章) |
| 断点续跑 | 不支持（全部重来） | 支持（检查终稿跳过） |
| novel_metadata | 自定义格式，脚本不兼容 | 标准格式，`add-chapter` 正常 |
| 章纲落盘 | 仅首章成功 | 每章独立落盘 |
| 上下文溢出风险 | 随章数线性增长 | **不随章数增长** |

---

## 六、后续建议

1. **Phase 2 初始化**：当前 Init session 中卷纲仅生成第 1 卷。多卷时需要在每卷开始时新增 Init 调用
2. **Phase 3 审稿**：当前审稿仍在单 session 中审全部章节。若日后章节数多（>10 章），可能也需要类似的拆分改造
3. **测试覆盖**：建议在 dryrun 后增加一个验证步骤 —— 自动检查章纲、终稿、纪要文件是否完整
