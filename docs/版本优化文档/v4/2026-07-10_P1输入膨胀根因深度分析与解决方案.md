## 一、背景

### 1.1 P1 问题首次暴露（v3 审核报告）

v3 总结性审核报告中，P1（输入膨胀失控）被评为 L6 输入监控 45/100，D 级致命：

| 环节 | Ch2 输入 | Ch5 输入 | 增长率 | 状态 |
|------|:---:|:---:|:---:|------|
| plot_planner | 16KB | 52KB | +217.8% | ❌ 触发归零告警（>200%） |
| content_writer | 49KB | 105KB | +111.9% | ❌ 触发归零告警（>200%） |

### 1.2 v4 修复方案（2026-07-09_v4修改清单 · D 项：滑动窗口 + 伏笔状态摘要）

v4 修改清单 D 项设计了针对 P1 的滑动窗口方案，涉及 4 个文件：

- **chief_editor.md**: 初始化 SOP 改为仅统计最近 2-3 章纪要 + 伏笔摘要；新增步骤 3e5 伏笔状态滚动摘要更新
- **content_writer.md**: 纪要输入明确限制"最近 2 章，不超过 2 章"；新增输入 3b 伏笔状态滚动摘要
- **quality_reviewer.md**: 新增 §4.1 纪要压缩规则（满分维度仅一行、15KB 上限）
- **伏笔状态滚动摘要-template.md**: 新增模板文件，替代"全量历史纪要拼接"

**预期效果**：content_writer 输入 145KB→60KB（-58%），输入膨胀率 +341%→+50% 以内。

### 1.3 v4 修复的实际效果

v4 滑动窗口方案于 **2026-07-09 18:20** 部署到斗破苍穹项目 agent 中。数据采集于 **18:47–19:40** 期间完成。

**结果**：修复已生效，但膨胀仍然显著：

| 环节 | Ch2 输入 | Ch5 输入 | 增长率 | 是否仍触发归零告警 |
|------|:---:|:---:|:---:|:---:|
| plot_planner | 16KB | 52KB | +217.8% | 是 |
| content_writer | 49KB | 105KB | +111.9% | 是 |

**结论**：v4 滑动窗口解决了"全量累积 O(N)"问题，但没有触及输入膨胀的其他主要来源。修复是**必要但不充分的**。

## 二、根因拆解：输入膨胀的真实来源

### 2.1 content_writer 输入构成拆解（Ch5 实测 104,677 bytes）

| 组成部分 | 体积 | 占比 | 增长趋势（Ch1→Ch5） |
|---------|------|:---:|------|
| **第5章章纲** | 70,359 | **67.2%** | 27KB→70KB，**+158%** 🔴 |
| 最近2章纪要 (Ch3+Ch4) | 24,132 | 23.1% | 滑动窗口已限，但单章纪要 Ch1(2KB)→Ch3(15KB) 峰值增长 +600% |
| 前章终稿 (Ch4) | 9,242 | 8.8% | 7-11KB，波动稳定 ✅ |
| 伏笔状态滚动摘要 | 944 | 0.9% | ~1KB，固定表格 ✅ |

### 2.2 plot_planner 输入构成拆解（Ch5 实测 51,651 bytes）

| 组成部分 | 体积 | 占比 | 增长趋势 |
|---------|------|:---:|------|
| 最近3章纪要 (Ch2+Ch3+Ch4) | 35,272 | 68.3% | 纪要单章体积增长为主因 |
| 仿写衍生总纲领 | 15,435 | 29.9% | 稳定（全文常驻）✅ |
| 伏笔状态滚动摘要 | 944 | 1.8% | ~1KB，固定 ✅ |

### 2.3 核心发现：章纲文件自身体积是最大膨胀源

```
Ch1章纲: 27,246 bytes  ────────────────────── (基线)
Ch2章纲: 39,488 bytes  ──────────────────────────── (+45%)
Ch3章纲: 50,318 bytes  ────────────────────────────────── (+27%)  
Ch4章纲: 59,635 bytes  ──────────────────────────────────────── (+19%)
Ch5章纲: 70,359 bytes  ────────────────────────────────────────────── (+18%)
```

**章纲以平均每章 +10KB 的速度持续增长**，占 content_writer 总输入的 67%。

### 2.4 章纲增长段位分析

章纲共 12 个节（§1-§12），其中以下四个节是"持续累积式增长"的主力：

| 节 | 内容 | 增长原因 | 增长模式 |
|----|------|---------|:---:|
| §3 逐段结构 | 五段式每段场景描写、角色动作、关键情节点 | 后期章节场景更复杂、涉及角色更多 | 章节递增 |
| §8 约束遵守确认 | 不可触碰内容检查表（9项）+ 伏笔信息禁止表（2项）+ 金手指限制表（8项）+ 字数约束表 + 高频词变异表 | 约束表内嵌约束原文，逐章重复粘贴 | 恒定（但内容重复冗余） |
| §10 角色信息获取清单 | 主角已知信息表 + 本章新获取信息表 + 无来源信息警告 | 故事推进后主角已知信息累积 | 章节累积 |
| §12 写作要点备忘 | 叙事节奏表 + 角色语言特征 + 环境侧写元素 + 句式模式应用 | 前章写作经验逐章累积 | 章节累积 |

另有 **v4 修改清单 C 项**（场景原创度强化）在 §9 中新增了 §9.2 场景骨架原创度自查和 §9.3 L2 替换模式判定，进一步推动了章纲体积增长。

### 2.5 纪要文件体积增长

```
Ch1纪要:  2,197 bytes  ────────── (基线)
Ch2纪要: 11,140 bytes  ────────────────────────────── (+407%)
Ch3纪要: 15,142 bytes  ─────────────────────────────────────── (峰值，触及 v4 15KB 上限)
Ch4纪要:  8,990 bytes  ────────────────────────── (回落)
Ch5纪要:  6,953 bytes  ──────────────────── (进一步回落)
```

Ch3 纪要恰好触及 v4 修改后的 15KB 上限。v4 §4.1 规则（满分维度仅一行）在 Ch3 时可能未完全生效——Ch3 和 Ch2 的纪要完全相同大小（15,142 vs 11,140），且 -v1 和终版完全一致。Ch4-5 纪要回落表明压缩规则后来可能生效了，但峰值已达上限。

### 2.6 v4 修复做了什么 vs 没做什么

| 维度 | 已修复 | 未修复 |
|------|:---:|------|
| 纪要全量累积 O(N)→O(1) | ✅ 窗口限制为 2-3 章 | — |
| 伏笔跨章上下文 | ✅ 伏笔状态滚动摘要（~1KB 固定） | — |
| 单章纪要体积增长 | ⚠️ 设了 15KB 上限 | ❌ 上限被触及，无缓冲空间 |
| **章纲体积膨胀** | **未涉及** | ❌ +158% 且无上限约束 |
| 常驻参考文件（总纲 15KB + salt 23KB） | **未涉及** | ❌ 始终全量加载 |
| 非纪要类跨章状态追踪 | ✅ 伏笔摘要做到了 | ❌ 角色信息、环境侧写等仍逐章复制 |

## 三、风险分析

### 3.1 当前卷（30 章）保守外推

| 章节 | 章纲体积 (预测) | content_writer 输入 | plot_planner 输入 | 状态 |
|------|:--:|:--:|:--:|------|
| Ch5 (实测) | 70KB | 105KB | 52KB | 🔴 归零告警已触发 |
| Ch10 | ~120KB | ~160KB | ~70KB | 🔴 持续恶化 |
| Ch15 | ~170KB | ~220KB | ~90KB | 🔴 接近模型有效上下文极限 |
| Ch20 | ~220KB | ~280KB | ~110KB | 🚨 可能触发 OOM |
| Ch30 | ~320KB | ~400KB | ~150KB | 🚨 几乎不可运行 |

### 3.2 风险清单

| # | 风险 | 严重程度 | 触发条件 | 后果 |
|:--:|------|:---:|------|------|
| R1 | **Token 成本指数级增长** | 🔴 高 | 当前已在发生 | 单章 content_writer 消耗从 Ch2 的 ~49KB 涨至 Ch30 的 ~400KB（约 8 倍），30 章卷费用额外浪费 4-5 倍 |
| R2 | **模型注意力稀释 → 生成质量断崖** | 🔴 高 | Ch10-15 区间 | 中后段指令被模型遗忘——伏笔遗漏、角色行为不一致、标签约束失效 |
| R3 | **质检回路崩溃** | 🔴 高 | Ch15+ | quality_reviewer 同样受大上下文影响，质检错误率上升 → 低质正文通过 → 下游伏笔断裂 |
| R4 | **管线 OOM 阻断** | 🚨 致命 | Ch15-20 区间 | 超过模型硬性上下文限制时直接运行失败；已投入的前半卷费用沉没；需回滚修复后续跑 |
| R5 | **34 卷规模的乘数效应** | 🚨 致命 | 第 2-3 卷起 | 每卷末章 input 比上卷更高，第 10 卷可能已触及天花板；全书 1200+ 章的经济总成本远超预期 |
| R6 | **生产可持续性归零** | 🚨 致命 | 已预警 | v3 报告判定"D 级致命"，两项归零告警已触发。34 卷项目无法在每章恶化环境下维持到终点 |

### 3.3 特别警告

> v3 100 分的章内容质量是在 Ch1-5 的短上下文下达成的。**当前的高评分不代表 Ch15-30 能维持同等质量**——输入膨胀是一个"前期不明显、中后期断崖式暴露"的问题。

## 四、解决方案矩阵

以下列出所有可行的技术路线。各项独立可选，也可组合使用。按改动成本从低到高排列。

### 方向 A：章纲本体瘦身（针对 67% 输入占比）

| 编号 | 方案 | 核心思路 | 预期效果 | 改动面 |
|:--:|------|---------|:---:|:---:|
| A1 | **章纲分层** | 拆分为"写作指令.md"（含 §1-4,§6-7,§11）+ "审计元数据.md"（含 §5,§8-10,§12）。content_writer 只读指令层 | 70KB→~35KB (-50%) | 修改 plot_planner 输出 + content_writer 输入约束 |
| A2 | **角色信息外部引用** | §2 不再复制角色全量，改为引用 `04-数据/角色状态汇总.md`（全书唯一一份，不逐章复制） | §2 从 ~5KB→~0.5KB | 新建角色状态汇总 + 修改 plot_planner + content_writer |
| A3 | **约束确认改为纯检查清单** | §8 当前逐条粘贴约束原文（高度重复）。改为仅输出 ✔/✘ + 一行简述，不重复粘贴原文 | §8 从 ~12KB→~2KB | 修改 plot_planner 输出格式 |
| A4 | **写作备忘收敛上限** | §12 仅保留"本章特有问题"，设置 ≤15 行硬上限。跨章经验沉淀到独立的"写作经验库.md" | §12 从 ~8KB→~2KB | 修改 plot_planner + 新建经验库 |
| A5 | **章纲体积硬上限** | plot_planner 收到指令"章纲总体积 ≤40KB，超限优先精简 §3 场景描述" | 70KB→≤40KB (-43%) | 修改 plot_planner 指令 |
| A6 | **约束表外置引用** | §8 不是逐条检查，而是引用总纲/盐值对应章节 + 输出一个纯结果矩阵（9 行 × 1 列 = ✔/✘） | §8 从 ~12KB→~0.5KB | 修改 plot_planner |

### 方向 B：纪要格式压缩（针对 23% 输入占比）

| 编号 | 方案 | 核心思路 | 预期效果 | 改动面 |
|:--:|------|---------|:---:|:---:|
| B1 | **纪要体积硬上限下调** | 从 15KB→10KB。quality_reviewer 违反计入扣分 | Ch3 纪要从 15KB→≤10KB | 修改 quality_reviewer 规则 |
| B2 | **纪要分层输出** | quality_reviewer 输出两份：核心版（≤8KB，分数+扣分+伏笔变更）+ 完整版（保留当前格式）。pipeline 传核心版给下游 | 纪要输入 ~24KB→~16KB | 修改 quality_reviewer + chief_editor 传参 |

### 方向 C：架构级重构（根本性方案）

| 编号 | 方案 | 核心思路 | 预期效果 | 改动面 |
|:--:|------|---------|:---:|:---:|
| C1 | **上下文预算制度** | 每个 agent 设总 context 预算（如 content_writer ≤80KB）。chief_editor 调用前计算总输入，超预算自动压缩 | 膨胀率 +112%→趋近 0% | 修改 chief_editor 调用逻辑 |
| C2 | **角色状态母表** | 创建 `04-数据/角色状态汇总.md`，全书唯一。plot_planner 在 §2 只写"本章变更"，不复制全量。content_writer 独立读取母表 | 消除逐章角色信息冗余 | 新建文件 + 修改 3 个 agent |
| C3 | **写作经验知识库** | 创建 `04-数据/写作经验库.md`，沉淀跨章通用经验。章纲 §12 不再累积，改为引用经验库 ID | §12 稳定在 ~1KB | 新建文件 + 修改 2 个 agent |
| C4 | **参考文件索引化（按需加载）** | agent 通过 tool call 按需查询参考文件，而非一次性全量预加载。当前 content_writer 第 41 行已有"不要将全量资料常驻上下文"指令，但第 27 行又明确要求读总纲全量——这两条规则存在矛盾，需统一 | 常驻参考 38KB→~5KB | 修改所有 agent 的输入约束 |
| C5 | **静态上下文预编译** | 将总纲、salt、角色母表中 content_writer 每次必读的稳定字段预编译为 ~5KB 的"写作快速参考卡" | 有效载荷 -80% | 新建预编译步骤 + 修改 content_writer |

### 方向 D：流程层优化（不改 agent 核心逻辑）

| 编号 | 方案 | 核心思路 | 预期效果 | 改动面 |
|:--:|------|---------|:---:|:---:|
| D1 | **章纲自动瘦身步骤** | 章纲生成后，chief_editor 运行瘦身脚本提取必需字段为精简版。content_writer 读取精简版 | 70KB→~30KB | 编写瘦身脚本 + 修改 chief_editor 流程 |
| D2 | **上下文渐进式加载** | content_writer 分两阶段：①读 §1-4,§7 写场景框架；②读 §5-6,§8-12 补充伏笔/约束/钩子 | 单次峰值 -40% | 修改 content_writer 流程 |

## 五、组合路线建议

### 短期（最小改动，1 次修改即可生效）

```
A3(约束确认纯清单) + A4(备忘收敛上限) + B1(纪要上限 15→10KB)
```

- **改动面**：3 个 agent 各加 1 句约束，不新建文件
- **预期效果**：综合输入 **-25~30%**
- **风险**：低

### 中期（真正解决章纲膨胀的转折点）

```
A1(章纲分层) + C2(角色状态母表)
```

- **改动面**：新建 1 个母表文件 + 修改 3 个 agent
- **预期效果**：content_writer 输入稳定在 **50-60KB**，不再随章节递增
- **风险**：需协调多个 agent 的文件读写逻辑

### 长期（一劳永逸，适合 v5+）

```
C1(上下文预算) + C4(参考文件索引化)
```

- **改动面**：重构 chief_editor 上下文管理逻辑 + 所有下游 agent 输入约束
- **预期效果**：输入膨胀率 **趋近 0%**，支持 34 卷规模
- **风险**：改动面大，需要充分测试

## 六、不可触碰的底线

无论选择哪个方案，**不能降低章纲的信息可执行率**。

v3 流程审计报告中 D2 维度满分（7/7）是当前管线质量的基石——章纲的每一条指令都是"可操作、可验证、可溯源"的。任何瘦身方案如果导致：

- content_writer 理解偏差
- 伏笔遗漏
- 约束丢失
- 角色行为不一致

那么省下的 token 成本会被后续质检-重写循环的额外消耗反超，得不偿失。
