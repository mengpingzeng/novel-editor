# forbidden_patterns 消费点上移方案 — phase 强度自适应检查

> 生成日期：2026-07-10
> 依据：v3_old 审核报告 P4（卷纲 forbidden_patterns 完全缺失）根因追溯
> 核心决策：将 forbidden_patterns 消费点从 plot_planner 上移到 destiny_designer 步骤 4/8

---

## 一、根因回顾

改动 A（2026-07-09 v4 修改清单）修改了 `plot_planner §1.2` 的 forbidden_patterns 输出模板，从空白占位符升级为强制性四列表格。但修复无效。

根因不是模板写得不够强制，而是**数据链路三层断裂**：

| 层 | 位置 | 问题 |
|:---:|------|------|
| 冻结层 | `destiny_designer §7` | "禁止模式"指令为"从总纲领和盐值继承"——是意图描述，非强制提取 |
| 传递层 | `chief_editor:176` | 传入 plot_planner 的字段清单显式遗漏了"禁止模式" |
| 消费层 | `plot_planner §0.2` | 确定性细化模式下 agent 以注入包为唯一数据源，不自主读 salt |

本方案将消费点前移到 **destiny_designer 步骤 4/8**——事件链生成后、注入包打包前，在此处做 forbidden_patterns 的结构化合规检查。

---

## 二、为什么不修传递链，要移消费点

forbidden_patterns 处理的是"什么事件不该发生 / 什么事件间关系不该出现"——这是**事件链设计**层面的约束。destiny_designer 是唯一同时持有以下所有信息的 agent：

- 白皮书（原著结构）
- 盐值（赛道约束 + forbidden_patterns）
- 总纲（全局叙事规则）
- 全量逐章事件链（纵向视野，可做跨章模式匹配）

plot_planner 的职责是"在给定事件骨架上设计实现方式（HOW）"，让它检查事件链层面的违规是职责错位。正确的分工是：

| agent | 检查什么 | 依据 |
|-------|---------|------|
| destiny_designer | 事件链本身是否违反禁止模式（WHAT 层） | salt forbidden_patterns |
| plot_planner | 事件实现方式是否弱化/绕过了约束（HOW 层） | 注入包 §7.5 的 J 项判定结果 |

---

## 三、方案设计

### 核心思路：选项 3——按 phase intensity 做自适应检查

一刀切的全局阈值（"连续 N 章"）无论卷位差异是不合理的。首卷前 5 章讲世界观展开是天经地义的，末卷连续 3 章无突破就是节奏崩塌。

本方案不修改 salt 数据格式，不硬编码卷位系数。改为在检查时按 phase 的 `intensity`（低/中/高/极高）和 `core_task` 语义动态判断：**这条 forbidden_pattern 在当前 phase 内是否适用。**

### 3.1 步骤 4：事件链表加"事件类型"列

现有输出表：

```
| 章号 | 核心事件 | 出场角色 | 爽点类型 | 伏笔埋设 | 伏笔回收 |
```

新增一列：

```
| 章号 | 核心事件 | 事件类型 | 出场角色 | 爽点类型 | 伏笔埋设 | 伏笔回收 |
```

事件类型枚举：

| 值 | 语义 | J 项检查用途 |
|----|------|-------------|
| 突破 | 境界/实力/技能晋级 | 匹配"连续 N 章无突破" |
| 战力展示 | 战斗/碾压/镇压/显威 | 匹配"突破后不展示战力" |
| 冲突升级 | 新威胁/挑战/背叛/对抗加剧 | 匹配"铺垫无核心冲突" |
| 情感转折 | 关系变化/心态蜕变 | 辅助判断 |
| 信息释放 | 秘密揭露/世界真相/系统通知 | 辅助判断 |
| 铺垫过渡 | 世界观展开/旅行/恢复/日常 | J 项不对此类做突破/战斗检查 |
| 伏笔埋设 | 以埋伏笔为主的事件 | — |
| 势力变更 | 同盟/叛变/吞并/建立 | 辅助判断 |

生成时机：agent 在步骤 4 生成每行事件时同步标注——它刚设计了事件内容，分类成本极低。

### 3.2 步骤 8：新增 J 项「禁止模式合规检查」

#### J1：逐 phase 确定每条规则适用范围

对每条 forbidden_pattern，生成适用性矩阵：

```
| Phase        | intensity | core_task                | P1(无突破) | P3(无爽点) | P4(战后碾压) |
|-------------|-----------|--------------------------|:---:|:---:|:---:|
| P0 觉醒期    | 中高       | 世界观展开+首个金手指亮相    | ✅ | ✅ | —  |
| P1 突破链期  | 高         | 连续小突破+首次打脸+秘境首探  | ✅ | ✅ | ✅  |
| P2 过渡铺垫  | 中         | 势力关系铺垫+情报收集        | ❌ | ✅ | —  |
| P3 大突破收官 | 极高       | 大境界跨越+势力格局重塑       | ✅ | ✅ | ✅  |
```

适用性规则：

- **"连续 N 章无突破"** → 仅 `intensity ≥ 中高` 且 `core_task` 含"突破/升级/变强"语义的 phase 适用。纯铺垫/过渡/伏笔 phase 跳过
- **"连续 N 章无爽点"** → 所有 phase 适用（全局底线）
- **"突破后不展示战力碾压"** → 仅在包含 事件类型=突破 的章及其后续 2 章适用
- **"突破轻描淡写"** → 仅在包含 事件类型=突破 的 phase 适用

**兜底逻辑**（agent 不确定时）：

```
| Phase | intensity | Pattern 适用 | 置信度 | 实际行为                     |
|-------|-----------|:---:|:---:|----------------------------|
| P2    | 中         | ⚠️ 不确定   | —   | 检查但判定上限为 WARN，不 FAIL |
```

若 agent 无法确定 phase 是否应适用规则 → 仍然执行检查，但判定上限为 WARN。

#### J2：对适用 phase 执行逐条检查

示例（玄幻赛道 forbidden_patterns）：

```
Pattern 1: "连续 5 章无境界/实力突破"
  适用 phase: P0, P1, P3（P2 跳过）
  扫描方式: 事件类型列 = 突破，计算间隔
  结果:     P0 Ch01=突破 ✅
            P1 Ch06=突破, Ch11=突破, 间隔 5 章 → ⚠️ WARN（接近阈值）
            P3 首章即大突破 ✅

Pattern 2: "突破场景轻描淡写不制造爽感"
  适用 phase: 包含 事件类型=突破 的 phase
  扫描方式: 突破行的爽点类型列是否非空
  结果:     所有突破行均标注爽点类型 ✅ PASS
           注: 本条的执行质量（是否真的不轻描淡写）需下游关注

Pattern 3: "前期铺垫超过 3 章无爽点"
  适用 phase: 所有 phase
  扫描方式: 爽点类型列空窗 ≥ 4 章
  结果:     最大空窗 Ch13-Ch14 (2章) ✅ PASS

Pattern 4: "境界突破后不展示战力碾压"
  适用章号: 突破事件所在章 + 后续 2 章
  扫描方式: 事件类型列 = 战力展示
  结果:     Ch01 突破 → Ch02 战力展示 ✅
            Ch06 突破 → Ch07 信息释放, Ch08 铺垫过渡 → ❌ FAIL
```

#### J3：判定汇总

复用 step 8 H 项已有判定框架（PASS / WARN / FAIL）：

| Pattern | 判定 | 说明 |
|---------|:---:|------|
| Pattern 1 | ⚠️ WARN | P1突破期 Ch06→Ch11 间隔 5 章，接近边界 |
| Pattern 2 | ✅ PASS | 所有突破事件均标注爽点类型 |
| Pattern 3 | ✅ PASS | 最大爽点空窗 2 章 |
| Pattern 4 | ❌ FAIL | Ch06 突破后 2 章无战力展示 |

最终判定：

| 有 FAIL | 有 WARN 无 FAIL | 全部 PASS |
|:---:|:---:|:---:|
| ❌ FAIL — 禁止输出，列出 FAIL pattern 和章号，要求修改事件链后重新自检 | ⚠️ WARN — 不影响打包，J 项 WARN 标记写入注入包 §7.5 | ✅ 正常放行 |

### 3.3 步骤 7 §7.5：注入包的禁止模式章节角色转变

**改动前**：`从总纲领和盐值继承：…禁止模式`——意图描述，非结构化数据。

**改动后**，§7.5 包含两部分：

#### a) J 项判定摘要（由步骤 8-J 自动生成）

```
## §7.5 禁止模式合规检查（来源：步骤 8-J）
### J 项判定结果

| 禁止模式 | J项判定 | 涉及章号 | 说明 |
|------|:---:|------|------|
| 连续 5 章无突破 | ⚠️ WARN | Ch06→Ch11 | 接近阈值，细化时确保 Ch06-10 有足够成长感知 |
| 突破场景轻描淡写 | ✅ PASS | — | 事件链已达标，执行时保证场面感 |
| 连续 3 章无爽点 | ✅ PASS | — | — |
| 突破后无战力碾压 | ❌ FAIL → 已阻断重生成 | — | 不应出现在打包后的注入包中 |

### 下游约束原文（来源：project_salt.json volume_rhythm_profile.forbidden_patterns）
- 连续 5 章无境界/实力突破
- 突破场景轻描淡写不制造爽感（突破必须有场面）
- 前期铺垫超过 3 章无爽点
- 境界突破后不展示战力碾压（突破即打脸）
```

两段并存的原因：J 项 PASS/WARN 只覆盖**事件链层面**的合规。plot_planner 仍需原始约束文本做**场景执行层面**的细粒度保证——destiny 保证"事件链结构不违规"，plot_planner 保证"执行质量不弱化"。

## 六、已知边界与后续

1. **J 项不能覆盖执行质量**：Pattern 2（"突破不轻描淡写"）的 PASS 只保证事件链层有突破标记 + 爽点关联，不保证 content_writer 的实际描写质量。此责任仍在下游
2. **J 项依赖 agent 做 phase 适用性判断**：兜底逻辑（不确定则降级为 WARN）覆盖了误判风险，但不能消除
3. **J 项判定与 H 项无冲突**：H 项检查的是差异化（是否雷同原著），J 项检查的是合规性（是否违反赛道规则），作用维度不同
4. **Phase 重建模式下**：J 项仅检查本 phase（同 H 项 Phase 重建规则），边界外不检查→不产生误报
