# 斗破苍穹流水线超时问题分析

## 背景

在 tmux `step` 会话中执行 `./scripts/run_pipeline.sh step`，斗破苍穹写至第5章时 content_writer agent 超时退出。

---

## 一、为什么会超时

### 1.1 触发链

```
chief_editor.md:78 → cat 总纲领 + 03-纪要/*.md (ALL)
         ↓
  每章递增 ~44KB（纪要×2份/章 × ~22KB/份）
         ↓
  第5章 content_writer 输入已达 245,922 bytes
         ↓
  LLM 推理耗时突破 CHAPTER_TIMEOUT=1800s → timeout 强杀 (exit code 124)
```

### 1.2 时间线

| 时间 | 事件 |
|------|------|
| 12:12:37 | 第5章章纲生成完成，启动 content_writer |
| 12:42:08 | 超时退出（约29分31秒，触达 1800s 阈值） |

### 1.4 根本原因（四个独立病灶）

#### 病灶A：编排者与执行者的输入约束不匹配（最核心）

`content_writer.md:21` 声明：

> 仅加载本章章纲、出场人设、**近两章剧情纪要**、总纲领字数标准页和分类标识页

但 `chief_editor.md:78` 实际传递：

```bash
total=$(cat "versions/{version}/仿写衍生总纲领.md" "versions/{version}/03-纪要/"*.md 2>/dev/null | wc -c)
```

`*.md` 通配符把所有纪要文件全部喂入，两个 agent 定义之间存在直接矛盾。chief_editor 作为编排者没有遵守子 agent 的输入约束。

#### 病灶B：纪要 v-variant 文件重复（double-counting）

当一章第一轮质检即通过（第1、3、4章），`03-纪要/` 下同时保留两份完全相同的纪要（如 `第1章纪要.md` + `第1章纪要-v1.md`），`*.md` 通配符将两份都计入，每章实际贡献 **2倍** 上下文开销。

#### 病灶C：缺少卷级压缩机制

没有卷级摘要。目标共 1500 章，按当前趋势仅纪要累积到第 30 章就会达到约 1.4MB 上下文。plot_planner 可能需要全量历史做规划，但没有层次化压缩策略，到中后期必然超出 LLM 上下文窗口。

#### 病灶D：章纲自身携带累积状态

每章章纲追加伏笔追踪表、角色状态表、信息获取清单等全局状态，导致章纲随章节线性增长（20KB → 58KB，5 章涨 2.8 倍）。

## 二、为什么超时重启后还要等 1800s

### 2.1 代码位置

`scripts/run_pipeline.sh:531-534`：

```bash
elif [ $exit_code -eq 124 ]; then
    if [ $current_attempt -lt $PIPELINE_MAX_ATTEMPTS ]; then
        warn "流水线超时，${CHAPTER_TIMEOUT}s 后自动重启..."
        sleep "$CHAPTER_TIMEOUT"
```

### 2.2 设计问题清单

| # | 问题 | 说明 |
|---|---|---|
| 1 | **变量语义混淆** | `CHAPTER_TIMEOUT` 同时表示"单章最长执行时间"和"超时后冷却等待时间"。前者是上限阈值，后者是退避策略——两个完全不同的概念共用一个变量，无法独立调节。 |
| 2 | **冷却过长（1800s = 30分钟）** | 任务因执行慢等了 30 分钟被强杀，现在再让用户额外等 30 分钟才重试。总延迟 = 2 × 超时 = 1 小时。正常退避 60~300s 即可覆盖 transient 错误窗口。 |
| 3 | **无递增退避** | 3 次重试全部等 1800s。合理的做法：第1次 60s → 第2次 300s → 第3次 900s。对 transient 问题（API 抖动），短等就够了；对 persistent 问题（上下文膨胀），等多久都没用。 |
| 4 | **不区分超时原因** | API 拥塞（等一等可能恢复）vs 上下文过大（等多久都不会变小）共用同一策略。前者等 1800s 浪费，后者等再久也无效。 |
| 5 | **无独立变量** | 没有 `RESTART_COOLDOWN` 环境变量。用户想调短冷却就不得不改短 `CHAPTER_TIMEOUT`，反而导致更多任务超时。 |
| 6 | **静默阻塞** | `sleep 1800` 期间 tmux pane 完全静默，用户无法判断脚本是否存活、剩余等待时间，也无法提前触发重试。 |
| 7 | **无中断机制** | 如果用户在冷却期间手动修复了问题（如清理纪要文件），无法提前跳过冷却，只能杀进程重启。 |

### 2.3 与问题1的交叉影响

对于**上下文膨胀引起的超时**，等待 1800s 完全无效——上下文不会自我缩小。脚本重试后会喂入完全相同的膨胀输入，再次 30 分钟超时，再等 1800s……三轮耗尽后（耗时 = 3 × 30min 执行 + 2 × 30min 等待 = 2.5h）彻底退出，全部白费。
