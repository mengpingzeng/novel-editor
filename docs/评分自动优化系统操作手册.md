# 自动化评分系统操作手册

> 版本：V1 | 日期：2026-07-02 | 平台：OpenCode

---

## 零、What？这是什么系统

这系统帮你做两件事：

1. **自动写书**：给定一本原著小说，AI 自动完成「拆解原著 → 赛道映射 → 生成书名简介 → 逐章写正文 → 质检合格 → 输出完整小说」
2. **严格打分**：对写出来的小说，AI 用编辑视角严格打分（60 分及格但不放水），告诉你哪里写得不好

写书的不认识做评分的，评分的也不关心这书谁写的——两个系统完全独立。

## 一、准备工作（第一次用）

### 1.1 检查原著文件

确认你要用的原著小说 txt 已经放在正确的位置：

```
workspace/repo/{原著名}/source.txt
```

例如：
```
workspace/repo/凡人修仙传/source.txt
workspace/repo/斗破苍穹/source.txt
workspace/repo/娘娘本纪/source.txt
```

> 如果你还没有白皮书（base_whitepaper.md），系统会在第一次写书时自动生成。已有则跳过。

### 1.2 配置迭代范围

编辑 `workspace/iteration-state.json`：

```json
{
  "target_chapters": 20,          // ← 每本书写几章，你可以改成 10 或 50
  "active_books": [               // ← 要写哪些书，减一本就少写一本
    {"name": "凡人修仙传", "platform": "番茄小说", "track": "仙侠"},
    {"name": "斗破苍穹", "platform": "番茄小说", "track": "玄幻"}
  ]
}
```

### 1.3 配置及格线

同样在 state 文件中：

```json
{
  "passing_score": 60             // ← 及格线，你可以调整
}
```

## 二、快速自测（先跑 dryrun！）

**第一次使用前必须跑 dryrun 验证系统正常。** 不要跳过。

### 命令

```
/iterate dryrun
```

这会在工作空间下创建一个独立的 `workspace/_dryrun/` 目录，用「娘娘本纪」（体量最小）写 3 章，然后审稿打分。

### 验收

dryrun 完成后会输出验收清单：

```
══════════════════════════════
  dryrun 验收清单
══════════════════════════════
[ ] 白皮书/盐值/总纲领正确生成
[ ] 3章正文+章纲+质检全部产出
[ ] execution-log.md 正确记录了每步的 Agent+模型+状态
[ ] signing_reviewer 给出了 9 维打分
[ ] 模型使用正确：框架层用 V4 Pro / 执行层用 V4 Flash / 审稿用 GLM 5.2
[ ] 正式目录 workspace/books/ 未被污染
```

全部打勾 → 系统正常，可以正式用。

### 进阶 dryrun

```
/iterate dryrun 2 5    # 2 本书 × 5 章
```

## 三、写作 + 审稿（正式使用）

### 3.1 命令

```
/iterate step
```

### 3.2 执行过程

系统会自动分三个阶段串行执行：

**阶段一：写书**

逐本调用写作流水线。你会在终端看到：

```
[{时间}] [automation_manager → book_factory] 模型: team-deepseek/deepseek-v4-flash | 凡人修仙传 v1 | 进行中
...
[{时间}] [automation_manager → book_factory] 模型: team-deepseek/deepseek-v4-flash | 凡人修仙传 v1 | ✅
```

第一本写完后自动开始第二本。

**阶段二：审稿**

全部写完后进入审稿。同样逐本进行：

```
[{时间}] [automation_manager → signing_reviewer] 模型: tokenhub/glm-5.2 | 凡人修仙传 审稿 | 进行中
...
[{时间}] [automation_manager → signing_reviewer] 模型: tokenhub/glm-5.2 | 凡人修仙传 审稿 | ✅  52分 未通过
```

**阶段三：汇总**

```
═════════════════════════════
  v1 轮审稿汇总
═════════════════════════════
| 书名        | 得分 | 判定    |
|------------|-----|--------|
| 凡人修仙传   | 52  | 未通过  |
| 斗破苍穹    | 73  | 通过    |
═════════════════════════════
通过率：1/2   平均分：62.5
```

**然后系统停止，不会自动做任何事。**

## 四、查看结果

### 4.1 看日志

每本书的执行日志在这里：

```
workspace/books/{原著名}/versions/{版本号}/execution-log.md
```

日志记录每一步用了哪个 Agent、哪个模型、什么状态：

```
# 执行日志 - v1
| 时间 | 步骤 | Agent | 模型 | 状态 |
|------|------|-------|------|------|
| 10:00:00 | 流水线启动 | book_factory | team-deepseek/deepseek-v4-flash | ✅ |
| 10:00:05 | 原作拆解 | original_analyst | team-deepseek/deepseek-v4-pro | ✅ |
...
```

### 4.2 看正文

```
workspace/books/{原著名}/versions/{版本号}/02-正文/
```

### 4.3 看审稿报告

```
workspace/books/{原著名}/versions/{版本号}/审稿报告-{版本号}.md
```

报告包含 9 个维度的详细打分 + 扣分定位 + 一票否决检查。

### 4.4 看状态

```
/iterate status
```

## 五、看完报告后的操作

### 场景 A：所有书通过了

你可以：
- 手动把 `target_chapters` 改成 50，重新 run step，写到 50 章
- 或者提取满意的书作为产出

### 场景 B：部分书没通过

审稿报告里写明了哪个维度扣分多、哪个角色负责。比如：

```
维度 2 追读欲：连续3章钩子同质化 → plot_planner 的问题（钩子类型未做轮换约束）
```

这时候你不需要自动优化，而是手动：
1. 告诉我「plot_planner 的钩子类型需要轮换」
2. 我来帮你改 agent 定义
3. 改完后把 state 文件的 phase 改为 `"writing"`，version 递增为 `"v2"`
4. 重新 `/iterate step`

**这样只针对你说的具体问题做修改，不会乱改其他东西。**

### 场景 C：想写新版

```json
// 改 iteration-state.json
{ "phase": "writing", "current_round": 2 }
// 以及每本书：
{ "凡人修仙传": { "version": "v2", "status": "pending", "score": null, "passed": false } }
```

旧版 v1 目录不会被删除。

## 六、单独使用（不跑自动化）

### 6.1 只写一本书

直接在 opencode 对话中输入：

```
@book_factory source_name=凡人修仙传 target_platform=番茄小说 style_track=仙侠 chapter_count=20 version=v1
```

不经过 automation_manager，book_factory 独立执行完整流水线。

### 6.2 只审一本书

写完后，单独打分：

```
@signing_reviewer book_version_path=workspace/books/凡人修仙传/versions/v1/ source_whitepaper_path=workspace/repo/凡人修仙传/base_whitepaper.md
```

signing_reviewer 不关心这本书是不是 book_factory 写的——任何已写完的书都可以拿来审。

## 八、常见问题

### Q: 怎么知道模型有没有用错？

看 `execution-log.md`，每一步都标了模型名。正规的模型分配应该是：

| 步骤 | 应该用的模型 |
|------|------------|
| 原作拆解、赛道映射、门面生成、盐值校验 | V4 Pro |
| 章纲、正文、质检、合规 | V4 Flash |
| 签约审稿 | GLM 5.2 |

有任何一步模型不对 → 模型切换出了问题，检查 agent 的 frontmatter。

### Q: 写一本 20 章小说要多久？

取决于模型响应速度。每章大约 2-5 分钟，20 章 + 前置准备约 1-2 小时。5 本书 × 20 章大约需要 5-10 小时。

### Q: 不想跑全部 5 本书怎么办？

改 `iteration-state.json` 的 `active_books`，只保留你想写的。

### Q: 多久算一"轮"？

从 phase=writing 开始到 phase=done（审完汇总）结束，是一轮 step。一轮结束后必须你手动操作才会开始下一轮。

## 九、完整命令速查

| 命令 | 作用 |
|------|------|
| `/iterate dryrun` | 快速自测（1本×3章，隔离目录） |
| `/iterate dryrun 2 5` | 快速自测（2本×5章） |
| `/iterate step` | 正式跑一轮（根据 state 配置） |
| `/iterate status` | 查看当前进度和分数 |
| `@book_factory source_name=... target_platform=... chapter_count=... version=...` | 单本写作 |
| `@signing_reviewer book_version_path=... source_whitepaper_path=...` | 单本审稿 |
