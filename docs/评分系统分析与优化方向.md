# 评分系统问题分析

> 基于白皮书→总纲→卷纲→章纲→正文→质检全链路，逐层排查评分系统中的实际问题。

---

## 一、当前评分架构总览

### 1.1 评分层级

| 阶段 | 层级 | 评分者 | 模型 | 满分 | 综合权重 |
|------|------|--------|------|------|---------|
| Phase 2 | 章内容质检 | quality_reviewer | V4 Flash | 100 | 20% |
| Phase 3 | 白皮书质量 | whitepaper_reviewer | GLM 5.2 | 100 | 10% |
| Phase 3 | 总纲领质量 | master_outline_reviewer | GLM 5.2 | 100 | 15% |
| Phase 3 | 卷纲质量 | volume_outline_reviewer | GLM 5.2 | 100 | 10% |
| Phase 3 | 章纲质量 | chapter_outline_reviewer | GLM 5.2 | 100 | 10% |
| Phase 3 | 输入监控 | input_monitor | V4 Flash | 100 | 5% |
| Phase 3 | 签约审稿 | signing_reviewer | GLM 5.2 | 100 | 20% |
| Phase 3 | 流程审计 | pipeline_auditor | GLM 5.2 | 100 | 10% |

综合加权：`白皮书×10% + 总纲×15% + 卷纲×10% + 章纲×10% + 章内容×20% + 输入监控×5% + 签约审稿×20% + 流程审计×10%`

### 1.2 三层流水线

```
Phase 1: 框架层（白皮书 → 赛道映射 → 门面 → 盐值 → 总纲领）
    ↓
Phase 2: 内容层（卷纲 → 章纲 → 正文初稿 → 质检重写循环 → 终稿）
    ↓
Phase 3: 评价层（全层级评分 → 签约审稿 → 流程审计 → 跨书总结）
```

### 1.3 设计意图确认

系统有两类正确且互补的目标：

| 类别 | 目标 | 对应评分维度 | 性质 |
|------|------|-------------|------|
| **仿写** | 对原著**结构设计**的提取和还原——爽点出现的频率/位置、钩子出现的规律、节奏曲线、文风形式特征 | D2/D2-B/D3/D4/D5/D6/D7 | 形式约束。这些是结构骨架和风格设定，不涉及具体内容。和原创性不矛盾。 |
| **创意写** | 在结构框架内，**自由创造**具体的事件、对话、爽点内容、钩子内容 | D9 / 签约审稿D9 | 内容检查。确保新作的具体情节、对白、场景不与原著雷同。 |

这两类目标不是矛盾的，是互补的。本章分析中不采用"鼓励 vs 惩罚"的对立框架。

## 二、核心设计：信息流的正确路径

### 2.1 白皮书应该产出什么

`original_analyst` 的核心方法论是"**拆法不拆事**"，为每种爽点类型独立提取专属触发公式。例如：

```
白皮书 §四 爽点触发公式与分类体系：
  ├── 身份打脸 → 公式A（5步：反派质疑→主角沉默→第三方揭露→反派震惊→主角回应）
  ├── 信息差碾压 → 公式B（不同步骤）
  ├── 情感守护 → 公式C（不同步骤）
  ├── 逆境逆袭 → 公式D（不同步骤）
  └── ...（每种类型独立公式）
```

类型不同，高潮结构不同，触发公式不同——这是正确的设计。

### 2.2 信息流应该怎么走

`workflow_optimizer` 已定义了正确的数据流：

```
爽点公式：original_analyst → 总纲 → plot_planner / content_writer / quality_reviewer
```

总纲领继承白皮书的爽点体系（含类型专属公式），然后同时到达：
- **content_writer**：知道每种爽点类型的正确结构，按类型写作
- **quality_reviewer**：用对应类型的专属公式验证本章的爽点落地

这条路径的设计意图是正确的。问题出在执行层。

## 三、核心问题一：信息断裂 —— "四步闭环"的溯源

### 3.1 问题现象

`quality_reviewer` 的 D3（权重18分），第157行：

```
| 爽点触发是否完整（四步闭环） | 不完整 -5 |
```

**文件中没有任何地方定义"四步闭环"是哪四步。** V4 Flash 只能凭训练数据猜测，或忽略此检查。

### 3.2 全代码库溯源

| 文件 | 行号 | 内容 | 关键发现 |
|------|------|------|---------|
| `quality_reviewer.md:157` | D3检查项 | `爽点触发是否完整（四步闭环）` | **无定义**。只有四个字，没有步骤说明。 |
| `quality_reviewer_bk.md:59` | D3检查项 | `① 对方自信→② 主角点破→③ 验证→④ 配角震惊` | **有定义**。但此公式 ≠ 白皮书中任何一个类型的专属公式。 |
| `content_writer.md:235` | 标签"信息差碾压" | `□ 四步完整闭环：对方自信→主角点破→验证→配角震惊` | **仅一个标签有公式**。其他标签（扮猪吃虎/赘婿逆袭/商战重生/隐藏大佬）没有对应公式。 |
| `original_analyst.md:99` | "身份打脸"公式 | `{反派公开质疑}→{主角沉默}→{第三方揭露}→{反派震惊+围观群众哗然}→{主角淡然回应}` | **五步公式，专属于"身份打脸"类型**。不同时等于backup版的四步，也不同时等于content_writer版的四步。 |

### 3.3 "四步闭环"的变异链

```
白皮书 (original_analyst)
  │  为"身份打脸"生成5步公式：
  │    反派质疑→主角沉默→第三方揭露→反派震惊→主角淡然回应
  │  每种爽点类型各有独立公式。
  │
  ▼  （某次迭代中，公式被简化+通用化）
  │
backup版 quality_reviewer
  │  简化为4步：
  │    对方自信→主角点破→验证→配角震惊
  │  合并了"主角沉默"和"主角回应"，丢失了"第三方揭露"这个关键步骤。
  │  且被静默通用化——从"身份打脸专属"变成"所有爽点类型通用"。
  │
  ▼  （到当前版本，连4步定义也丢失了）
  │
当前 quality_reviewer
  │  只剩四个字："四步闭环"
  │  无定义、无步骤、无数据源指向。
  │  V4 Flash 执行时只能猜。
  │
  ▼  （同时，一条分支在 content_writer 中硬编码）
  │
content_writer
  │  仅在"信息差碾压"标签下写了这个公式。
  │  其他标签缺失。公式来源不明。
```

### 3.4 断裂的后果

1. **白皮书做的工作被浪费了**。它正确提取了类型专属公式，但 quality_reviewer 没有动态读取它们。workflow_optimizer 定义了正确的数据流（爽点公式 → quality_reviewer），但实际 agent 实现中没有执行这个读取动作。

2. **不同类型被同一公式验收**。一个"情感守护"类爽点自然不需要"配角震惊"，但如果 V4 Flash 猜的"四步闭环"包含这一步骤，它就会被误扣分。

3. **content_writer 收到的指令不一致**。总纲中有完整的爽点体系（含各类型专属公式），但 content_writer §七 的标签落地清单却只给一个标签硬编码了一个简化版公式，与总纲中的完整体系脱节。

4. **V4 Flash 执行时无据可依**。当前 quality_reviewer 的 D3 检查项对执行模型来说是不完整的指令——它有一个概念名称但没有任何可操作的检查步骤。

### 3.5 这不是评分方向的问题，是信息传递的工程问题

这不是"评分标准应该鼓励什么"的哲学问题。正确的评分标准（按类型专属公式验证）在白皮书中已经存在，quality_reviewer 的设计意图（读取总纲中的爽点体系来验证）在 workflow_optimizer 中已经定义。问题在于 agent 实现层面没有把这个信息流接上。

## 四、核心问题二：反AI规则集的自损循环

### 4.1 问题描述

content_writer §9（约200行）的核心逻辑是：

```
发现问题表达 → 禁止该表达 → 提供替换词 → 替换词用多了被发现 → 加大替换池 → 替换池轮换规律被发现 → 加更多规则...
```

系统自认的案例（来自总结报告）：

- 清除"不是X——是Y" → 引入"沉默了X"（21次/20章）
- 清除"瞳孔一缩" → 引入"目光凝住"（6-7次/20章）
- 清除"全场鸦雀无声" → 引入"矿石嗡鸣+矿灯压低"（15+次/20章）

结论：**每修一tic引入一新tic。** 系统性模式，不是个案。

### 4.2 根本原因

轮换池本身是程序化思维。真人作者不需要一个预定义的词库来避免重复——他们的表达变化是混沌的、偏好驱动的前后不一致，不是按池子轮换的。用轮换池解决AI痕迹，本质是用一种程序化机制替代另一种程序化机制。

### 4.3 规则层叠

| 版本阶段 | content_writer 行数 | 新增的反AI规则 |
|----------|-------------------|---------------|
| 初期 | ~200 | 基础句式禁令 |
| v2 | ~300 | 轮换池、Grep扫描 |
| v3 | ~400 | 替换词轮换、比喻上限、金手指出发变体 |
| v4 | ~450 | 现代词转化、环境侧写轮换 |
| v5 | ~480 | 五个批次检查、配角弧线、反公堂模式同质化 |

每轮审稿发现新问题 → 加新规则 → 新规则产生新固化的替换模式 → 下轮审稿发现新问题。

### 4.4 与核心问题一的关联

这两个问题是独立的，但会互相放大：
- 信息断裂导致 quality_reviewer 评分不准确，可能误报问题
- content_writer 被误报的问题驱动叠加更多反AI规则
- 更多反AI规则压缩创意空间，让 AI 写得更像 AI
- quality_reviewer 发现更多"AI痕迹"→ 继续加规则...

## 五、核心问题三：权重分配

### 5.1 当前权重

| 评分层 | 权重 | 面向谁 |
|--------|------|--------|
| 白皮书 | 10% | AI写给AI看的中间产物 |
| 总纲领 | 15% | AI写给AI看的中间产物 |
| 卷纲 | 10% | AI写给AI看的中间产物 |
| 章纲 | 10% | AI写给AI看的中间产物 |
| **规划文档合计** | **45%** | |
| 章内容 | 20% | **读者唯一接触到的产出** |
| 输入监控 | 5% | 流程技术指标 |
| 签约审稿 | 20% | 整书综合评估 |
| 流程审计 | 10% | 流程优化参考 |

### 5.2 问题

实际运行数据（来自总结报告）：

| 层级 | 得分 | 权重 | 加权贡献 |
|------|------|------|---------|
| 白皮书 | 100 | 10% | 10.0 |
| 总纲 | 88 | 15% | 13.2 |
| 卷纲 | 97 | 10% | 9.7 |
| 章纲 | 96.8 | 10% | 9.68 |
| **规划层合计** | **均分~95.5** | **45%** | **42.58** |
| 章内容 | 79.4 | 20% | 15.88 |
| 签约审稿 | 90 | 20% | 18.0 |
| 流程审计 | 89 | 10% | 8.9 |
| **综合加权** | — | — | **85.36** |

规划层用 45% 的权重（得分~95.5），把 79.4 分的正文拉到了 85.36 分的综合评价。**但读者只会看到正文。**

满分白皮书 + 近乎满分的卷纲/章纲 ≠ 读者阅读体验好。评分体系中最重的不是读者体验，是 AI 写给 AI 看的中间文件。

## 六、quality_reviewer 各维度具体问题

### 6.1 D3 爽点落地（18分）— 信息断裂

**问题**：如前所述，"四步闭环"未定义，白皮书中的类型专属公式未传递到评分层。

### 6.2 D4-B 角色语言指纹（3分）

**当前做法**：要求角色对话匹配从白皮书提取的5维指纹（句长签名/句式签名/词汇签名/转折签名/功能签名）。

**问题**：白皮书提取的是**原著角色**的说话方式。要求新书的角色模仿原著角色的说话方式，这条指令本身在鼓励内容层面的模仿，而不是结构层面的还原。正确的用法应该是：白皮书展示了"如何为角色设计差异化的说话方式"这个**方法论**，而不是要求新角色**照搬**原著角色的指纹数值。

### 6.3 D5 标签落地（18分）— 标签专属检查项缺失

**当前做法**：检查本章是否覆盖了章纲申报的标签，以及是否完成了标签对应的"必做动作"。

**问题**：content_writer §七 的标签落地清单中，仅"信息差碾压"标签有对应的公式检查（四步闭环），其他标签无对应公式。quality_reviewer 也只在 D5 中笼统要求"每缺1个标签覆盖的必做动作-10"，但**没有为不同标签定义不同的必做动作检查项**。这与 D3 的问题是同源的——类型专属验证标准缺失。

### 6.4 D2 文风仿写（7分）— 阈值弹性不足

**当前做法**：句长15-25字、对话比55-65%、段落2-4句。

**问题**：方向正确（文风形式仿写），但不同场景类型下这些数值天然会波动——战斗场景句长自然缩短、日常场景对话比自然上升。用固定阈值卡所有场景类型，会导致 content_writer 为了通过评分而在所有场景中强行维持统一数值，制造机械感。应该允许场景类型驱动的自然波动。

### 6.5 D2-B 短段落风格（3分）— 定位澄清

这个维度是**移动端平台设计规范**，不是文风仿写。放在 D2 的 sub-dimension 容易混淆。建议独立为"平台可读性"维度。

### 6.6 D7 钩子质量（8分）— 类型分配表 vs 自然分布

方向正确（钩子是结构约束）。钩子类型轮换（悬念钩/金句钩/反转钩不得连续3章重复）合理。但 plot_planner 中的"钩子类型分配表"要求悬念钩≤50%、金句钩≥20%，这是把统计规律变成了操作手册。真人的钩子类型分布是自然形成的。

### 6.7 D9 原著相似度（4分）— 权重偏低

当前仅4分，仅在 quality_reviewer 层面检查。签约审稿 D9（15分）补充了整书层面的原创度检查。但质量自检环节的原创度检查权重太低——如果评分循环中 D3/D5 在检查章纲规定的爽点和标签是否落地（结构检查），D9 在检查这些落地的内容是否与原著雷同（内容检查），那 D9 应该与 D3/D5 的权重更接近，而不是 4 vs 18。

## 七、问题总结

### 按类型分类

| 问题类型 | 具体问题 | 严重程度 |
|----------|---------|---------|
| **信息断裂** | quality_reviewer D3 "四步闭环"未定义；白皮书类型专属公式未动态传递到评分层；content_writer 标签清单公式不完整 | 高 — 直接影响18分维度的准确性 |
| **规则自损** | content_writer §9 反AI规则集叠加约200行，轮换池方案无法终止"每修一tic引入一新tic"循环 | 高 — content_writer 40%的内容在教AI怎么写不像AI |
| **权重错配** | 规划文档45% vs 正文20%；正文权重在所有层中只排第三 | 中 — 读者体验在评分体系中代表性不足 |
| **指纹越界** | D4-B 将原著角色的语言指纹作为新书角色的验收标准 | 中 — 从方法论滑向了内容模仿 |
| **标签公式缺失** | D5 仅"信息差碾压"有对应公式，其他标签无专属检查项 | 中 — 与D3问题同源 |
| **阈值刚性** | D2 文风阈值对所有场景类型一视同仁 | 低 — 方向正确，粒度需细化 |
| **标注混淆** | D2-B 短段落应为"平台规范"，当前放在了"文风仿写"下 | 低 — 语义混淆 |

### 问题优先级

1. **P0 — 修复信息断裂**：让 quality_reviewer 动态读取白皮书/总纲中的爽点触发公式（按类型），替换当前未定义的"四步闭环"检查。同时为 D5 的各个标签补充对应的验证公式。这是最小的改动，只涉及 agent 内部指令的文件读取路径和验证逻辑，不需要动评分维度结构。

2. **P1 — 精简反AI规则**：从 content_writer 中删除轮换池（§9.2）、替换词轮换（§9.5）、金手指变体库（§9.6）、句式变异概率（§二-C）等机制。信任创意自由。保留纯技术性规则（章纲引用清洗、标签元数据禁令、现代术语转化、文件收尾检查）。

3. **P2 — 调整权重**：提升章内容（20%→35%），降低规划文档合计（45%→30%）。

4. **P3 — 修正指纹维度**：D4-B 改为检查"角色间对话是否有可辨识的差异化"，而非"是否匹配原著的指纹数值"。

## 八、不需要改的部分（确认正确）

| 部分 | 理由 |

| 五段式结构及段占比 | 必要的结构约束。方向正确。问题仅在于阈值应允许场景类型驱动的弹性，而非取消。 |

| D2-B 短段落的数值要求 | 移动端平台设计规范，保留。仅需要独立归类。 |
| D7 钩子类型轮换 | 钩子多样性是结构要求，合理。 |
| 签约审稿 D9 原创改写度 | 内容层面的原创性检查，方向正确。 |

## 九、架构层面分析

以下分析基于已确认的设计框架：多角度评分是合理分工、多模型评分是交叉校验、加权复合是分层重要性的体现。

### 9.1 评分数据源的不对称

quality_reviewer（Phase 2）和 signing_reviewer（Phase 3）的评分基础不同：

```
Phase 2 每章流程：
  初稿 → quality_reviewer评分 → 若<60分 → 重写(最多3轮) → 取最佳版本 → 终稿
                                                              ↓
                                                    quality_reviewer评分写入纪要
                                                              ↓
Phase 3 流程：                                              ↓
  orchestrator 读取纪要中的分数 → 计算"章内容均分" ← ← ← ← ← ┘
  signing_reviewer 一次性通读全部终稿 → 整书评分
```

**差异**：

- quality_reviewer 的章内容均分来自**每章经过最多3轮重写优选后的最佳版本**。例如第3章可能写了3版（55/58/62），取62分写入纪要。
- signing_reviewer 一次性面对全部终稿，基于**整书连续阅读**的体验打分。

这产生了两个后果：

**(1) 优选的单章未必组成好的全书。** quality_reviewer 在逐章评分时看不到章与章的排列效果——连续3章同类型爽点可能在单章视角下各自达标，但从签约编辑通读视角会产生疲劳感。signing_reviewer 的"连续两章爽点同质-2分"检查的是 quality_reviewer 无法覆盖的跨章维度。

**(2) 两套评分之间无交叉校验。** 如果 quality_reviewer 说第7章"钩子质量优秀"(D7=8/8)，而 signing_reviewer 说"第7章是钩子链断裂的起点"(D2扣分)，这两个信号在 composite 中被直接相加，矛盾被埋掉。orchestrator 不检查"同一章在不同评分层的评价是否矛盾"。

### 9.2 规划评分与内容评分的断层

实际运行数据展示了一个显著的分层落差：

| 评分层 | 得分 | 评分对象 |
|--------|------|---------|
| L1 白皮书 | 100 | 对原著的拆解分析 |
| L2 总纲 | 88 | 新书的蓝图设计 |
| L3 卷纲 | 97 | 卷级规划 |
| L4 章纲 | 96.8 | 每章的施工图纸 |
| L5 章内容 | 79.4 | 读者实际读到的文字 |

规划层（L1-L4）评分 88-100，执行层（L5）评分 79.4。规划好≠执行好，这种情况本身是正常的。但在评分体系中有两个结构性问题：

**(1) 复合权重未能反映断层的严重性。** 规划层在 composite 中占 45%，把 79.4 的章内容拉到了 85.36 的综合分。一本"读了会觉得一般"的书被系统判定为"良好"。

**(2) 没有评分机制解释这个断层的根因。** 章纲 96.8 分是独立评估章纲自身质量的结果。章内容 79.4 分是因为没有执行好 96.8 分的章纲？还是因为 96.8 分的章纲本身虽然结构正确但不适合实际写作（过刚、过细、缺乏弹性）？现有评分体系无法区分这两种情况。orchestrator 的模板中有"各层详细总结"和"最致命问题"，但没有"规划-执行落差分析"这一项。

### 9.3 pipeline_auditor 的检查盲区

pipeline_auditor 是唯一做端到端链路检查的 agent。核心方法是逆向溯源：

```
章内容中的核心事件
  → 是否能在章纲中找到对应项？
    → 是否能在卷纲中找到来源？
      → 是否能在总纲中找到来源？
        → 是否能在白皮书中找到来源？
```

**但它的检查逻辑是"存在性检查"，不是"正确性检查"。**

具体举例：

```
章内容第3章：方尘在山洞中引气入体，触发金手指觉醒
  → 章纲第3章确实提到了"引气入体"事件 → ✅ 可溯源，通过

但以下情况不会被发现：
  → 章纲中的"引气入体"应按照白皮书定义的"逆境逆袭"爽点公式来设计
  → 白皮书中"逆境逆袭"的完整公式是 A→B→C→D→E（5个步骤）
  → 章纲的设计实际只覆盖了 A→C→E，缺失 B 和 D
  → pipeline_auditor 验证到了"章内容→章纲"的对应关系，所以判定通过
  → 但章纲→白皮书的对应关系虽然在"存在性"上成立（章纲确实引用了白皮书），
     在"正确性"上不成立（章纲对该类型的公式使用有遗漏）
```

也就是说，pipeline_auditor 验证的是**信息没有凭空出现**，但不验证**信息在传递过程中没有变形**。"可追溯"和"保真"是两个独立的检查维度，当前只覆盖了前者。

### 9.4 跨层保真度验证缺失

白皮书为每种爽点类型生成了专属的结构公式：

```
白皮书 §四：
  "身份打脸"公式 = 反派质疑→主角沉默→第三方揭露→反派震惊→主角回应（5步）
  "情感守护"公式 = 关心触发→主角回应→关系升温或心结解开（3步）
  "逆境逆袭"公式 = 绝境触发→挣扎→突破→逆转→余波（5步）
  ...（每种类型独立公式）
```

这些公式经过总纲→卷纲→章纲，最终指导 content_writer 写作。但在这条链路上：

| 验证节点 | 是否有人检查 | 检查什么 |
|----------|------------|---------|
| 总纲是否完整继承了白皮书的各类型公式 | L2 master_outline_reviewer | D2 检查"与白皮书一致性"，但只验证到"爽点体系与赛道映射对齐"，不逐类型验证公式结构是否完整继承 |
| 卷纲的爽点分配是否使用了正确的类型公式 | L3 volume_outline_reviewer | D3 检查爽点密度和轮换，不检查单类型公式的正确使用 |
| 章纲的爽点设计是否匹配其申报类型的专属公式 | L4 chapter_outline_reviewer | D4 检查"压制→反杀闭环"，但这是通用检查，不是按类型验证 |
| 章内容的实际爽点是否符合其类型的公式 | L5 quality_reviewer | D3 只有未定义的"四步闭环"，不区分类型 |

**结论**：白皮书生成类型专属公式的工作，在下游评分环节没有对应的验证机制。没有人检查"情感守护"爽点在章纲/章内容中是否使用了正确的3步公式而非用"身份打脸"的5步公式来套用。这是跨层保真度的系统性缺失。

### 9.5 Phase 3 评分结果无反馈回路

当前系统的信息流是单向的：

```
Phase 1 (框架生成) → Phase 2 (内容写作+即时质检) → Phase 3 (全层级评分)
                                                           ↓
                                                    总结性审核报告
                                                    （含共性通病+优化建议）
                                                           ↓
                                                    没有回到 Phase 1/2 的路径
```

总结报告中的发现（如"开篇3段未进入核心矛盾"、"爽点密度标准内部冲突"）是目前最有价值的产出——来自全系统最强模型对全流程的综合分析。但这些发现的下游消费路径是空白的。如果有10本书都出现了同样的问题，报告会标记为"共性通病 P1"，但不会触发 content_writer 或 plot_planner 的指令调整。

这不是要建立完全自动化的改写循环（这可能引入新问题），而是当前设计中，跨书总结提取的规律性发现——这些发现对优化 agent 指令最有价值——没有被结构化为可消费的形式。

### 9.6 orchestrator 的能力未被充分利用

orchestrator 使用 V4 Pro（全系统最强模型），当前的核心逻辑是：

```
串行调用子agent → 填充分数表格 → 加权求和 → 生成报告
```

这些操作中只有"生成报告"需要推理能力——加权求和和表格填充是任何脚本都能完成的。

V4 Pro 能做但 orchestrator 没有让它做的是：

- **跨层矛盾标记**：quality_reviewer 说某章钩子优秀而 signing_reviewer 说该章钩子断裂时，orchestrator 有能力发现并标注这个矛盾
- **级联效应识别**：当 L3/L4/L5 的扣分项可以追溯到 L2 的同一根因时，orchestrator 有能力做归因分析，而不是把各层扣分当作独立事件
- **离群值识别**：单章评分远低于均值（如 30 vs 70+）时，orchestrator 有能力单独标注异常章节
- **规划-执行落差趋势**：多本书的"章纲均分 - 章内容均分"差值是否稳定，是否存在系统性执行偏差

现有 Step 4 模板中的"各层详细总结""最致命问题""优化建议"三个占位符依靠 V4 Pro 的常识来填充，但没有结构化的分析逻辑来支撑这些判断的产生过程。
