## 一、卷数计算算法（步骤 1）

按以下优先级尝试计算（上层失败则降级到下层）：

**第 1 优先级（精确计算）**：
```
总卷数 = 目标总字数 / (每卷预估章数 × 平台目标单章字数)，向上取整
```

**第 2 优先级（赛道估算）**：
```
总卷数 = 赛道常规总字数中位数 / (每卷预估章数 × 平台目标单章字数)
```

**第 3 优先级（原作等比缩放）**：
```
总卷数 = 原作卷数 × (节奏调整系数)
节奏调整系数 = 1 / (1 - 大高潮间隔缩短比例)
```

若三层计算全部失败 → 禁止继续，返回错误，列出缺失参数。

## 二、自检规则 A~J

### A. 文件存在性检查

- [ ] 00-全书命运总谱.md 存在
- [ ] 01-人物命运谱/ 文件数 ≥ 总纲领命名角色数
- [ ] 02-剧情命运谱/ 文件数 = 总卷数
- [ ] 03-伏笔命运谱/全书伏笔网络.md 存在
- [ ] 04-世界观展开谱/世界观展开节奏.md 存在
- [ ] 05-卷级注入/ 文件数 = 总卷数

### B. 逐章细化度检查

逐卷扫描 02-剧情命运谱/，每卷事件链行数 = 预期章数 → PASS/FAIL。

### C. 逐行完善度检查

- [ ] 所有"核心事件"字段均 ≥20字
- [ ] 无"_____"或"骨架级占位"等占位符
- [ ] 所有"新埋设伏笔ID"有效（非空=可在 03 中找到 / ="—"=合法空值）

### D. 跨文件一致性

- [ ] 00 声明总卷数 = 02 文件数 = 05 文件数
- [ ] 00 中声明的角色在 01 中均有对应文件
- [ ] 伏笔ID引用一致性：03 中的每个伏笔ID在至少 1 个 02 卷中有"埋设"+至少 1 个卷中有"回收"
- [ ] 世界观展开无冲突：04 中"本卷可揭示"与该卷 05"不可触碰"不矛盾

### E. 叙事一致性

- [ ] 角色死亡卷之后，02/05 中不再出现该角色
- [ ] 境界递进跨卷线性，无回退
- [ ] 关键物品跨卷追踪无断裂

### F. 注入包完整性

- [ ] 每卷的注入包 §3 事件链行数 = 02 对应卷的事件链行数
- [ ] 注入包 §4 中列出的伏笔 ID 在 03 中均有对应条目
- [ ] 注入包 §2 中列出的角色名在 01-人物命运谱/ 中均有对应文件

### G. 与仿写衍生总纲领的一致性

**G1 角色集合校验**：
- [ ] 总纲声明的命名角色列表与 01-人物命运谱/ 已生成的命名角色文件对比
- [ ] 差异角色（总纲有但 01 无 / 01 有但总纲无）→ 判定 PASS/FAIL

**G2 爽点类型校验**：
- [ ] 总纲声明的爽点类型集合与 02 事件表中实际使用的爽点类型集合对比
- [ ] 02 中存在但不在总纲中的爽点类型 → 判定 PASS/FAIL

**G3 节奏参数粗略校验**：
- [ ] 总纲大高潮间隔与本书实际卷均章数对比
- [ ] 偏差 > 30% → "⚠️ 警告"

**G4 分类标签校验**：
- [ ] 总纲声明的标签集合在事件链中的覆盖检查
- [ ] 列出未在任何卷的事件链中有对应场景的标签

### H. 差异化自检（精简版·安全网模式）

仅在「全量模式」或「增量重建模式」（含 Phase 级重建）时执行。仅全局模式下跳过。

**H1 约束表落地检查**：

对每个 phase 执行：
a. 读取约束表本 phase 行 → 提取 ban_forbidden_event_mapping、required_new_event_types、overlap_assessment、danger
b. 禁止事件映射检查：事件链中"核心事件"是否命中 ban_forbidden_event_mapping → 命中 ≥1 个 → FAIL
c. 强制新事件类型检查：required_new_event_types 是否真实出现 → 缺 ≥1 个 → WARN
d. Danger phase 检查：若 danger=true → ban_forbidden 必须全部通过 + required_new 必须全部满足

输出判定表：
```
| Phase | overlap 预检 | danger | ban 通过 | required_new 满足 | 判定 |
```

**H2 差异化衰减趋势检查**：

- 若本卷有 3+ 个 phase，检查 ban 违反次数是否从后段开始逐 phase 增加 → "差异化衰减" WARN
- 若连续 2 个 phase 的 required_new_event_types 全部缺失 → 升级为 FAIL
- 若最后一个 phase 的 ban 违反次数 > 第一个 phase → "后段失控" WARN

**H3 高辨识度节点命中检查**（条件激活）：

若白皮书 §X 存在：
- 逐章扫描"核心事件"字段，与白皮书 §X 中标注为"极高"的节点做关键词模糊匹配
- 极高节点命中任意 1 个 → FAIL
- 高节点命中 ≥3 个 → FAIL
- 若白皮书 §X 不存在 → 跳过 H3

判定标准：

| H1 | H2 | H3 | 最终判定 | 行为 |
|:---:|:---:|:---:|:---:|:---|
| PASS | PASS | PASS | ✅ PASS | 正常放行 |
| 任一 WARN（无 FAIL）| — | — | ⚠️ WARN | 注入包 §2.1 标记 DIFF_WARNING |
| 任一 FAIL | — | — | ❌ FAIL | 禁止输出，记录 FAIL phase 号 |
| — | — | H3 极高命中 | ❌ FAIL | 同上 |

### J. 禁止模式合规检查

全量模式 + 增量重建模式下执行。仅全局模式下跳过。Phase 级重建仅检查本 phase。

**J1 逐 phase 确定每条规则的适用范围**：

| forbidden_pattern 类别 | 适用范围 |
|------------------------|---------|
| "连续 N 章无突破" | intensity ≥ 中高 且 core_task 含"突破/升级/变强"语义的 phase |
| "连续 N 章无爽点" | 所有 phase |
| "突破后不展示战力碾压" | 仅含 事件类型=突破 的章及其后续 2 章 |
| "突破场景轻描淡写" | 仅含 事件类型=突破 的 phase |

**J2 逐条执行检查**：

检查输出格式：
```
### 本卷 Pattern 逐一自检

| Phase | 章号 | 事件类型 | 违反 Pattern | 严重度 | 说明 |
```

**J3 判定汇总与行为**：

| 判定 | 行为 |
|:---:|------|
| ✅ 全部 PASS | J 项通过，结果写入注入包 §7.5 |
| ⚠️ 存在 WARN（无 FAIL）| J 项放行。WARN 标记写入注入包 §7.5 |
| ❌ 存在 FAIL | 禁止输出。记录 FAIL 的 pattern 和章号。回到步骤 4 修改事件链 |

J 项 FAIL 后的修正：
- 修正方式：修改事件链对应行的核心事件/事件类型/爽点类型
- 3 轮修正后仍 FAIL → 输出事件链 + 自检报告，标注"⚠️ J 项未通过"

## 三、质量铁律

以下铁律在生成全流程中强制执行，违反任何一项即禁止输出：

1. **ALL_OR_NOTHING·剧情**：所有卷必须完成逐章事件链后才输出
2. **MINIMUM_DETAIL_PER_CHAPTER**：每章核心事件≥20字
3. **NO_SKELETON_PLACEHOLDER**：禁止"详见后续细化"等替换表述
4. **CHARACTER_CONSISTENCY**：每个出场角色的状态变化必须与其人物命运谱一致
5. **FORESHADOW_CLOSURE**：每个伏笔ID的埋设卷和回收卷都必须在02和03中对应存在
6. **INJECTION_ISOLATION**：每卷注入包不包含任何未来卷的剧情、伏笔回收、角色结局
